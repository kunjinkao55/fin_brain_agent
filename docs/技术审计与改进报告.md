### 2.1 SSL 配置集中管理
   
**改前（17 处分散的重复代码）：**
```python
# fetch_stock_price
ctx = ssl.create_default_context()
ctx.check_hostname = False
ctx.verify_mode = ssl.CERT_NONE

# fetch_stock_history （重复）
ctx = ssl.create_default_context()
ctx.check_hostname = False
ctx.verify_mode = ssl.CERT_NONE

# ... 共 17 处相同的 3 行代码
```

**改后：**
```python
# 模块级常量，一次定义，全文件复用
_SSL_CTX = ssl.create_default_context()
_SSL_CTX.check_hostname = False
_SSL_CTX.verify_mode = ssl.CERT_NONE

# 使用处只需一行
ctx = _SSL_CTX
```

**为什么 SSL 必须绕过（技术背景）：**

新浪财经 `hq.sinajs.cn` 和东方财富 `push2his.eastmoney.com` 等免费 API 服务器使用的 TLS 证书存在以下问题之一：

1. **证书链不完整** — 服务器只发送了叶子证书，缺少中间 CA 证书。Python 3.12 的 `ssl` 模块（基于 OpenSSL 3.x）对此更加严格，在 TLS 握手阶段就断开连接，报错 `[SSL: UNEXPECTED_EOF_WHILE_READING]`。

2. **自签名证书** — 部分服务器使用内部 CA 签发的证书，不在公共信任链中。

3. **SNI（Server Name Indication）不匹配** — 服务器端 SNI 配置与实际证书 CN 不一致。

**Python `ssl` 模块的三个关键参数：**

| 参数 | 默认值 | 我们设的值 | 含义 |
|------|--------|-----------|------|
| `check_hostname` | `True` | `False` | 是否验证证书 CN/SAN 与请求域名一致 |
| `verify_mode` | `CERT_REQUIRED` | `CERT_NONE` | 是否验证证书链的有效性 |
| `SSLContext` 来源 | `create_default_context()` 返回包含系统信任 CA 的上下文 | 同上但关闭验证 | 控制 TLS 握手行为 |

**工程师面试应答：**

> "我们的数据源是国内免费金融 API，这些服务器的 TLS 证书不完整。这是行业常见问题 — AkShare 和 Tushare 的底层 HTTP 库也做了同样的处理。我们在代码里用模块级常量集中管理 SSL 配置，并加了详细的免责注释。生产环境建议用付费数据源（Wind 金融终端 / Tushare Pro），它们的 API 服务器证书是完整的，可以直接启用默认 SSL 验证。"

### 2.2 LangGraph SqliteSaver 资源管理

**改前：**
```python
_CK = None

def _make_checkpointer():
    global _CK
    if _CK is None:
        cm = SqliteSaver.from_conn_string(_DB_PATH)
        _CK = cm.__enter__()      # 只进了，永远没出！
    return _CK
```

**改后：**
```python
_CK = None
_CK_CM = None

def _make_checkpointer():
    global _CK, _CK_CM
    if _CK is None:
        _CK_CM = SqliteSaver.from_conn_string(_DB_PATH)
        _CK = _CK_CM.__enter__()
    return _CK

def _cleanup_checkpointer():
    global _CK_CM, _CK
    if _CK_CM is not None:
        _CK_CM.__exit__(None, None, None)

atexit.register(_cleanup_checkpointer)
```

**SqliteSaver 工作原理详解：**

`SqliteSaver.from_conn_string()` 返回的是一个 **Python Context Manager**（上下文管理器），不是直接可用的 checkpointer 对象。上下文管理器的协议是：

```python
class ContextManager:
    def __enter__(self):   # 进入 with 块时调用，返回被管理对象
        return self.obj

    def __exit__(self, exc_type, exc_val, exc_tb):  # 退出 with 块时调用
        self.obj.close()  # 关闭 SQLite 连接等资源
```

正确用法是 `with` 语句：
```python
with SqliteSaver.from_conn_string(path) as checkpointer:
    agent = create_agent(LLM, tools, checkpointer=checkpointer)
    # ... 在 with 块内使用
# 退出 with 块时自动调用 __exit__，关闭 DB 连接
```

我们的场景需要一个**应用级单例**（多个 Agent 共享同一个 checkpointer），所以不能每次都用 `with`。解决方案是：

1. 手动调用 `__enter__()` 获取 checkpointer（进入托管）
2. 保留 context manager 引用 `_CK_CM`
3. 注册 `atexit` 处理器，在进程退出前调用 `__exit__()` 释放资源

**`atexit` 模块详解：**

```python
import atexit

def cleanup():
    print("清理资源...")

atexit.register(cleanup)
# 进程正常退出时（包括 Ctrl+C）自动调用 cleanup()
# 注册顺序与调用顺序相反（LIFO）
# 注意：os._exit() 和 SIGKILL 不会触发 atexit
```

**工程师面试应答：**

> "LangGraph 的 SqliteSaver.from_conn_string() 返回上下文管理器，但我们需要的 checkpointer 是应用级单例。我们手动管理了 __enter__/__exit__ 生命周期，并通过 atexit 注册了清理函数。这是在了解 Python 上下文管理器协议的基础上，针对应用级单例场景的合理折中。另一个方案是用 AsyncSqliteSaver 配合 FastAPI lifespan，但当前 Streamlit 同步架构下这个方案更简洁。"

### 2.3 Portfolio 并发安全

**改前：**
```python
class Portfolio:
    def _save(self):
        with open(self.file, "w") as f:
            json.dump({...}, f)  # 多线程同时写 → 数据损坏
```

**改后：**
```python
import threading

class Portfolio:
    def __init__(self):
        self._lock = threading.Lock()  # 每个 Portfolio 实例一把锁

    def _save(self):
        with self._lock:               # 获取锁
            with open(self.file, "w") as f:
                json.dump({...}, f)
```

**`threading.Lock` 详解：**

```python
lock = threading.Lock()

# 方式1: acquire/release
lock.acquire()
try:
    # 临界区代码
finally:
    lock.release()

# 方式2: with 语句（推荐，等同上面）
with lock:
    # 临界区代码
```

`Lock` 是**不可重入互斥锁**：
- 同一时刻只有一个线程能持有锁
- 同一个线程不能重复获取自己持有的锁（会死锁），需要用 `RLock` 如果需要重入
- `with lock:` 保证异常时也能正确释放

**为什么比文件锁（`fcntl.flock`）更好：**

| 方案 | 线程安全 | 跨进程 | 复杂度 |
|------|:--:|:--:|------|
| `threading.Lock` | ✅ | ❌ | 低 |
| `fcntl.flock` | ✅ | ✅ | 中 |
| SQLite WAL | ✅ | ✅ | 高 |

当前场景是单进程 Streamlit，`threading.Lock` 足够。

**工程师面试应答：**

> "Portfolio 使用 JSON 文件持久化，每个读写操作都通过 threading.Lock 保护。在 Streamlit 的多线程执行模型下，这可以防止并发写入导致的 JSON 文件损坏。如果后续需要支持多进程部署，可以升级为 SQLite 存储（我们已经有 SqliteSaver 的基础设施），利用 SQLite 的 WAL 模式提供原生的并发读写安全。"

### 2.4 对话状态持久化（LangGraph SqliteSaver）

这是我们之前完成的改动，这里补充完整的技术原理。

**LangGraph Checkpointer 架构：**

```
Agent.invoke(messages, config={"configurable": {"thread_id": "xxx"}})
        │
        ▼
   CompiledStateGraph
        │
   ┌────▼────┐
   │ Node 1  │ → (每个节点执行后)
   ├─────────┤    自动调用 checkpointer.put(state, config)
   │ Node 2  │    将当前状态写入 SQLite
   ├─────────┤
   │ Node 3  │
   └─────────┘
        │
        ▼
   下次 invoke 相同 thread_id → checkpointer.get(config) → 恢复状态
```

**三种 Checkpointer 对比：**

| 方案 | 持久化 | 适用场景 |
|------|:--:|------|
| `MemorySaver` | ❌ 进程重启丢失 | 开发调试 |
| `SqliteSaver` | ✅ SQLite 文件 | 单机部署 |
| `PostgresSaver` | ✅ PostgreSQL | 生产多实例 |

我们选 `SqliteSaver` — 单机部署的最优解，零运维成本。

**工程师面试应答：**

> "对话记忆使用 LangGraph 内置的 SqliteSaver 实现，这是框架官方推荐的持久化方案。每次 Agent.invoke 传 thread_id 作为 config，框架自动在状态图每个节点执行后序列化到 SQLite。下次相同 thread_id 调用时自动恢复完整状态。对比业界其他方案：LangChain Memory 只存文本摘要会丢失工具调用上下文；Redis 需要额外运维；SqliteSaver 是零依赖嵌入式的正确选择。"

### 5.1 导入时副作用 — 为什么是反模式

**改前**：
```python
# agent.py 模块顶层
_DB_PATH = os.path.join(...)
os.makedirs(os.path.dirname(_DB_PATH), exist_ok=True)  # import 时执行!

_STRATEGY, ALL_STRATEGIES = _load_strategy()            # import 时读文件!

LLM = ChatOpenAI(...)                                    # import 时联网校验 key!
```

**改后**：
```python
_STRATEGY = None
_LLM = None

def _get_strategy():
    global _STRATEGY
    if _STRATEGY is None:
        _STRATEGY = _load_from_file()
    return _STRATEGY

def _get_llm():
    global _LLM
    if _LLM is None:
        _LLM = ChatOpenAI(...)
    return _LLM
```

**知识点：Python 模块导入的执行模型**

```python
# 当你写 import foo 时，Python 实际做了：
# 1. 查找 foo.py
# 2. 逐行执行 foo.py 的顶层代码（包括函数定义、类定义、变量赋值、函数调用！）
# 3. 将执行结果存入 sys.modules['foo']
# 4. 返回模块对象

# 所以如果在顶层写 os.makedirs()，import 时就会创建目录
# 如果在顶层写 open()，import 时就会打开文件
# 如果在顶层写 ChatOpenAI()，import 时就会初始化网络连接
```

**后果**：
- 单元测试 import 模块 → 触发文件系统操作 → 测试环境可能没有那个路径 → 崩溃
- CI/CD build 阶段 import 模块 → 触发网络请求 → build 变慢或失败
- IDE 静态分析 import 模块 → 触发副作用 → 不可预期的行为

**面试讲法**：
> "所有重资源（数据库连接、文件I/O、模型加载）都采用懒初始化 — `if _x is None: _x = heavy_init()`。这样 import 模块时零副作用，首次调用时才触发初始化。这是 Python 社区处理模块级重资源的通行模式。"

### 5.2 确定性评分 + LLM 微调 — 混合评分模式

**设计思路**：

```
纯函数层 (calculate_scores)         LLM 层 (Analyst)
├── 盈利能力: ROE公式 → 9分         → "ROE虽高但环比下降，扣1分" → 8分
├── 成长性: 营收增速公式 → 10分      → LLM不改: 10分
├── 财务健康: 负债率+现金流公式 → 8分 → LLM不改: 8分
├── 估值合理: PE公式 → 3分          → LLM不改: 3分
├── 行业前景: 默认5分               → "AI光模块龙头" → 9分
└── 资金认可: 默认5分               → "净流入50亿" → 10分
                                 总分: 40/60 → 48/60
```

**为什么这样做（而不是纯 LLM 或纯规则）**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 纯 LLM | 灵活，能理解上下文 | 非确定性，同一输入不同输出 |
| 纯规则 | 确定性，可审计 | 僵化，无法处理边缘情况 |
| **混合（我们的方案）** | 4维确定 + 2维灵活 | 需要维护两套逻辑 |

**面试讲法**：
> "金融领域对评分可解释性和一致性有严格要求。我们采用规则引擎打底、LLM 微调的混合模式 — ROE、增速、负债率、PE 四个客观维度用纯函数计算（100% 确定性），行业前景和资金认可用 LLM 判断（利用它的语义理解能力）。这是业界金融风控系统的标准架构，比如银行的信用评分卡就是规则+模型混合。"

### 5.3 工具调用缓存 — Memoization 模式

**知识点：Memoization（记忆化）**

```
第一次调用: stock_price("300502") → HTTP请求(1465ms) → 结果存入缓存
第二次调用: stock_price("300502") → 缓存命中(0ms)    → 直接返回
...30秒后:  stock_price("300502") → TTL过期 → 重新请求
```

**TTL 分层策略**：

| 数据类型 | TTL | 原因 |
|---------|-----|------|
| 实时股价 | 30s | 盘中3秒刷新，30s是精度/开销的平衡点 |
| K线 | 5min | 日线数据日内不变 |
| 财报/估值 | 30min | 季报频率，日内不会变 |
| 行业分类 | 1h | 行业分类基本不变 |
| 龙虎榜 | 5min | 盘中持续更新 |

**线程安全实现**：

```python
_lock = threading.Lock()

def get(key):    with _lock: ...   # 读也加锁，防止读到半写状态
def set(key, v): with _lock: ...   # 写加锁，原子更新
```

**为什么不用 `functools.lru_cache`**：
- `lru_cache` 无法设置 TTL（只能按调用次数淘汰）
- `lru_cache` 无法按参数选择性缓存（同一函数对不同 symbol 需要不同 TTL 吗？不需要，但某些场景可能需要）
- 手动实现可以加统计（命中率、条目数），生产环境需要这些指标

**面试讲法**：
> "Agent 在一次对话中可能对同一只股票反复调用多个工具，每个工具都发 HTTP 请求会造成大量冗余。我们实现了一个 TTL 分层的线程安全缓存 — 股价 30s、财报 30min、行业 1h。LangChain 内置 InMemoryCache 但粒度太粗，我们手写的版本可以按工具类型分别设置 TTL。这是典型的空间换时间 trade-off。"

### 5.4 Python 异常层级 — 为什么不能 `except Exception`

```python
BaseException              # 所有异常的基类
├── KeyboardInterrupt      # Ctrl+C — 通常不应捕获！
├── SystemExit             # sys.exit() — 通常不应捕获！
├── GeneratorExit          # 生成器关闭 — 通常不应捕获！
└── Exception              # 常规异常的基类
    ├── ValueError
    ├── TypeError
    ├── json.JSONDecodeError
    ├── urllib.error.URLError
    └── ... 数百个子类
```

**改前**：`except Exception: data = {}`  
**问题**：如果用户按 Ctrl+C 退出，`KeyboardInterrupt` 被吞掉，程序卡死。  
**改后**：`except (json.JSONDecodeError, TypeError): data = {}`  
**效果**：只捕获 JSON 解析失败，Ctrl+C 正常传播。

**面试讲法**：
> "Python 的异常层级中，KeyboardInterrupt 和 SystemExit 是 BaseException 的子类但不是 Exception 的子类 — 但 `except Exception` 仍会捕获大部分编程错误比如 AttributeError、NameError。正确的做法是只捕获你预期会发生的具体异常类型，让意外错误向上传播到统一的错误处理层。"

### 5.5 StateGraph `processing_log` 数据流

**问题背景**：LangGraph 的 StateGraph 中，每个节点返回的 dict 会与当前 state **合并**。对于普通 `list` 字段（没有 `Annotated` reducer），返回的新值会**完全覆盖**旧值，而不是追加。

```
State: {processing_log: []}  ← 初始值

DataCollector 返回: {processing_log: [step1]}  → 覆盖
    State 变为:    {processing_log: [step1]}

Analyst 读 state["processing_log"]  → [step1]
          append step2
          返回: {processing_log: [step1, step2]}  → 覆盖
    State 变为:    {processing_log: [step1, step2]}

Reporter 读 → [step1, step2]
           append step3
           返回: {processing_log: [step1, step2, step3]}
```

**关键点**：每个节点必须在返回前**读取旧值、追加、返回完整列表**。如果有节点只返回自己的条目而不包含旧值，就会丢失前面的日志。

**面试讲法**：
> "LangGraph 的 state 合并策略对不同类型的字段不同：messages 用 add_messages reducer 自动追加，普通 list 字段则是覆盖语义。所以 processing_log 的传递需要每个节点显式读取旧值、追加新条目、返回完整列表。这不是 bug，是对框架 state 合并语义的理解。"

---

### 5.6 ChromaDB RAG 嵌入模型选型

我们最终用了 ChromaDB 内置的 ONNX `all-MiniLM-L6-v2` (80MB)，而不是更大但需要 HuggingFace 下载的 BGE 模型。

**选型过程**：

| 方案 | 大小 | 下载源 | 中文效果 | 决定 |
|------|------|--------|---------|:--:|
| BGE-small-zh (sentence-transformers) | 400MB | HuggingFace (被墙) | 最佳 | ❌ |
| text2vec-base-chinese (ModelScope) | 400MB | ModelScope (国内CDN) | 优秀 | ❌ 依赖太重 |
| ONNX MiniLM (ChromaDB 内置) | 80MB | ChromaDB CDN | 中等 | ✅ 选择 |
| OpenAI text-embedding-3-small | API | OpenAI | 优秀 | ❌ 花钱 |

**权衡原则**：
- 10 条游资记录不需要 SOTA 中文模型
- 80MB ONNX 模型零额外依赖，ChromaDB 自带
- 如果数据量增长到 1000+ 条，再升级到 BGE

**面试讲法**：
> "嵌入模型选型是一个 trade-off：大模型中文效果好但需要 400MB 下载且在境内网络不可达。最终选择 ChromaDB 内置的 ONNX 方案 — 80MB、零额外安装、一次下载永久缓存。这是对成本和效果的最优平衡。生产环境数据量大时可无缝切换到 OpenAI embedding API。"

# 第十二轮技术审计（2026-07-17 批次7）：三路Critic + Repair闭环 + FCF引擎 + Company Classifier

## 架构升级

### Pipeline: 6节点 → 8节点

### 三路并行Critic (critics_node)

# 第十三轮技术审计（2026-07-17 批次8）：国网信通报告质检暴露的系统性缺陷修复

> 日期：2026-07-17

## 质检发现 → 修复映射

### 1. 时效性（P0）：业绩快报已发布，结论却仍在"等待半年报"

- `tools.py` `get_recent_announcements`：业绩快报/业绩预告标题触发正文抓取，新增 `_extract_flash_report` 提取结构化 `快报数据`（营收/归母净利/扣非净利/同比增速），`_format_flash_hint` 写入"提示"字段供 format_report 渲染（激活了原死字段）。

- 快报正文为表格排版：关键词被换行拆散（"归属于上市公司股东的净…利润"）、变动幅度为裸数字（-12.59 而非"下降12.59%"）、单位在文首"单位：万元"。提取器兼容表格/散文双格式，用第三字（净/扣）区分归母与扣非归母。
- ANALYST_PROMPT 新增时序规则：快报已覆盖分析期时禁止再写"等待半年报/中报确认拐点"，须基于快报数据直接给出判断。
- AUDITOR_PROMPT 新增第9项审查：时效性矛盾（公告列了快报但结论还在等财报 → ❌）。

### 2. 估值算术（P0）：情景估值三错二、无桥接

- `_FORMAT_MANDATORY` 情景估值模板新增结构化 `EPS`/`PE` 数值字段；ANALYST_PROMPT 核心约束3：价格=EPS×PE，概率合计=100%，声明代码将强制校验。
- 新增模块级 `_validate_scenarios()`：算术校验（容差15%，不符按 EPS×PE 重算并记录）、价格单调性（悲观≤基准≤乐观）、概率合计归一、概率加权价值代码重算（不信任 LLM 算术）。结果写入 `item["_scenario_check"]`。
- format_report 情景估值渲染 `[EPS=0.25×PE=20]` 与概率加权价值。
- 实测（质检报告原数字）：悲观 7.00→5.0、乐观 12.50→14.75、概率加权重算 9.85——与质检手工计算一致。

### 3. PE 三口径打架（正文25/校验33/审计28）

- 删除 `pe_est = opt_p/fv*18` 拍脑袋启发式（"33倍"出处），校验Agent改读结构化 PE 字段。
- AUDITOR_PROMPT 明确：情景算术与单调性已由代码校验，禁止 LLM 从文本重新推导 PE。
- 审计表第8项"情景EPS单调性"从硬编码 ⚠️ 改为读 `_scenario_check` 真实结果（✅/⚠️含修正说明）。

### 4. 前瞻PE 986倍（季节性失真）

- Q1归母净利 < 年报净利15% → 判定季节性失真，禁用前瞻PE（显示"季节性失真"），注入 `_fwd_pe_note` 引导参考 TTM PE。

### 5. 安全边际三重叠加（3.10元 ≈ 0.57倍PB）

- `_quality_adjustment` 钳制 bug：`max(limits[0]-0.10, ...)` 实际等价 `max(0.05, ...)`，任何公司安全边际只增不减 → 改为允许负值（-0.05~+0.15），优质公司可放宽。
- 新增 PB 地板：当前 PB≥1 时买入价不得低于 0.8×每股净资产（破净股如银行豁免），触发时写入估值明细。实测：2.60→4.33元。

### 6. 内部矛盾

- **风险清单被清空的真 bug**：`(item.get("风险", []) if isinstance(item, list) else [])` 中 `item` 恒为 dict → 条件恒 False → Q1现金流/FCF/色彩标签预警触发时 LLM 写的风险全部丢失（4处同构 bug，全部修复）。这是"风险清单漏掉真正风险"的结构性根因之一。

- 浮点格式化：盈利能力/财务健康"依据"字符串 `:.1f`（17.1648694075% → 17.2%）。

### 7. 分析遗漏（代码可覆盖部分）

- 归母/扣非增速缺口 >30pp → 自动注入风险："非经常性损益约X亿，表观利润受非经常项目影响，主业趋势以扣非为准"。
- 应收账款 > 年净利润×3 → 自动注入回款/信用减值风险。

### 8. 顺带修复

- `calculate_scores` 现金流数据为空时 `UnboundLocalError`（FCF预警块引用了块内变量，补齐默认值）。
- 代码预检启用原死代码维度越界检查（评分>10→❌），情景价格倒挂纳入预检阻止跳过审计。

# 第十三轮补充②（2026-07-19 批次10）：财报快源 RPT_FCI_PERFORMANCEE 接入

3. **TTM 公式泛化**（tools.py + agent.py 两处）：`年报 - 去年Q1 + 最新Q1` → `年报 - 去年同期 + 最新同期`（一季报/中报/三季报通用）。
4. **前瞻PE年化系数期间感知**（agent.py）：一季报×4 / 中报×2 / 三季报×4/3（原一律×4，中报数据下会低估一倍）。

## 第十八轮技术审计补丁2：analyst_node 结构化输出失败修复

> 日期：2026-07-20  
> 触发：完整 graph 生成 300502 报告时，在 `analyst_node` 因 `情景估值.概率加权价值` 被模型输出为字符串而非 `Scenario` 对象，导致 `RuntimeError: 结构化输出失败，且手动解析也失败`。

### 问题根因

`AnalystOutput` schema 中 `情景估值` 类型为 `Dict[str, Scenario]`。当 LLM 把 `概率加权价值` 当作情景估值的一个键输出为字符串（如 `"277.5元（=150*0.25+280*0.5+400*0.25）"`），LangChain function calling 路径会直接触发 Pydantic 校验失败，且 `_StructuredFallbackChain` 原本会抛出异常，导致整个 graph 中断。

此外，`_normalize_llm_output()` 原本只被 `_parse_structured()` 调用，function calling 路径没有经过规范化。

### 已落地改动

| # | 改动 | 文件 | 类型 |
|---|------|------|:--:|
| 1 | `_StructuredFallbackChain.invoke()` 在结构化解析失败后，新增 `_normalize_llm_output_from_text()` 兜底：从原始文本提取 JSON 并调用 `_normalize_llm_output()`，返回规范化后的 dict 而非抛异常 | `backend/agent.py` | 健壮性 |

## 第十八轮技术审计补丁3：LLMFallbackChain 缺少 bind_tools

> 日期：2026-07-20  
> 触发：用户调用 `create_agent` 时报错 `'LLMFallbackChain' object has no attribute 'bind_tools'`。

### 问题根因

LangGraph `create_agent(llm, tools, ...)` 内部会调用 `llm.bind_tools(tools)`。Phase 1 引入的 `LLMFallbackChain` 只实现了 `invoke()` 和 `with_structured_output()`，没有 `bind_tools()`，导致所有使用 `create_agent` 的 Agent（data_collector、chat、phantom）初始化失败。

### 已落地改动

| # | 改动 | 文件 | 类型 |
|---|------|------|:--:|
| 1 | `LLMFallbackChain` 新增 `bind_tools()` 方法，返回 `_ToolsBoundFallbackChain` | `backend/agent.py` | 兼容 |
| 2 | 新增 `_ToolsBoundFallbackChain`：内部按槽位熔断调用 `base.bind_tools(tools).invoke()` | `backend/agent.py` | 兼容 |
| 3 | `_ToolsBoundFallbackChain` 支持 `with_structured_output()`，保持与 `_StructuredFallbackChain` 行为一致 | `backend/agent.py` | 兼容 |
