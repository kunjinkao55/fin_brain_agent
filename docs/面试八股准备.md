## 零、设计哲学（面试必答）

**Q: 为什么不做实时PE/PB/目标价的精确计算？**

> 这是刻意为之的设计决策。FinBrain 定位为基本面研究 Agent，不是实时行情终端。免费 API 拿不到 Level-2 数据，硬追精确 PE 会在最不该犯错的地方犯错——之前出现过把去年净利润当今年一致预期、PE 差 30% 的情况。所以所有 PE/PB/市值标注"基于财报，非实时行情"，砍掉了精确买入价、期望收益、仓位百分比。Agent 的算力应该花在它能做好的事情上：三年毛利率趋势、现金流为何与利润背离、护城河能否持续。

**Q: 怎么防止 LLM 乱改数据？**

> 分层防御。第一层：工具权限隔离，Analyst 无 tools。第二层：Reporter 后处理强制覆盖——`_fix_and_decide` 对每只股票调 `calculate_scores`，覆盖盈利能力/成长性/财务健康/估值合理四个维度 + 估值水位 PE/PB/市值。第三层：关键信号代码检查，不依赖 LLM"注意"——Q1 经营现金流为负自动注入风险警告。第四层：证伪条件——任何投资逻辑都有可验证的失败条件。

**Q: 你这个系统怎么部署？能支持多少用户？**

> 两种模式。本地模式：用户 `pip install` 后直接 `streamlit run`，全部在自己电脑上跑，天然单用户。远程模式：服务器跑 FastAPI + Redis + 定时预取调度器，客户端只做 LLM 推理。用户的 API Key 只在本地 `.env`，永不传到服务器。数据 API 是纯读缓存（毫秒级），配合 FastAPI 的异步并发 + Nginx 反向代理，轻松支撑数千并发。三层缓存（scheduler 预取 → Redis → Nginx proxy_cache）保证冷数据请求也不排队。

**Q: 你的 Agent 能做什么、不能做什么？**

> 能做：读懂财报（营收/利润/现金流趋势）、判断商业模式（护城河来源+复制难度）、行业周期定位、识别财务预警信号。不能做：精确预测股价、给出精确买卖点、替代交易终端。FinBrain 是行业研究员，不是交易员。

## 一、Agent 架构与工作模式

### 1. 你怎么理解 Agent？它和普通 LLM 应用最大的区别是什么？

**核心区别：Agent 能自主决策行动序列，普通 LLM 应用只能做单次推理。**

三个关键能力：

| 能力 | 说明 | FinBrain 实例 |
|------|------|-------------|
| **工具使用** | 调用外部函数/API 获取真实数据 | 13 个 `@tool`，LLM 自主决定调用哪个 |
| **记忆** | 跨轮次保留上下文 | SqliteSaver 持久化对话状态 |
| **规划** | 分解复杂任务为多步执行 | DataCollector→Analyst→Reporter 三节点流水线 |

### 2. Agent 常见的工作模式有哪些？

| 模式 | 核心思想 | 项目落地 |
|------|---------|---------|
| **ReAct** | Thought→Action→Observation 循环 | DataCollector 子 Agent 用 `create_agent` |
| **Plan-and-Execute** | 先生成计划再逐步执行 | 三节点流水线 |
| **Router** | 根据输入分类路由 | `_classify_request()` + 显式模式切换 |
| **Multi-Agent** | 多个专业化 Agent 协作 | DataCollector + Analyst + Reporter + Phantom |
| **Post-Processing** | 执行后代码修正 | Reporter 强制覆盖评分 + 计算投资评级 |

### 3. 项目里有没有用到现成框架？

**LangGraph + LangChain**。

架构分层：
```
编排层: LangGraph StateGraph (手写节点和边)
  └── 子Agent: LangChain create_agent (ReAct循环)
        └── Tool层: LangChain @tool 装饰器
              └── 数据层: tools.py (纯Python，零框架依赖)
              └── 决策层: scoring.py (纯Python，零LLM依赖)
```

选择 LangGraph 而非 AutoGen/CrewAI：我的 Data→Analyze→Report 流程固定，不需要 Agent 间"协商"。StateGraph 编译时确定图结构，执行路径可预测。Analyst 和 Reporter 无 tools，从框架层面保证权限隔离。

### 4. 投资评级是 LLM 算的还是代码算的？

**代码算的**。LLM 只写叙事（亮点/风险/逻辑链）。`scoring.py` 的 `compute_investment_rating()` 按公司类型动态赋权 + 算安全边际 + 出 BUY/HOLD/SELL。Reporter 的 `_fix_and_decide` 强制覆盖 LLM 可能乱改的分数和评级。同一只股票跑 100 次得分完全一致。

### 5. 怎么避免路径震荡（重复规划）？

1. **StateGraph 固定流水线** — 节点间不形成循环
2. **DataCollector 只跑一轮** — Prompt 明确"调完所有工具后汇总"
3. **max_tokens 限制** — 天然限制循环深度
4. **评分后处理覆盖** — 代码最终说了算

## 二、工具调用与 MCP

### 6. 13 个工具怎么管理？

`tools.py` 纯 Python 函数（零 Agent 框架依赖） → `agent.py` 的 `@tool` 装饰器薄包装。不同 Agent 注册不同工具集实现权限隔离。工具可独立测试、可同时给 LangGraph Agent、MCP Server 和 Streamlit 前端使用。

### 7. 工具调用 vs Function Calling vs 普通 Prompt

| 维度 | 普通 Prompt | Function Calling | Tool Use (LangChain) |
|------|-----------|-----------------|---------------------|
| LLM 输出 | 自由文本 | JSON `{name, arguments}` | JSON `{name, arguments}` |
| 谁执行 | 无 | 调用方代码 | 框架自动拦截+执行+回传 |
| 多轮 | 不支持 | 需手动循环 | 框架内置 ReAct |

### 8. 你怎么理解 MCP / Prompt / Harness？

| 概念 | FinBrain 实例 |
|------|-------------|
| **MCP** | `MCP_server.py`暴露 5 个 Tool 给 Claude Code |
| **Prompt** | `ANALYST_PROMPT`、`_FORMAT_MANDATORY`、策略热加载 |
| **Harness** | 工具分级、SqliteSaver、压缩、路由、降级、评分确定性 |

## 三、RAG 与知识库

### 9. 知识库是怎么搭的？

```
文档上传(PDF/DOCX/TXT) → 格式识别 → 解析器分发
  → 三级切片(段落→句子→字符, 500字/块, 50字重叠)
  → ONNX MiniLM 向量化 → ChromaDB 持久化
```

4 个知识库：会计准则(15条预置+用户上传)、行业研报、交易策略、游资风格。

### 10. Lost in the Middle 怎么处理？

1. 切片时加元数据标注（文档标题/章节/页码）
2. 分层检索：先搜粗粒度再搜细粒度
3. Prompt 显式提示优先使用靠前内容
4. 过滤过短的块（<20字符）

### 11. 怎么控制模型幻觉？

| 层 | 手段 |
|------|------|
| **RAG 层** | 每个 chunk 带 source 标注 |
| **Prompt 层** | "缺失数据标 N/A""依据栏引用具体数值" |
| **输出约束** | 固定 JSON Schema，缺失字段标 null |
| **工具层** | Analyst 无 tools，只能分析已有数据 |
| **代码层** | Reporter 强制覆盖客观维度分数 |
| **逻辑层** | 证伪条件——任何投资逻辑必须有可验证的失败条件 |

## 四、系统工程与性能

### 12. 针对可靠性与并发，做了哪些优化？

| 优化 | 方案 |
|------|------|
| **并发安全** | Portfolio `threading.Lock`，SqliteSaver 单例 |
| **资源管理** | `atexit` 注册清理，SQLite 连接不泄露 |
| **上下文管理** | 超阈值自动摘要压缩 |
| **容错** | 工具返回 `{"error": "..."}` 不崩溃，Agent 标 N/A 继续 |
| **评分确定性** | 代码后处理强制覆盖 LLM 输出 |
| **懒初始化** | 重资源 double-check locking，不在 import 时加载 |

### 13. 怎么提升模型响应速度？

| 方向 | 手段 |
|------|------|
| **架构层** | Analyst 节点不用 ReAct（单次 `invoke`） |
| **工具层** | TTL 分层缓存（30s股价/30min财报/1h行业） |
| **Prompt 层** | 精简 System Prompt，策略热加载按需注入 |
| **网络层** | 数据接口加缓存，避免重复 API 调用 |

### 14. 一个比较完整的 Prompt 包含哪些部分？

```
1. Role（角色）         → "你是 FinBrain 高级投资分析师"
2. Framework（框架）     → 十步分析链
3. Rules（规则）         → "纯JSON""多股票数组"
4. Context（上下文）      → 行业分析模板
5. Format（输出格式）     → JSON Schema
6. Quality（质量标准）    → 归母≠扣非/现金流≠含金量/证伪条件
7. Examples（示例）       → 正确vs错误的数组格式
8. Guardrails（护栏）    → "不构成投资建议"
```

## 五、投资决策引擎

### 15. 动态权重是怎么设计的？

5 种公司类型 × 6 维权重矩阵。价值型重估值(30%)+财务(25%)，成长型重成长(30%)+行业(20%)+壁垒(20%)，周期型重行业周期(30%)+估值(25%)。

安全边际 = 类型基准(20-50%) + ROE质量调整 + 负债率调整。

合理价值 = 行业PE基准 × ROE质量系数(6档, 0.3x~1.6x) × EPS。

买入区间 = 合理价值 × (1 - 安全边际)。全部代码计算，不依赖 LLM。

### 16. 十步分析链 vs 传统财务分析

传统工具只回答"公司财务健康吗"。FinBrain 回答"是否值得投资"：

公司画像 → 商业模式 → 护城河 → 行业周期 → 财务验证 → 成长逻辑 → 市场预期 → 情景估值 → 催化剂 → 投资决策

关键洞察：增加"市场预期拆解"模块——分析当前估值隐含的增长率、市场主要担忧、以及可能的预期差。股票投资的核心是找到市场预期与实际价值之间的差距。

## 六、场景设计与开放题

### 17. 用 Agent 实现 AI 面试程序怎么设计？

```
Candidate → [Question Agent] 出题 → [Evaluator Agent] 评估
  → [Follow-up Agent] 追问 → [Scoring Agent] 综合评分
```

### 18. 这个产品的缺点？

1. **单用户设计** — 全局单例 Agent，`st.session_state` 无法多租户
2. **免费数据源不稳定** — push2 间歇性断开，THS 鉴权依赖第三方库
3. **无自动化测试** — 工具和 Agent 都靠手动跑
4. **行业分类依赖 LLM** — 未做公司分类的代码化
5. **DCF 缺失** — 估值只有 PE 法，无现金流折现

### 19. Agent 后续怎么发展？

**容易落地**：客服（规则明确）、代码审查（结构化输入输出）、数据查询（Text-to-SQL+RAG）、合规审查（对照检查清单）。

**趋势**：Agent 从"单步工具调用"走向"多步规划+执行+反思"。模型能力提升会让 Agent 越来越可靠，但永远需要护栏。MCP 协议会标准化。

## 七、FinBrain 财报分析 Agent 专项面试问题

> 以下问题围绕 FinBrain 的 Multi-Agent 财报分析系统，按 **Agent 设计、数据/RAG、估值模型、财务分析、Critic、工程实现、开放题** 七个方向组织。每个答案控制在面试可展开的 2–4 句话。

---

### Agent 架构设计

**Q1: 你为什么要做一个财报分析 Agent？它解决了传统财报分析流程中的什么问题？**

传统研报生产周期长、依赖人工 Bloomberg/Wind 查询、不同分析师口径不一。FinBrain 把“读财报→算指标→写逻辑→审风险”压缩到分钟级，且用代码固定计算规则，保证同一家公司每次打分一致，解决人工分析的时效性、一致性和可复现性问题。

**Q2: 你的系统和直接让 GPT 分析一家公司的区别是什么？**

直接让 GPT 分析容易编造 PE、错把扣非当归母、给出无法验证的买入价。FinBrain 把数据抓取、指标计算、评分、估值、审计全部代码化，LLM 只负责叙事和推理；最终报告里的数字有工具调用 trace 和审计摘要，可追溯、可证伪。

**Q3: 为什么不用一个 LLM Prompt 完成财报分析？**

单 Prompt 任务太重，模型会遗漏规则、算错数字、混淆口径。拆成多个 Agent 后，每个角色职责单一、Prompt 更聚焦；中间状态可审计、可缓存、可重试；代码层能在每个节点做确定性修正。

**Q4: 你的系统为什么设计成 Multi-Agent，而不是单 Agent？**

财报分析需要三类能力：数据工程师（抓数）、行业分析师（叙事）、财务审计（查错）。单 Agent 同时承担这三种角色会互相干扰，且无法并行审查。Multi-Agent 让 Data/Analyst/Valuation/Critic 专业化，形成生成→质疑→修正的闭环。

**Q5: 你的 Agent 架构有哪些角色？每个 Agent 的职责是什么？**

| Agent | 职责 | 是否用 LLM |
|-------|------|:----------:|
| Data Agent | 并发抓取财报、行情、行业、公告 | 否（纯代码） |
| Classifier | 按 ROE/增速/CAPEX 给公司贴混合权重 | 否（纯代码） |
| Analyst | 写投资逻辑、风险、催化剂 | 是 |
| Valuation Agent | 选估值框架、输出 PE 折价链 | 是 |
| Critics（3路） | 检查逻辑/财务/行业问题 | 是 + 代码预检 |
| Repair | 根据 Critics 清单定向修改 JSON | 是 |
| Reporter | 渲染最终报告 | 是 + 代码后处理 |
| Audit Engine | 代码级 9 项检查 + FCF 预警 | 否（纯代码） |

**Q6: 为什么 Data Agent、Analysis Agent、Valuation Agent、Critic Agent 要拆开？**

数据抓取是确定性 I/O，不该让 LLM 决定；分析是概率性推理，需要 LLM 语义能力；估值需要框架选择 + 代码计算；Critic 需要反向验证。混在一起会导致 LLM 越权调用工具、数字被叙事污染、审计失去独立性。

**Q7: Agent 之间的数据流是什么？能画一下系统架构吗？**

```
用户输入 → Classifier → Data Agent（并发工具）
                   ↓
           Analyst（JSON：逻辑/风险/催化剂）
                   ↓
           Valuation Agent（JSON：PE 折价链/情景估值）
                   ↓
           Critics（Logic/Financial/Industry 三路并行）
                   ↓
           Repair Agent（修正标记字段）
                   ↓
           Reporter（渲染报告）
                   ↓
           Audit Engine（代码校验 + 强制覆盖）
                   ↓
           结构化报告 + Execution Trace
```

所有数据通过 LangGraph 的 `State` 对象传递，每个节点返回 dict 与 state 合并。

**Q8: 你的 Workflow 是固定流程还是动态规划？为什么？**

固定流程。财报分析任务的目标、输入、输出、关键步骤是稳定的：数据→分析→估值→审查→报告。固定流程更易调试、可预测、缓存友好。动态规划会增加不可控性和调试成本，对于确定性研究任务收益有限。

**Q9: 你的系统有没有 Planner Agent？为什么没有？**

当前没有。因为任务 DAG 已经固定，不需要 Planner 临时生成。Planner Agent 更适合工具种类多、任务目标开放、执行路径不确定的场景（如通用编程、复杂研究）。财报分析路径收敛，用 Planner 反而会让流程不稳定、增加延迟。

**Q10: 如果加入 Planner Agent，它解决什么问题？**

如果未来扩展到“用户问：对比三只光模块股票，给出组合建议”，任务路径会变化（单股 vs 多股、是否要加组合优化、是否要回测）。Planner 可以动态决定调用哪些工具、哪些 Agent、是否需要并行。

**Q11: 对于财报分析任务，Planner Agent 是否真的必要？**

目前不必要。固定流程已覆盖 95% 场景。Planner 的动态性更适合异常分支（如数据缺失时切换数据源、用户要求跨行业对比）。保留固定流程作为默认路径，Planner 只处理异常分支是更务实的做法。

**Q12: 你的 Agent 如何决定下一步执行什么任务？**

LangGraph 编译时确定边。每个节点返回后，框架按图结构进入下一个节点。没有运行时“Agent 自己决定下一步”的开放规划，但节点内部可以用 LLM 做微观决策（如 Critic 判断哪些问题需要修复）。

**Q13: 为什么你的流程不是 数据获取 → RAG → GPT → 报告，而是多个 Agent？**

因为“RAG → GPT → 报告”没有校验层。数字错了、逻辑矛盾、风险遗漏都无法被发现。Multi-Agent 在生成后插入 Critics、Repair、Audit 三层检查，把一次生成变成“生成→质疑→修正→验证”的闭环。

**Q14: 你的系统里面哪里体现了 Agent 的自主性？**

1. Data Agent 自主决定调用哪些工具（财报、行情、行业、公告按需并发）。
2. Analyst 自主从数据中提炼投资逻辑和风险。
3. Critic 自主识别文本和数字中的不一致。
4. Repair 自主决定如何修改被标记字段。自主性体现在“选择做什么”和“如何修正”，而不是无约束地规划任务。

**Q15: 如果去掉 Critic Agent，结果会有什么变化？**

报告质量会显著下降：估值过高但建议买入、现金流风险被忽略、情景估值倒挂、评级与操作矛盾等问题无法被发现。Critic 是质量守门员，去掉后系统退化为“数据 + 一次 LLM 生成”。

### Critic 与 Audit

**Q16: Critic Agent 为什么需要存在？**

因为 LLM 生成内容会自洽但错误。Critic 作为反向验证者，强制检查逻辑一致性、财务指标合理性、行业事实准确性。它是“自我纠错”机制，不是装饰。

**Q17: Critic Agent 是否调用 LLM？为什么？**

是。纯代码只能检查数字和格式（如 PE 范围、评分合计），但无法判断“投资逻辑是否过度自信”“行业竞争描述是否准确”。这类语义检查需要 LLM。但 Critic 里的 Financial Critic 会先跑代码预检，把确定性错误先过滤掉，再交给 LLM。

**Q18: 为什么不用代码规则替代 Critic Agent？**

代码规则能覆盖“已知的已知”，但无法覆盖“未知的错误”。例如：发现“建议买入但合理价值远低于现价”需要语义理解，规则写不全。Critic 用 LLM 做语义层审查，与代码层形成互补。

**Q19: Rule-based Audit 和 LLM Critic 的区别是什么？**

| 维度 | Rule-based Audit | LLM Critic |
|------|------------------|------------|
| 检查内容 | 数字、格式、阈值、单位 | 语义、逻辑、行业事实 |
| 速度 | 快 | 慢 |
| 确定性 | 100% | 概率性 |
| 可扩展性 | 加规则即可 | 改 Prompt 即可 |
| 典型场景 | 评分合计、PE 存在性、情景单调性 | 投资逻辑矛盾、过度自信、行业描述 |

**Q20: 哪些检查适合代码完成，哪些适合 LLM 完成？**

代码：加减乘除、评分合计、PE/EPS 范围、单位转换、数据缺失、时间一致性。LLM：投资逻辑是否闭环、定性描述是否准确、风险是否遗漏、措辞是否过度自信。

**Q21: 为什么评分卡使用代码生成，而不是让 LLM 打分？**

金融领域要求一致性和可解释性。代码打分可复现、可调参、可审计；LLM 打分可能因 Prompt 温度、模型版本不同而变化。用代码打底，LLM 只能在行业前景、资金认可两个主观维度微调。

**Q22: 为什么估值计算不能完全交给 LLM？**

LLM 会算错 EPS×PE、混淆单位、把悲观价写得比基准价高。FinBrain 把估值框架选择交给 LLM，但具体数字（行业 PE、质量乘数、安全边际、合理价值）由代码计算，并做算术校验。

**Q23: 你的系统如何减少 LLM 幻觉？**

五层防御：1) 工具隔离，Analyst 无 tools；2) RAG 注入行业模板；3) 代码后处理强制覆盖客观分数；4) Critic 反向检查；5) Audit Engine 做最终校验。LLM 不能调用工具编造数据，也不能随意改写数字。

**Q24: 如何保证最终报告中的数字可信？**

所有数字来自工具返回或代码计算，不是 LLM 生成。Reporter 渲染时，盈利能力/成长性/财务健康/估值合理/合理价值/安全边际等关键数字被代码强制覆盖。Critic 和 Audit 检查这些数字是否自洽。

**Q25: 如果 LLM 编造了一个财务指标，你如何发现？**

如果 LLM 在文本中编造数字，它不会被代码后处理覆盖（因为代码只覆盖结构化 JSON 字段）。但 Critic 会检查文本数字与 JSON 字段的一致性，Audit 会检查文本中的数值是否合理。发现矛盾会触发 Repair 或降级。

**Q26: 你的系统有没有数据溯源机制？**

有。每个工具调用记录返回时间、数据源、状态，写入 `processing_log`。最终报告里显示 `[调用证据]` 和 `[审计校验]`，用户能看到数据来自哪个接口、是否命中缓存、是否有告警。

**Q27: 为什么需要记录 Tool Calling Trace？**

Trace 是 Debug 和可解释性的基础。没有 Trace，无法回答“这个数字哪来的”“为什么评分这么低”“为什么没查到这个公告”。Trace 也是未来自动评估 Agent 效果的输入。

**Q28: Execution Trace 对用户有什么价值？**

让用户知道系统不是黑箱。Trace 展示每个 Agent 执行了什么、耗时多少、调用了哪些数据源、发现了哪些问题。用户质疑结论时，可以回到 Trace 看依据。

**Q29: 如果用户质疑某个结论，你如何解释来源？**

打开 Trace 中对应的步骤：例如“估值合理 8 分”来自 `scoring.py` 的 `compute_investment_rating`，行业 PE 锚点 32 倍来自 `scoring.json` 的 IT 服务条目，数据来自东财接口。结论 = 数据源 + 计算链 + 模型判断，每层都可拆解。

**Q30: 你的系统如何做到 Explainable AI？**

三层可解释：1) 数据层，每个数字有工具来源；2) 计算层，评分/估值公式透明；3) 推理层，投资逻辑链要求“行业趋势→公司优势→财务表现→股价影响”闭环。最终报告还包含 `[估值明细]`、`[审计校验]`、`[框架分歧]` 等透明段落。

### 数据与 RAG

**Q31: 你的数据来源有哪些？**

行情（新浪、东财）、财报（东财 RPT_DMSK + RPT_FCI 快源）、估值（同花顺）、行业分类（东财）、公告（东财 np-cnotice）、资金流向（东财）。默认全免费公开接口。

**Q32: 为什么选择这些财务数据接口？**

因为项目定位是个人/小团队可用，优先零成本、无需授权的数据源。东财接口覆盖 A 股财报、行情、公告较全；新浪行情稳定。免费接口是约束，也是设计挑战。

**Q33: 免费 API 的数据质量如何保证？**

1) 多源交叉验证（如财报快源与公告快报互校）；2) TTL 缓存避免重复请求；3) 工具返回异常时标记 N/A 而不是崩溃；4) Audit Engine 检查数据一致性；5) 用户可见 `[调用证据]`，能人工复核。

**Q34: 如果财报 API 返回错误数据怎么办？**

工具层会捕获异常并返回 `{"error": ...}`。Data Agent 会标记该字段缺失。如果关键数据缺失，会触发降级（如禁用某维度评分）。Critic 也会检查数据完整性，提示“数据不足，谨慎参考”。

**Q35: 如何验证财务数据的一致性？**

代码层检查：营收与利润增速是否同向、归母与扣非差距是否过大、现金流与利润是否背离、资产负债率是否在合理区间。Critic 层检查：投资逻辑是否与财务数据一致。

**Q36: 为什么财报分析需要 RAG？**

因为 LLM 训练数据有截止日期，且不同行业估值逻辑不同。RAG 把“半导体封测怎么估值”“周期股为什么不能看单季利润”等行业知识按需注入，避免模型用通用模板硬套。

**Q37: 你的 RAG 知识库存储什么内容？**

四类：会计准则（如收入确认、商誉减值）、行业模板（半导体、IT 服务、电网信息化等）、交易策略（趋势/价值/妖股模式）、游资风格。未来可扩展 ESG、机构持仓解读等。

**Q38: 为什么不直接把所有财报文本输入 LLM？**

上下文窗口不够，且财报冗余信息多。RAG 只检索与分析主题相关的行业模板和会计准则，降低 token 成本，提高信息密度。

**Q39: 你的 RAG 检索流程是什么？**

用户输入股票 → 获取行业分类 → 用行业名作为 query 检索行业模板 → 用具体问题（如“现金流异常”）检索会计准则 → 把检索结果注入 Analyst Prompt。

**Q40: Query 是怎么生成的？**

Query 主要来自行业分类字段（如“半导体”）和当前分析主题（如“经营现金流/扣非净利润”）。不是让 LLM 自由生成 query，而是结构化字段 + 关键词组合，保证检索可预测。

**Q41: 你的 RAG 是静态知识库还是动态检索？**

是静态知识库 + 动态检索。知识库内容用户可上传更新，但检索是在每次分析时动态发生。目前不是动态更新知识库（如自动爬研报），但预留了扩展点。

**Q42: 为什么需要动态 RAG？**

因为不同公司需要不同知识。分析银行股需要“净息差、拨备覆盖率”，分析半导体需要“CAPEX、稼动率、先进封装”。固定知识库无法按需加载，动态 RAG 按行业/问题检索最相关内容。

**Q43: 动态 RAG 如何判断需要检索什么？**

目前按行业分类硬路由：每个行业模板对应一个检索 query。未来可加入 LLM 意图识别，让 Agent 根据用户问题决定检索“估值方法”“风险指标”还是“催化剂”。

**Q44: 如果检索到无关行业知识怎么办？**

1) 按行业分类过滤，减少误召回；2) 设置相似度阈值；3) 在 Prompt 中提示“如以下知识与当前公司无关可忽略”。Critic 也会检查行业描述是否与公司实际匹配。

**Q45: 如何评价你的 RAG 效果？**

目前主要靠端到端报告质量评估：RAG 是否命中了相关行业模板、报告中的行业描述是否准确、是否避免了通用套话。未来可补 recall@k、NDCG 等检索指标，以及人工标注相关性。

**Q46: 如何优化 RAG Recall？**

1) 换更强的中文 embedding（如 bge-large-zh）；2) 增加行业同义词映射；3) 按粗粒度→细粒度分层检索；4) 对过短 chunk 做合并；5) 加入 query 扩展（如“半导体”扩展出“封测、晶圆、先进封装”）。

**Q47: 为什么不用 Fine-tuning 替代 RAG？**

Fine-tuning 让模型“记住”知识，但知识更新成本高，且模型仍可能幻觉。RAG 把知识外置，更新只需改文档，不需要重新训练。金融领域知识更新快（政策、会计准则变化），RAG 更适合。

**Q48: 金融领域为什么更适合 RAG？**

金融知识结构化、时效性强、行业差异大。RAG 能把行业估值模板、会计准则、公司特异性信息按需注入，而不需要模型记住所有领域知识。

**Q49: 财报分析知识库应该如何设计？**

按“行业模板 + 会计规则 + 风险清单 + 估值框架”四维组织。每类模板有明确适用场景、禁忌事项、典型反例。例如半导体模板会强调“CAPEX 周期”“稼动率”“先进封装技术路线”。

**Q50: 为什么行业模板比通用知识库更有效？**

因为 LLM 默认用通用财务分析框架，会误判行业特性。例如“营收集中在四季度”对普通公司不重要，但对电网信息化公司是核心季节性特征。行业模板把这些先验知识直接注入，避免模型用错框架。

### 财务分析与估值

**Q51: Analysis Agent 输入是什么？**

结构化财务数据（营收、利润、现金流、毛利率、ROE 等）、行业分类、RAG 检索到的行业模板、近期公告、当前股价/估值、评分卡 JSON。

**Q52: Analysis Agent 输出是什么？**

结构化 JSON，包含：公司画像、商业模式、护城河、行业周期、成长逻辑、催化剂、风险、情景估值、投资评级、操作建议等。

**Q53: 为什么投资逻辑需要单独生成？**

投资逻辑是“叙事”，需要 LLM 把数据串成因果链。但叙事必须建立在结构化数据之上，不能凭空生成。单独生成便于 Critic 检查“数据是否支持结论”。

**Q54: 如何避免投资逻辑变成“AI 胡编故事”？**

1) 强制逻辑链模板：行业趋势→公司优势→财务表现→股价影响；2) 每个论点必须引用具体财务数据；3) 必须有证伪条件；4) Critic 检查逻辑是否与数据一致；5) 代码预检发现数据与结论冲突时强制标注框架分歧。

**Q55: 你的系统如何验证投资逻辑链？**

Critic 检查逻辑链是否闭环：行业趋势是否真实存在、公司优势是否可复制、财务数据是否支撑优势、估值是否反映市场预期。缺失任何一环都标记为逻辑缺陷。

**Q56: 为什么投资逻辑需要：行业趋势 → 公司优势 → 财务表现 → 股价影响？**

这是“从宏观到微观、从定性到定量”的验证链。缺少行业趋势，优势可能是孤例；缺少公司优势，财务表现可能只是周期；缺少财务数据，优势无法量化；缺少股价影响，投资价值无法落地。

**Q57: 如何判断一个竞争优势是否真实？**

代码层看毛利率是否高于行业、ROE 是否稳定、现金流是否匹配利润。LLM 层分析护城河来源（成本、品牌、网络效应、牌照）和可复制性。两者结合：高毛利 + 稳定 ROE + 可复制壁垒 = 真优势。

**Q58: LLM 如何分析护城河？**

Prompt 要求 Analyst 从“成本优势、规模效应、品牌/客户粘性、技术壁垒、牌照/渠道垄断”五个维度打分，并结合财务指标（毛利率、ROE、营收集中度）论证。避免空泛说“公司有护城河”。

**Q59: 财报中的定性信息如何结构化？**

通过 Prompt 强制输出固定 JSON 字段：护城河来源、行业周期阶段、核心催化剂、主要风险、证伪条件。LLM 的定性判断必须填入这些字段，才能被下游 Critic 检查。

**Q60: 管理层、客户、行业信息未来如何加入？**

通过 datasource tier 的 PREMIUM 插槽扩展：管理层画像、机构持仓、产业链图谱、ESG 治理。每个插槽用 `@premium_slot` 装饰器，等级不足时自动回退，不破坏主流程。

**Q61: 如果增加 ESG、机构持仓数据，需要改哪些模块？**

1) `datasource_tier.py` 增加插槽函数；2) `tools.py` 增加对应工具；3) `ANALYST_PROMPT` 增加分析维度；4) `scoring.json` 可能增加 ESG 评分权重；5) 测试用例补充。主流程节点不变。

**Q62: 为什么需要独立 Valuation Agent？**

估值不仅是算术，还需要选择框架（PE/PB/EV/EBITDA/PEG）。不同公司类型适用不同方法。Valuation Agent 负责框架选择，具体计算由代码完成。独立后便于审计和迭代。

**Q63: 为什么 PE 估值不能直接套行业 PE？**

行业 PE 只是中枢。同行业内公司质量差异大：ROE 高低、成长性、资本开支强度、周期位置都会让合理 PE 偏离中枢。直接套行业 PE 会忽略公司特异性。

**Q64: 为什么长电科技不能简单按照成长股估值？**

长电科技有半导体周期属性，利润随稼动率波动，不是纯成长。用成长股 PE 会高估周期高点、低估周期低点。系统判定为“周期复苏 + 成长叠加”，用正常化利润 + 周期折价 + 成长溢价综合估值。

**Q65: 你的公司分类为什么只有“周期复苏型”不够？**

很多公司同时具备多重属性。单标签会丢失信息。例如长电科技既是周期股又有 AI 成长属性，单标签“周期复苏”会忽略成长溢价，导致估值偏低。

**Q66: 为什么要改成混合权重分类？**

混合权重能同时表达“周期、成长、价值、主题”等多重属性。例如“50% 周期制造 + 30% 成长制造 + 20% AI 主题”比单标签更贴近真实公司画像，估值框架也能混合加权。

**Q67: “50%周期制造+30%成长制造+20% AI主题”的依据是什么？**

来自 Classifier 对公司财务指标（ROE 波动、增速趋势、CAPEX 强度）的归一化计算。它不是 LLM 拍脑袋，而是代码根据历史财务数据自动给出的权重。

**Q68: 权重应该由人工设定还是模型学习？**

目前由人工规则 + 财务指标推导。人工设定可解释、可审计；模型学习需要大量标注数据，且金融样本少。未来可以半监督：代码给初值，LLM/Critic 微调，但必须有上下限和解释。

**Q69: 如何避免估值模型主观化？**

1) 权重由财务指标计算，不是 LLM 写；2) 每个折价/溢价因子都有明确公式和阈值；3) 最终估值透明展示为 PE 折价链；4) Critic 和 Audit 检查估值合理性。

**Q70: 为什么需要 PE 折价链？**

让估值从黑盒变成白盒。用户能看到“行业 PE 40 倍 → 周期折价 -30% → ROE 不足 -20% → CAPEX 压力 -15% → 成长溢价 +20% = 最终 22 倍”。每个因子都有依据，便于质疑和迭代。

**Q71: 为什么行业 PE 40 倍最后折算成 22 倍？**

因为行业中枢是给“普通公司”的，具体公司可能处于周期高点、ROE 低于行业、资本开支过大。折价链把公司特异性量化，得到合理 PE，避免简单套用行业倍数。

**Q72: 折价因子应该由 LLM 判断还是代码计算？**

框架由 LLM 选择（判断周期/成长/价值），具体折价幅度由代码根据规则计算。这样既利用 LLM 语义理解，又保证可复现性。

**Q73: ROE 为什么影响估值？**

ROE 反映资本回报能力。同样 PE 下，高 ROE 公司更值钱。系统用 ROE 质量系数（0.3x–1.6x）调整 PE：ROE 高且稳定 → 溢价；ROE 低或波动大 → 折价。

**Q74: 资本开支为什么应该影响估值？**

高 CAPEX 意味着利润含金量低，自由现金流被挤压。系统用 CAPEX/CFO 比例作为质量因子，CAPEX 过高会降低合理 PE，并触发 FCF 预警。

**Q75: 为什么高增长公司也可能不值得投资？**

如果高增长来自周期反弹、一次性订单或并表，不可持续；如果估值已经透支未来增长（高 PE + 高 PEG），也没有安全边际。系统会区分“真成长”和“伪成长”。

**Q76: PEG 为什么不能简单使用？**

PEG 需要“可持续增速”。单季暴增、周期高点、并表基数都会扭曲增速。系统要求用 2–3 年复合增速或机构一致预期，否则标注“PEG 暂不可用”。

**Q77: 周期股为什么应该使用正常化利润？**

周期股利润波动大，单年利润不能代表真实盈利能力。用周期高点利润会高估，用周期低点会低估。正常化利润（多年平均或中性周期利润）更能反映真实价值。

**Q78: 为什么 EV/EBITDA 适合半导体封测行业？**

半导体封测是重资产行业，折旧大、资本开支高。净利润容易被折旧和财务费用扭曲，EBITDA 更能反映经营现金流能力。EV 剔除了资本结构影响，适合跨公司比较。

**Q79: 什么时候使用 PE，什么时候使用 PB，什么时候使用 EV/EBITDA？**

| 方法 | 适用场景 |
|------|----------|
| PE | 盈利稳定、利润有代表性的公司 |
| PB | 银行、保险等净资产驱动的行业，或亏损公司 |
| EV/EBITDA | 重资产、高折旧、资本开支大、利润被非经营因素扭曲的公司 |

**Q80: 三种估值方法结果冲突怎么办？**

系统会呈现框架分歧：价值框架（PE 锚点）vs 趋势框架（成长溢价）。不把冲突压平，而是告诉用户两套逻辑的假设和适用范围，让用户自己判断。

**Q81: 如何融合多个估值结果？**

目前用“主估值 + 情景估值”双轨。主估值来自代码规则（PE/质量乘数/EPS）；情景估值来自 LLM 对悲观/基准/乐观假设的判断，但结果由代码校验。最终报告同时展示两者，并解释差异。

**Q82: 你的情景估值如何生成？**

Analyst 输出悲观/基准/乐观的 EPS、PE、假设、概率；代码层校验算术（EPS×PE=价格）、单调性（悲观≤基准≤乐观）、概率合计；然后代码重算概率加权价值。

**Q83: 悲观、中性、乐观概率怎么确定？**

由 LLM 根据公司确定性、行业周期、催化剂可兑现程度判断。但系统要求概率合计为 100%，并在代码层归一化。未来可以引入历史回测校准概率。

**Q84: 为什么情景估值不能完全依赖 LLM？**

因为 LLM 会算错 EPS×PE、概率合计不为 100%、价格排序颠倒。代码层强制校验并修正，保证数值自洽。

**Q85: 如何验证估值模型合理性？**

1) 回测：历史报告的交易建议用历史 K 线验证胜率；2) 横向对比：同行业公司估值是否在同一区间；3) 异常检测：合理价值与当前价差距过大时触发框架分歧；4) 专家评估：请有行业经验的人判断估值逻辑是否成立。

**Q86: 为什么不能只看净利润？**

净利润容易被非经常性损益、会计估计、并表范围操纵。扣非净利润、经营现金流、自由现金流更能反映主业真实盈利能力。

**Q87: 为什么需要 FCF？**

FCF 是“利润扣除维持性资本开支后的真现金”。一家公司净利润很高但 FCF 为负，说明它赚的钱都变成了设备、厂房，股东拿不到真金白银。

**Q88: 为什么长电科技 CFO 很高但是 FCF 为负？**

因为半导体封测行业资本开支巨大，经营现金流（CFO）虽然高，但扣除 CAPEX 后自由现金流（FCF）为负。这说明公司处于扩张期，利润含金量不高。

**Q89: 经营现金流优秀是否代表现金流健康？**

不一定。如果 CAPEX 远高于 CFO，公司仍在烧钱扩张。还要看 FCF、应收账款变化、资本开支方向。系统用 FCF 为负 + CAPEX/CFO>0.3 自动预警。

**Q90: CAPEX 对半导体公司的意义是什么？**

CAPEX 决定技术路线和产能规模。先进封装（如 Chiplet）需要大量资本投入，CAPEX 不足会落后，CAPEX 过高会拖累 ROE 和 FCF。系统把它作为估值折价因子和风险信号。

**Q91: 为什么资本开支过大会降低 ROE？**

因为 ROE = 净利润 / 净资产。大规模 CAPEX 增加净资产（分母），但产能投产和利润释放有滞后，短期内净利润（分子）增长慢，ROE 被拉低。

**Q92: 如何自动发现财报中的风险？**

代码层规则：应收账款/净利润 > 3、经营现金流为负、归母扣非差距 > 30pp、FCF 为负且 CAPEX/CFO 高、毛利率连续下滑。LLM 层：Critic 检查风险描述是否遗漏或轻描淡写。

**Q93: 你的系统如何判断利润质量？**

看扣非净利润/归母净利润比例、经营现金流/净利润比例、应收账款增速 vs 营收增速、毛利率稳定性。比例异常或趋势背离都标记为利润质量风险。

**Q94: 如何判断收入增长是否可持续？**

1) 看收入是否来自核心客户/产品；2) 看订单/合同负债；3) 看行业周期位置；4) 看毛利率是否同步提升。如果收入增长但毛利率下降，可能是降价抢市场，不可持续。

**Q95: 如何分析毛利率变化原因？**

Prompt 要求 Analyst 从“产品结构变化、原材料价格、产能利用率、竞争格局、汇率/政策”五个维度分析。并结合行业模板判断是周期因素还是公司竞争力变化。

**Q96: 如果公司收入增长但现金流下降，你怎么看？**

可能是赊销增加、应收账款积压、库存占用资金。这是危险信号，说明收入增长没有转化为现金。系统会注入回款风险警告。

**Q97: 如果利润下降但现金流增加，如何解释？**

可能是一次性资产处置、压缩库存、收紧信用政策、前期资本开支进入收获期。需要区分是一次性因素还是经营改善。Critic 会检查叙事是否与数据一致。

**Q98: 如何分析周期行业？**

看行业产能利用率、库存周期、价格指标、资本开支周期、龙头公司毛利率变化。系统用行业模板提示周期分析框架，并要求给出当前周期位置。

**Q99: 如何判断公司处于周期哪个阶段？**

用财务指标组合：营收增速、毛利率、ROE、库存周转、CAPEX 强度。如果营收和毛利率同时回升但 CAPEX 还没大规模启动，可能是复苏早期；如果 CAPEX 高峰已过、利润率开始下滑，可能是周期顶点。

**Q100: 如何避免把周期反弹误认为成长？**

用混合权重分类：如果 ROE 波动大、CAPEX 强度高、行业产能周期明显，就赋予周期属性权重。估值时用正常化利润和周期折价，而不是直接用高增速给高 PE。

### Critic 深挖

**Q101: Critic Agent 的输入输出格式是什么？**

输入：Analyst/Valuation 输出的 JSON、原始财务数据、行业模板、评分卡、当前股价。输出：结构化修复清单 `[CRITIC_FIXES]`，每项包含 `字段`、`问题描述`、`严重等级`、`修正建议`。不直接改报告，只提清单。

**Q102: Critic 如何发现“投资逻辑矛盾”？**

Critic 检查 JSON 中“投资逻辑→风险→评级→操作建议”是否一致。例如：逻辑说“公司处于周期高点、估值过高”，但评级是 BUY，操作是“重仓”，就是矛盾。

**Q103: Critic 为什么能够发现：“估值过高但建议持有”的矛盾？**

因为 Critic 同时读取估值结果（合理价值、当前价、安全边际）和操作建议（评级、仓位、止损）。如果合理价值远低于当前价但评级不是 SELL，代码层和 LLM 层都会标记。

**Q104: Critic 是重新分析公司还是检查已有分析？**

Critic 不重新分析公司，只检查已有分析。这是“验证者”而非“替代者”角色。重新分析会引入新的偏见，只做检查才能保持独立性。

**Q105: Critic 是否需要访问工具？**

Financial Critic 需要访问工具做代码预检（FCF、PE/ROE 匹配）。Logic/Industry Critic 主要依赖已有数据和 LLM 语义判断，不需要新增工具调用。

**Q106: Critic 发现错误后是否自动修改报告？**

不直接修改。Critic 输出修复清单，交给 Repair Agent 修改。这样分离“找问题”和“改问题”的角色，避免 Critic 既当裁判又当运动员。

**Q107: 为什么你的 Critic 不直接生成最终报告？**

如果 Critic 直接生成报告，就失去了反向验证的意义。Critic 的产出是“问题清单”，Repair 根据清单修改，Audit 再验证，形成闭环。

**Q108: Critic Agent 是否可能产生新的幻觉？**

可能。Critic 也是 LLM，可能误报或漏报。缓解方法：1) Financial Critic 先做代码预检，减少主观判断；2) 代码层 Audit 二次过滤；3) 动态重试预算，避免过度修改；4) 用户可见审计摘要，可人工复核。

**Q109: 如何评价 Critic 的准确率？**

用离线 benchmark：构造已知错误的报告，看 Critic 能否检出。统计精确率、召回率、误报率。还可以用真实报告的人工质检结果做对照。

**Q110: Critic 和 Human Analyst 的角色区别是什么？**

Critic 是自动化“找茬”角色，覆盖规则明确、重复性高的检查。Human Analyst 负责判断复杂商业逻辑、行业洞察、非结构化信息。Critic 把人从“查错”中解放出来，让人专注于“判断”。

### 工程实现

**Q111: 为什么选择这个技术栈？**

LangGraph：固定流程 + 状态持久化；LangChain：Tool/Model 统一接口；FastAPI：数据服务；Streamlit：快速前端；ChromaDB：本地 RAG；Redis：共享缓存。选型原则是“够用、可本地运行、零运维成本”。

**Q112: Agent 调用链如何实现？**

LangGraph StateGraph 定义节点和边。每个节点是 Python 函数，接收 `state`、返回 `dict`。`state` 通过 Checkpointer（SqliteSaver）持久化，支持跨轮次恢复。

**Q113: 如何管理多个 Agent 状态？**

用 LangGraph 的 `State` 对象 + SqliteSaver。每个节点返回的 dict 会与 state 合并，普通 list 字段需要节点自己读取旧值追加。`thread_id` 区分不同用户会话。

**Q114: Agent 输出如何结构化？**

强制 JSON 输出，用 `Pydantic`/`TypedDict` 定义 schema。LLM 输出先用 `raw_decode` 或正则提取 JSON，再校验字段。缺失字段会触发重试或注入默认值。

**Q115: 如何保证 JSON Schema 正确？**

1) Prompt 中给出完整 JSON 示例；2) 使用 `temperature=0` 降低随机性；3) 代码层解析后校验字段类型和必填项；4) 错误时重试一次；5) 最终用默认值兜底，避免崩溃。

**Q116: LLM 输出格式错误怎么办？**

先用 `raw_decode` 从文本中提取第一个有效 JSON；如果失败，用正则匹配代码块；再失败，返回降级 JSON 或错误提示，并记录到 `processing_log`。不会直接崩溃。

**Q117: 如何设计 Retry 机制？**

工具层：同一工具连续失败 3 次熔断；LLM 层：JSON 解析失败重试 1 次；审计层：发现问题最多重试 3 次。重试有预算，避免无限循环。

**Q118: 如何降低 API 调用成本？**

1) TTL 缓存（30s 股价/30min 财报/1h 行业）；2) 工具调用去重；3) Analyst 节点单次 invoke，不用 ReAct；4) 小模型处理简单任务，大模型只用于复杂分析；5) Prompt 压缩和上下文摘要。

**Q119: 哪些步骤可以缓存？**

数据工具结果：股价、财报、估值、行业、公告。分析结果：同一只股票短期内复用。RAG 检索结果：知识库不变时可缓存。Prompt 模板：热加载但可缓存。

**Q120: 如果同时分析100家公司，系统如何扩展？**

远程模式：FastAPI 数据服务 + Redis 共享缓存 + scheduler 预取。分析请求可以并行（ThreadPoolExecutor）。LLM 调用是瓶颈，可引入异步队列或批量请求。

**Q121: 如何设计异步执行？**

分析请求可拆分为数据获取（I/O 密集型，async）和 LLM 推理（可并发）。使用 FastAPI async + Redis + 任务队列。LangGraph 节点目前同步，可逐步改造成 async。

**Q122: 如何保存历史分析结果？**

当前通过 Checkpointer 保存 graph 状态。未来可写入数据库（PostgreSQL/SQLite），按用户 + 股票 + 时间索引，支持查询和对比。

**Q123: 如何支持用户查看过去报告？**

在 Streamlit 前端增加历史报告页面，按 `thread_id` 或用户 ID 查询数据库。展示报告列表、生成时间、关键评分、可点击查看完整报告和 Trace。

**Q124: 如何监控 Agent 性能？**

记录每个节点耗时、LLM token 消耗、工具调用次数、缓存命中率。日志写入文件或发送到监控。关键指标：端到端延迟、成功率、Critic 检出率、Audit 重试率。

**Q125: 如何定位一次分析失败原因？**

查看 `processing_log` 和 Execution Trace：哪个节点失败、哪个工具返回 error、哪次 LLM 输出格式错误、哪条审计规则触发。Trace 是 Debug 的核心。

### 面试官挑战问题

**Q126: “你的系统是不是只是 Prompt Chain？”**

不是。Prompt Chain 是线性调用 LLM，没有反馈和修正。FinBrain 有工具调用、代码层计算、Critic 反向检查、Repair 自动修正、Audit 最终校验，形成闭环。每个节点不是简单 LLM，而是有明确输入输出和状态管理。

**Q127: “为什么不用 GPT-5 直接分析？”**

GPT-5 再强也会幻觉、也会算错数、也会用错行业框架。金融分析需要可复现、可审计、可追溯。FinBrain 用代码负责确定性计算，用 LLM 负责语义推理，两者互补。模型能力提升会让 LLM 部分更好，但护栏和代码层仍然必要。

**Q128: “你的 Agent 哪里体现智能？”**

智能体现在：1) 自主决定调用哪些工具获取数据；2) 从数据中提炼投资逻辑；3) 识别自身输出中的矛盾和错误；4) 根据修复清单定向修改。不是无约束地“思考”，而是在结构化流程中做有边界的有用决策。

**Q129: “如果 LLM 能力继续提升，你这个系统还有价值吗？”**

有价值。LLM 能力提升会让 Analyst/Valuation/Critic 的推理更强，但系统的价值在于：数据工程、代码化计算、审计机制、多 Agent 协作、可解释性。这些不是模型能力本身，而是工程和组织能力。

**Q130: “你的估值模型是不是人工规则？”**

部分是。核心公式（行业 PE × 质量乘数 × EPS）是人工设计的规则，但输入（ROE、增速、CAPEX）是真实数据，权重由 Classifier 代码计算。规则提供可解释性和稳定性，LLM 负责框架选择和叙事补充。未来可以用数据部分优化参数，但规则骨架保留。

**Q131: “既然用了规则，为什么还需要 Agent？”**

规则只能回答“值多少钱”，无法回答“为什么”。Agent 负责把数据、行业知识、投资逻辑串成故事，识别规则覆盖不到的风险，处理规则边界案例。规则是骨架，Agent 是血肉。

**Q132: “你的系统最大的创新点是什么？”**

最大创新是“确定性计算 + 概率性推理的强制边界”。不是简单用 LLM 写研报，而是用代码固定财务指标和评分，用 LLM 做叙事和审查，用 Critic 做反向验证，用 Audit 做最终兜底。这种设计在金融这种对数字敏感、对一致性要求高的领域特别有效。

**Q133: “你的系统和同花顺、雪球的区别是什么？”**

同花顺/雪球是信息终端，提供数据但不产生投资逻辑。FinBrain 是研究 Agent，主动抓取数据、生成分析逻辑、检查一致性、输出可审计报告。它不会替代行情软件，而是帮助用户从数据到决策。

**Q134: “如果让你继续迭代三个月，你会做什么？”**

1) 接入更多数据源（Wind/Tushare Pro 插槽、ESG、机构持仓）；2) 完善回测引擎，加入夏普、最大回撤；3) 加入 Human-in-the-loop 审批节点；4) 构建离线 benchmark，量化 Agent 相对普通 GPT 的提升；5) 优化 RAG  embedding 和 chunk 策略。

**Q135: “你认为当前系统最大的不足是什么？”**

1) 免费数据源不稳定且数据质量有限；2) 回测引擎只检测信号触发，没有策略收益评估；3) 估值模型参数主要靠人工设定，未做历史回测校准；4) 没有 Human-in-the-loop；5) 单元测试覆盖还不够细。

**Q136: “如果加入更多数据，你优先加入什么？”**

优先加入机构一致预期（盈利预测、目标价）和产业链/订单数据。一致预期能修复当前报告“预期陈旧”问题；产业链/订单数据能验证公司成长逻辑，而不只是看历史财报。

**Q137: “如何让这个系统达到机构研究员水平？”**

需要三层：1) 数据层接入 Wind/Choice 等付费数据，覆盖一致预期、产业链、管理层；2) 模型层引入更专业的金融模型（DCF、分部估值、产业链模型）；3) 人机协作层加入研究员反馈闭环，让 Critic 从专家标注中学习。

**Q138: “如何验证你的分析比普通 GPT 更好？”**

构建 benchmark：选一组股票，用 FinBrain 和普通 GPT 同时出报告，人工盲评以下维度：数据准确性、逻辑一致性、风险覆盖、估值合理性、操作建议具体性。统计胜率。也可以用回测验证交易建议的表现。

**Q139: “你的系统有没有实际投资价值？”**

当前定位是研究辅助工具，不是直接给出买卖信号。它帮助研究员快速生成分析框架、发现数据矛盾和风险点，但投资决策仍需人工判断。随着数据源和回测完善，可以逐步向量化策略辅助演进。

**Q140: “如果 AI 分析错了一家公司，责任如何处理？”**

系统层面：明确免责声明，报告标注“不构成投资建议”，保留数据溯源和审计摘要。产品层面：保留 Human-in-the-loop，关键决策人工确认。技术层面：持续优化 Critic 和 Audit，降低错误率。责任最终由使用者承担，系统只是辅助工具。
