## 零、设计哲学（面试必答）

**Q: 为什么不做实时PE/PB/目标价的精确计算？**

> 这是刻意为之的设计决策。FinBrain 定位为基本面研究 Agent，不是实时行情终端。免费 API 拿不到 Level-2 数据，硬追精确 PE 会在最不该犯错的地方犯错——之前出现过把去年净利润当今年一致预期、PE 差 30% 的情况。所以所有 PE/PB/市值标注"基于财报，非实时行情"，砍掉了精确买入价、期望收益、仓位百分比。Agent 的算力应该花在它能做好的事情上：三年毛利率趋势、现金流为何与利润背离、护城河能否持续。

**Q: 怎么防止 LLM 乱改数据？**

> 分层防御。第一层：工具权限隔离，Analyst 无 tools。第二层：Reporter 后处理强制覆盖——`_fix_and_decide` 对每只股票调 `calculate_scores`，覆盖盈利能力/成长性/财务健康/估值合理四个维度 + 估值水位 PE/PB/市值。第三层：关键信号代码检查，不依赖 LLM"注意"——Q1 经营现金流为负自动注入风险警告。第四层：证伪条件——任何投资逻辑都有可验证的失败条件。

**Q: 你这个系统怎么部署？能支持多少用户？**

> 两种模式。本地模式：用户 `pip install` 后直接 `streamlit run`，全部在自己电脑上跑，天然单用户。远程模式：服务器跑 FastAPI + Redis + 定时预取调度器，客户端只做 LLM 推理。用户的 API Key 只在本地 `.env`，永不传到服务器。数据 API 是纯读缓存（毫秒级），配合 FastAPI 的异步并发 + Nginx 反向代理，轻松支撑数千并发。三层缓存（scheduler 预取 → Redis → Nginx proxy_cache）保证冷数据请求也不排队。

**Q: 你的 Agent 能做什么、不能做什么？**

> 能做：读懂财报（营收/利润/现金流趋势）、判断商业模式（护城河来源+复制难度）、行业周期定位、识别财务预警信号。不能做：精确预测股价、给出精确买卖点、替代交易终端。FinBrain 是行业研究员，不是交易员。

## 一、Agent 架构与工作模式

### 1. 你怎么理解 Agent？它和普通 LLM 应用最大的区别是什么？

**核心区别：Agent 能自主决策行动序列，普通 LLM 应用只能做单次推理。**

三个关键能力：

| 能力 | 说明 | FinBrain 实例 |
|------|------|-------------|
| **工具使用** | 调用外部函数/API 获取真实数据 | 13 个 `@tool`，LLM 自主决定调用哪个 |
| **记忆** | 跨轮次保留上下文 | SqliteSaver 持久化对话状态 |
| **规划** | 分解复杂任务为多步执行 | DataCollector→Analyst→Reporter 三节点流水线 |

### 2. Agent 常见的工作模式有哪些？

| 模式 | 核心思想 | 项目落地 |
|------|---------|---------|
| **ReAct** | Thought→Action→Observation 循环 | DataCollector 子 Agent 用 `create_agent` |
| **Plan-and-Execute** | 先生成计划再逐步执行 | 三节点流水线 |
| **Router** | 根据输入分类路由 | `_classify_request()` + 显式模式切换 |
| **Multi-Agent** | 多个专业化 Agent 协作 | DataCollector + Analyst + Reporter + Phantom |
| **Post-Processing** | 执行后代码修正 | Reporter 强制覆盖评分 + 计算投资评级 |

### 3. 项目里有没有用到现成框架？

**LangGraph + LangChain**。

架构分层：
```
编排层: LangGraph StateGraph (手写节点和边)
  └── 子Agent: LangChain create_agent (ReAct循环)
        └── Tool层: LangChain @tool 装饰器
              └── 数据层: tools.py (纯Python，零框架依赖)
              └── 决策层: scoring.py (纯Python，零LLM依赖)
```

选择 LangGraph 而非 AutoGen/CrewAI：我的 Data→Analyze→Report 流程固定，不需要 Agent 间"协商"。StateGraph 编译时确定图结构，执行路径可预测。Analyst 和 Reporter 无 tools，从框架层面保证权限隔离。

### 4. 投资评级是 LLM 算的还是代码算的？

**代码算的**。LLM 只写叙事（亮点/风险/逻辑链）。`scoring.py` 的 `compute_investment_rating()` 按公司类型动态赋权 + 算安全边际 + 出 BUY/HOLD/SELL。Reporter 的 `_fix_and_decide` 强制覆盖 LLM 可能乱改的分数和评级。同一只股票跑 100 次得分完全一致。

### 5. 怎么避免路径震荡（重复规划）？

1. **StateGraph 固定流水线** — 节点间不形成循环
2. **DataCollector 只跑一轮** — Prompt 明确"调完所有工具后汇总"
3. **max_tokens 限制** — 天然限制循环深度
4. **评分后处理覆盖** — 代码最终说了算

## 二、工具调用与 MCP

### 6. 13 个工具怎么管理？

`tools.py` 纯 Python 函数（零 Agent 框架依赖） → `agent.py` 的 `@tool` 装饰器薄包装。不同 Agent 注册不同工具集实现权限隔离。工具可独立测试、可同时给 LangGraph Agent、MCP Server 和 Streamlit 前端使用。

### 7. 工具调用 vs Function Calling vs 普通 Prompt

| 维度 | 普通 Prompt | Function Calling | Tool Use (LangChain) |
|------|-----------|-----------------|---------------------|
| LLM 输出 | 自由文本 | JSON `{name, arguments}` | JSON `{name, arguments}` |
| 谁执行 | 无 | 调用方代码 | 框架自动拦截+执行+回传 |
| 多轮 | 不支持 | 需手动循环 | 框架内置 ReAct |

### 8. 你怎么理解 MCP / Prompt / Harness？

| 概念 | FinBrain 实例 |
|------|-------------|
| **MCP** | `MCP_server.py`暴露 5 个 Tool 给 Claude Code |
| **Prompt** | `ANALYST_PROMPT`、`_FORMAT_MANDATORY`、策略热加载 |
| **Harness** | 工具分级、SqliteSaver、压缩、路由、降级、评分确定性 |

## 三、RAG 与知识库

### 9. 知识库是怎么搭的？

```
文档上传(PDF/DOCX/TXT) → 格式识别 → 解析器分发
  → 三级切片(段落→句子→字符, 500字/块, 50字重叠)
  → ONNX MiniLM 向量化 → ChromaDB 持久化
```

4 个知识库：会计准则(15条预置+用户上传)、行业研报、交易策略、游资风格。

### 10. Lost in the Middle 怎么处理？

1. 切片时加元数据标注（文档标题/章节/页码）
2. 分层检索：先搜粗粒度再搜细粒度
3. Prompt 显式提示优先使用靠前内容
4. 过滤过短的块（<20字符）

### 11. 怎么控制模型幻觉？

| 层 | 手段 |
|------|------|
| **RAG 层** | 每个 chunk 带 source 标注 |
| **Prompt 层** | "缺失数据标 N/A""依据栏引用具体数值" |
| **输出约束** | 固定 JSON Schema，缺失字段标 null |
| **工具层** | Analyst 无 tools，只能分析已有数据 |
| **代码层** | Reporter 强制覆盖客观维度分数 |
| **逻辑层** | 证伪条件——任何投资逻辑必须有可验证的失败条件 |

## 四、系统工程与性能

### 12. 针对可靠性与并发，做了哪些优化？

| 优化 | 方案 |
|------|------|
| **并发安全** | Portfolio `threading.Lock`，SqliteSaver 单例 |
| **资源管理** | `atexit` 注册清理，SQLite 连接不泄露 |
| **上下文管理** | 超阈值自动摘要压缩 |
| **容错** | 工具返回 `{"error": "..."}` 不崩溃，Agent 标 N/A 继续 |
| **评分确定性** | 代码后处理强制覆盖 LLM 输出 |
| **懒初始化** | 重资源 double-check locking，不在 import 时加载 |

### 13. 怎么提升模型响应速度？

| 方向 | 手段 |
|------|------|
| **架构层** | Analyst 节点不用 ReAct（单次 `invoke`） |
| **工具层** | TTL 分层缓存（30s股价/30min财报/1h行业） |
| **Prompt 层** | 精简 System Prompt，策略热加载按需注入 |
| **网络层** | 数据接口加缓存，避免重复 API 调用 |

### 14. 一个比较完整的 Prompt 包含哪些部分？

```
1. Role（角色）         → "你是 FinBrain 高级投资分析师"
2. Framework（框架）     → 十步分析链
3. Rules（规则）         → "纯JSON""多股票数组"
4. Context（上下文）      → 行业分析模板
5. Format（输出格式）     → JSON Schema
6. Quality（质量标准）    → 归母≠扣非/现金流≠含金量/证伪条件
7. Examples（示例）       → 正确vs错误的数组格式
8. Guardrails（护栏）    → "不构成投资建议"
```

## 五、投资决策引擎

### 15. 动态权重是怎么设计的？

5 种公司类型 × 6 维权重矩阵。价值型重估值(30%)+财务(25%)，成长型重成长(30%)+行业(20%)+壁垒(20%)，周期型重行业周期(30%)+估值(25%)。

安全边际 = 类型基准(20-50%) + ROE质量调整 + 负债率调整。

合理价值 = 行业PE基准 × ROE质量系数(6档, 0.3x~1.6x) × EPS。

买入区间 = 合理价值 × (1 - 安全边际)。全部代码计算，不依赖 LLM。

### 16. 十步分析链 vs 传统财务分析

传统工具只回答"公司财务健康吗"。FinBrain 回答"是否值得投资"：

公司画像 → 商业模式 → 护城河 → 行业周期 → 财务验证 → 成长逻辑 → 市场预期 → 情景估值 → 催化剂 → 投资决策

关键洞察：增加"市场预期拆解"模块——分析当前估值隐含的增长率、市场主要担忧、以及可能的预期差。股票投资的核心是找到市场预期与实际价值之间的差距。

## 六、场景设计与开放题

### 17. 用 Agent 实现 AI 面试程序怎么设计？

```
Candidate → [Question Agent] 出题 → [Evaluator Agent] 评估
  → [Follow-up Agent] 追问 → [Scoring Agent] 综合评分
```

### 18. 这个产品的缺点？

1. **单用户设计** — 全局单例 Agent，`st.session_state` 无法多租户
2. **免费数据源不稳定** — push2 间歇性断开，THS 鉴权依赖第三方库
3. **无自动化测试** — 工具和 Agent 都靠手动跑
4. **行业分类依赖 LLM** — 未做公司分类的代码化
5. **DCF 缺失** — 估值只有 PE 法，无现金流折现

### 19. Agent 后续怎么发展？

**容易落地**：客服（规则明确）、代码审查（结构化输入输出）、数据查询（Text-to-SQL+RAG）、合规审查（对照检查清单）。

**趋势**：Agent 从"单步工具调用"走向"多步规划+执行+反思"。模型能力提升会让 Agent 越来越可靠，但永远需要护栏。MCP 协议会标准化。

## 七、FinBrain 财报分析 Agent 专项面试问题

> 以下问题围绕 FinBrain 的 Multi-Agent 财报分析系统，按 **Agent 设计、数据/RAG、估值模型、财务分析、Critic、工程实现、开放题** 七个方向组织。每个答案控制在面试可展开的 2–4 句话。

---

### Agent 架构设计

**Q1: 你为什么要做一个财报分析 Agent？它解决了传统财报分析流程中的什么问题？**

传统研报生产周期长、依赖人工 Bloomberg/Wind 查询、不同分析师口径不一。FinBrain 把“读财报→算指标→写逻辑→审风险”压缩到分钟级，且用代码固定计算规则，保证同一家公司每次打分一致，解决人工分析的时效性、一致性和可复现性问题。

**Q2: 你的系统和直接让 GPT 分析一家公司的区别是什么？**

直接让 GPT 分析容易编造 PE、错把扣非当归母、给出无法验证的买入价。FinBrain 把数据抓取、指标计算、评分、估值、审计全部代码化，LLM 只负责叙事和推理；最终报告里的数字有工具调用 trace 和审计摘要，可追溯、可证伪。

**Q3: 为什么不用一个 LLM Prompt 完成财报分析？**

单 Prompt 任务太重，模型会遗漏规则、算错数字、混淆口径。拆成多个 Agent 后，每个角色职责单一、Prompt 更聚焦；中间状态可审计、可缓存、可重试；代码层能在每个节点做确定性修正。

**Q4: 你的系统为什么设计成 Multi-Agent，而不是单 Agent？**

财报分析需要三类能力：数据工程师（抓数）、行业分析师（叙事）、财务审计（查错）。单 Agent 同时承担这三种角色会互相干扰，且无法并行审查。Multi-Agent 让 Data/Analyst/Valuation/Critic 专业化，形成生成→质疑→修正的闭环。

**Q5: 你的 Agent 架构有哪些角色？每个 Agent 的职责是什么？**

| Agent | 职责 | 是否用 LLM |
|-------|------|:----------:|
| Data Agent | 并发抓取财报、行情、行业、公告 | 否（纯代码） |
| Classifier | 按 ROE/增速/CAPEX 给公司贴混合权重 | 否（纯代码） |
| Analyst | 写投资逻辑、风险、催化剂 | 是 |
| Valuation Agent | 选估值框架、输出 PE 折价链 | 是 |
| Critics（3路） | 检查逻辑/财务/行业问题 | 是 + 代码预检 |
| Repair | 根据 Critics 清单定向修改 JSON | 是 |
| Reporter | 渲染最终报告 | 是 + 代码后处理 |
| Audit Engine | 代码级 9 项检查 + FCF 预警 | 否（纯代码） |

**Q6: 为什么 Data Agent、Analysis Agent、Valuation Agent、Critic Agent 要拆开？**

数据抓取是确定性 I/O，不该让 LLM 决定；分析是概率性推理，需要 LLM 语义能力；估值需要框架选择 + 代码计算；Critic 需要反向验证。混在一起会导致 LLM 越权调用工具、数字被叙事污染、审计失去独立性。

**Q7: Agent 之间的数据流是什么？能画一下系统架构吗？**

```
用户输入 → Classifier → Data Agent（并发工具）
                   ↓
           Analyst（JSON：逻辑/风险/催化剂）
                   ↓
           Valuation Agent（JSON：PE 折价链/情景估值）
                   ↓
           Critics（Logic/Financial/Industry 三路并行）
                   ↓
           Repair Agent（修正标记字段）
                   ↓
           Reporter（渲染报告）
                   ↓
           Audit Engine（代码校验 + 强制覆盖）
                   ↓
           结构化报告 + Execution Trace
```

所有数据通过 LangGraph 的 `State` 对象传递，每个节点返回 dict 与 state 合并。

**Q8: 你的 Workflow 是固定流程还是动态规划？为什么？**

固定流程。财报分析任务的目标、输入、输出、关键步骤是稳定的：数据→分析→估值→审查→报告。固定流程更易调试、可预测、缓存友好。动态规划会增加不可控性和调试成本，对于确定性研究任务收益有限。

**Q9: 你的系统有没有 Planner Agent？为什么没有？**

当前没有。因为任务 DAG 已经固定，不需要 Planner 临时生成。Planner Agent 更适合工具种类多、任务目标开放、执行路径不确定的场景（如通用编程、复杂研究）。财报分析路径收敛，用 Planner 反而会让流程不稳定、增加延迟。

**Q10: 如果加入 Planner Agent，它解决什么问题？**

如果未来扩展到“用户问：对比三只光模块股票，给出组合建议”，任务路径会变化（单股 vs 多股、是否要加组合优化、是否要回测）。Planner 可以动态决定调用哪些工具、哪些 Agent、是否需要并行。

**Q11: 对于财报分析任务，Planner Agent 是否真的必要？**

目前不必要。固定流程已覆盖 95% 场景。Planner 的动态性更适合异常分支（如数据缺失时切换数据源、用户要求跨行业对比）。保留固定流程作为默认路径，Planner 只处理异常分支是更务实的做法。

**Q12: 你的 Agent 如何决定下一步执行什么任务？**

LangGraph 编译时确定边。每个节点返回后，框架按图结构进入下一个节点。没有运行时“Agent 自己决定下一步”的开放规划，但节点内部可以用 LLM 做微观决策（如 Critic 判断哪些问题需要修复）。

**Q13: 为什么你的流程不是 数据获取 → RAG → GPT → 报告，而是多个 Agent？**

因为“RAG → GPT → 报告”没有校验层。数字错了、逻辑矛盾、风险遗漏都无法被发现。Multi-Agent 在生成后插入 Critics、Repair、Audit 三层检查，把一次生成变成“生成→质疑→修正→验证”的闭环。

**Q14: 你的系统里面哪里体现了 Agent 的自主性？**

1. Data Agent 自主决定调用哪些工具（财报、行情、行业、公告按需并发）。
2. Analyst 自主从数据中提炼投资逻辑和风险。
3. Critic 自主识别文本和数字中的不一致。
4. Repair 自主决定如何修改被标记字段。自主性体现在“选择做什么”和“如何修正”，而不是无约束地规划任务。

**Q15: 如果去掉 Critic Agent，结果会有什么变化？**

报告质量会显著下降：估值过高但建议买入、现金流风险被忽略、情景估值倒挂、评级与操作矛盾等问题无法被发现。Critic 是质量守门员，去掉后系统退化为“数据 + 一次 LLM 生成”。

### Critic 与 Audit

**Q16: Critic Agent 为什么需要存在？**

因为 LLM 生成内容会自洽但错误。Critic 作为反向验证者，强制检查逻辑一致性、财务指标合理性、行业事实准确性。它是“自我纠错”机制，不是装饰。

**Q17: Critic Agent 是否调用 LLM？为什么？**

是。纯代码只能检查数字和格式（如 PE 范围、评分合计），但无法判断“投资逻辑是否过度自信”“行业竞争描述是否准确”。这类语义检查需要 LLM。但 Critic 里的 Financial Critic 会先跑代码预检，把确定性错误先过滤掉，再交给 LLM。

**Q18: 为什么不用代码规则替代 Critic Agent？**

代码规则能覆盖“已知的已知”，但无法覆盖“未知的错误”。例如：发现“建议买入但合理价值远低于现价”需要语义理解，规则写不全。Critic 用 LLM 做语义层审查，与代码层形成互补。

**Q19: Rule-based Audit 和 LLM Critic 的区别是什么？**

| 维度 | Rule-based Audit | LLM Critic |
|------|------------------|------------|
| 检查内容 | 数字、格式、阈值、单位 | 语义、逻辑、行业事实 |
| 速度 | 快 | 慢 |
| 确定性 | 100% | 概率性 |
| 可扩展性 | 加规则即可 | 改 Prompt 即可 |
| 典型场景 | 评分合计、PE 存在性、情景单调性 | 投资逻辑矛盾、过度自信、行业描述 |

**Q20: 哪些检查适合代码完成，哪些适合 LLM 完成？**

代码：加减乘除、评分合计、PE/EPS 范围、单位转换、数据缺失、时间一致性。LLM：投资逻辑是否闭环、定性描述是否准确、风险是否遗漏、措辞是否过度自信。

**Q21: 为什么评分卡使用代码生成，而不是让 LLM 打分？**

金融领域要求一致性和可解释性。代码打分可复现、可调参、可审计；LLM 打分可能因 Prompt 温度、模型版本不同而变化。用代码打底，LLM 只能在行业前景、资金认可两个主观维度微调。

**Q22: 为什么估值计算不能完全交给 LLM？**

LLM 会算错 EPS×PE、混淆单位、把悲观价写得比基准价高。FinBrain 把估值框架选择交给 LLM，但具体数字（行业 PE、质量乘数、安全边际、合理价值）由代码计算，并做算术校验。

**Q23: 你的系统如何减少 LLM 幻觉？**

五层防御：1) 工具隔离，Analyst 无 tools；2) RAG 注入行业模板；3) 代码后处理强制覆盖客观分数；4) Critic 反向检查；5) Audit Engine 做最终校验。LLM 不能调用工具编造数据，也不能随意改写数字。

**Q24: 如何保证最终报告中的数字可信？**

所有数字来自工具返回或代码计算，不是 LLM 生成。Reporter 渲染时，盈利能力/成长性/财务健康/估值合理/合理价值/安全边际等关键数字被代码强制覆盖。Critic 和 Audit 检查这些数字是否自洽。

**Q25: 如果 LLM 编造了一个财务指标，你如何发现？**

如果 LLM 在文本中编造数字，它不会被代码后处理覆盖（因为代码只覆盖结构化 JSON 字段）。但 Critic 会检查文本数字与 JSON 字段的一致性，Audit 会检查文本中的数值是否合理。发现矛盾会触发 Repair 或降级。

**Q26: 你的系统有没有数据溯源机制？**

有。每个工具调用记录返回时间、数据源、状态，写入 `processing_log`。最终报告里显示 `[调用证据]` 和 `[审计校验]`，用户能看到数据来自哪个接口、是否命中缓存、是否有告警。

**Q27: 为什么需要记录 Tool Calling Trace？**

Trace 是 Debug 和可解释性的基础。没有 Trace，无法回答“这个数字哪来的”“为什么评分这么低”“为什么没查到这个公告”。Trace 也是未来自动评估 Agent 效果的输入。

**Q28: Execution Trace 对用户有什么价值？**

让用户知道系统不是黑箱。Trace 展示每个 Agent 执行了什么、耗时多少、调用了哪些数据源、发现了哪些问题。用户质疑结论时，可以回到 Trace 看依据。

**Q29: 如果用户质疑某个结论，你如何解释来源？**

打开 Trace 中对应的步骤：例如“估值合理 8 分”来自 `scoring.py` 的 `compute_investment_rating`，行业 PE 锚点 32 倍来自 `scoring.json` 的 IT 服务条目，数据来自东财接口。结论 = 数据源 + 计算链 + 模型判断，每层都可拆解。

**Q30: 你的系统如何做到 Explainable AI？**

三层可解释：1) 数据层，每个数字有工具来源；2) 计算层，评分/估值公式透明；3) 推理层，投资逻辑链要求“行业趋势→公司优势→财务表现→股价影响”闭环。最终报告还包含 `[估值明细]`、`[审计校验]`、`[框架分歧]` 等透明段落。

### 数据与 RAG

**Q31: 你的数据来源有哪些？**

行情（新浪、东财）、财报（东财 RPT_DMSK + RPT_FCI 快源）、估值（同花顺）、行业分类（东财）、公告（东财 np-cnotice）、资金流向（东财）。默认全免费公开接口。

**Q32: 为什么选择这些财务数据接口？**

因为项目定位是个人/小团队可用，优先零成本、无需授权的数据源。东财接口覆盖 A 股财报、行情、公告较全；新浪行情稳定。免费接口是约束，也是设计挑战。

**Q33: 免费 API 的数据质量如何保证？**

1) 多源交叉验证（如财报快源与公告快报互校）；2) TTL 缓存避免重复请求；3) 工具返回异常时标记 N/A 而不是崩溃；4) Audit Engine 检查数据一致性；5) 用户可见 `[调用证据]`，能人工复核。

**Q34: 如果财报 API 返回错误数据怎么办？**

工具层会捕获异常并返回 `{"error": ...}`。Data Agent 会标记该字段缺失。如果关键数据缺失，会触发降级（如禁用某维度评分）。Critic 也会检查数据完整性，提示“数据不足，谨慎参考”。

**Q35: 如何验证财务数据的一致性？**

代码层检查：营收与利润增速是否同向、归母与扣非差距是否过大、现金流与利润是否背离、资产负债率是否在合理区间。Critic 层检查：投资逻辑是否与财务数据一致。

**Q36: 为什么财报分析需要 RAG？**

因为 LLM 训练数据有截止日期，且不同行业估值逻辑不同。RAG 把“半导体封测怎么估值”“周期股为什么不能看单季利润”等行业知识按需注入，避免模型用通用模板硬套。

**Q37: 你的 RAG 知识库存储什么内容？**

四类：会计准则（如收入确认、商誉减值）、行业模板（半导体、IT 服务、电网信息化等）、交易策略（趋势/价值/妖股模式）、游资风格。未来可扩展 ESG、机构持仓解读等。

**Q38: 为什么不直接把所有财报文本输入 LLM？**

上下文窗口不够，且财报冗余信息多。RAG 只检索与分析主题相关的行业模板和会计准则，降低 token 成本，提高信息密度。

**Q39: 你的 RAG 检索流程是什么？**

用户输入股票 → 获取行业分类 → 用行业名作为 query 检索行业模板 → 用具体问题（如“现金流异常”）检索会计准则 → 把检索结果注入 Analyst Prompt。

**Q40: Query 是怎么生成的？**

Query 主要来自行业分类字段（如“半导体”）和当前分析主题（如“经营现金流/扣非净利润”）。不是让 LLM 自由生成 query，而是结构化字段 + 关键词组合，保证检索可预测。

**Q41: 你的 RAG 是静态知识库还是动态检索？**

是静态知识库 + 动态检索。知识库内容用户可上传更新，但检索是在每次分析时动态发生。目前不是动态更新知识库（如自动爬研报），但预留了扩展点。

**Q42: 为什么需要动态 RAG？**

因为不同公司需要不同知识。分析银行股需要“净息差、拨备覆盖率”，分析半导体需要“CAPEX、稼动率、先进封装”。固定知识库无法按需加载，动态 RAG 按行业/问题检索最相关内容。

**Q43: 动态 RAG 如何判断需要检索什么？**

目前按行业分类硬路由：每个行业模板对应一个检索 query。未来可加入 LLM 意图识别，让 Agent 根据用户问题决定检索“估值方法”“风险指标”还是“催化剂”。

**Q44: 如果检索到无关行业知识怎么办？**

1) 按行业分类过滤，减少误召回；2) 设置相似度阈值；3) 在 Prompt 中提示“如以下知识与当前公司无关可忽略”。Critic 也会检查行业描述是否与公司实际匹配。

**Q45: 如何评价你的 RAG 效果？**

目前主要靠端到端报告质量评估：RAG 是否命中了相关行业模板、报告中的行业描述是否准确、是否避免了通用套话。未来可补 recall@k、NDCG 等检索指标，以及人工标注相关性。

**Q46: 如何优化 RAG Recall？**

1) 换更强的中文 embedding（如 bge-large-zh）；2) 增加行业同义词映射；3) 按粗粒度→细粒度分层检索；4) 对过短 chunk 做合并；5) 加入 query 扩展（如“半导体”扩展出“封测、晶圆、先进封装”）。

**Q47: 为什么不用 Fine-tuning 替代 RAG？**

Fine-tuning 让模型“记住”知识，但知识更新成本高，且模型仍可能幻觉。RAG 把知识外置，更新只需改文档，不需要重新训练。金融领域知识更新快（政策、会计准则变化），RAG 更适合。

**Q48: 金融领域为什么更适合 RAG？**

金融知识结构化、时效性强、行业差异大。RAG 能把行业估值模板、会计准则、公司特异性信息按需注入，而不需要模型记住所有领域知识。

**Q49: 财报分析知识库应该如何设计？**

按“行业模板 + 会计规则 + 风险清单 + 估值框架”四维组织。每类模板有明确适用场景、禁忌事项、典型反例。例如半导体模板会强调“CAPEX 周期”“稼动率”“先进封装技术路线”。

**Q50: 为什么行业模板比通用知识库更有效？**

因为 LLM 默认用通用财务分析框架，会误判行业特性。例如“营收集中在四季度”对普通公司不重要，但对电网信息化公司是核心季节性特征。行业模板把这些先验知识直接注入，避免模型用错框架。

### 财务分析与估值

**Q51: Analysis Agent 输入是什么？**

结构化财务数据（营收、利润、现金流、毛利率、ROE 等）、行业分类、RAG 检索到的行业模板、近期公告、当前股价/估值、评分卡 JSON。

**Q52: Analysis Agent 输出是什么？**

结构化 JSON，包含：公司画像、商业模式、护城河、行业周期、成长逻辑、催化剂、风险、情景估值、投资评级、操作建议等。

**Q53: 为什么投资逻辑需要单独生成？**

投资逻辑是“叙事”，需要 LLM 把数据串成因果链。但叙事必须建立在结构化数据之上，不能凭空生成。单独生成便于 Critic 检查“数据是否支持结论”。

**Q54: 如何避免投资逻辑变成“AI 胡编故事”？**

1) 强制逻辑链模板：行业趋势→公司优势→财务表现→股价影响；2) 每个论点必须引用具体财务数据；3) 必须有证伪条件；4) Critic 检查逻辑是否与数据一致；5) 代码预检发现数据与结论冲突时强制标注框架分歧。

**Q55: 你的系统如何验证投资逻辑链？**

Critic 检查逻辑链是否闭环：行业趋势是否真实存在、公司优势是否可复制、财务数据是否支撑优势、估值是否反映市场预期。缺失任何一环都标记为逻辑缺陷。

**Q56: 为什么投资逻辑需要：行业趋势 → 公司优势 → 财务表现 → 股价影响？**

这是“从宏观到微观、从定性到定量”的验证链。缺少行业趋势，优势可能是孤例；缺少公司优势，财务表现可能只是周期；缺少财务数据，优势无法量化；缺少股价影响，投资价值无法落地。

**Q57: 如何判断一个竞争优势是否真实？**

代码层看毛利率是否高于行业、ROE 是否稳定、现金流是否匹配利润。LLM 层分析护城河来源（成本、品牌、网络效应、牌照）和可复制性。两者结合：高毛利 + 稳定 ROE + 可复制壁垒 = 真优势。

**Q58: LLM 如何分析护城河？**

Prompt 要求 Analyst 从“成本优势、规模效应、品牌/客户粘性、技术壁垒、牌照/渠道垄断”五个维度打分，并结合财务指标（毛利率、ROE、营收集中度）论证。避免空泛说“公司有护城河”。

**Q59: 财报中的定性信息如何结构化？**

通过 Prompt 强制输出固定 JSON 字段：护城河来源、行业周期阶段、核心催化剂、主要风险、证伪条件。LLM 的定性判断必须填入这些字段，才能被下游 Critic 检查。

**Q60: 管理层、客户、行业信息未来如何加入？**

通过 datasource tier 的 PREMIUM 插槽扩展：管理层画像、机构持仓、产业链图谱、ESG 治理。每个插槽用 `@premium_slot` 装饰器，等级不足时自动回退，不破坏主流程。

**Q61: 如果增加 ESG、机构持仓数据，需要改哪些模块？**

1) `datasource_tier.py` 增加插槽函数；2) `tools.py` 增加对应工具；3) `ANALYST_PROMPT` 增加分析维度；4) `scoring.json` 可能增加 ESG 评分权重；5) 测试用例补充。主流程节点不变。

**Q62: 为什么需要独立 Valuation Agent？**

估值不仅是算术，还需要选择框架（PE/PB/EV/EBITDA/PEG）。不同公司类型适用不同方法。Valuation Agent 负责框架选择，具体计算由代码完成。独立后便于审计和迭代。

**Q63: 为什么 PE 估值不能直接套行业 PE？**

行业 PE 只是中枢。同行业内公司质量差异大：ROE 高低、成长性、资本开支强度、周期位置都会让合理 PE 偏离中枢。直接套行业 PE 会忽略公司特异性。

**Q64: 为什么长电科技不能简单按照成长股估值？**

长电科技有半导体周期属性，利润随稼动率波动，不是纯成长。用成长股 PE 会高估周期高点、低估周期低点。系统判定为“周期复苏 + 成长叠加”，用正常化利润 + 周期折价 + 成长溢价综合估值。

**Q65: 你的公司分类为什么只有“周期复苏型”不够？**

很多公司同时具备多重属性。单标签会丢失信息。例如长电科技既是周期股又有 AI 成长属性，单标签“周期复苏”会忽略成长溢价，导致估值偏低。

**Q66: 为什么要改成混合权重分类？**

混合权重能同时表达“周期、成长、价值、主题”等多重属性。例如“50% 周期制造 + 30% 成长制造 + 20% AI 主题”比单标签更贴近真实公司画像，估值框架也能混合加权。

**Q67: “50%周期制造+30%成长制造+20% AI主题”的依据是什么？**

来自 Classifier 对公司财务指标（ROE 波动、增速趋势、CAPEX 强度）的归一化计算。它不是 LLM 拍脑袋，而是代码根据历史财务数据自动给出的权重。

**Q68: 权重应该由人工设定还是模型学习？**

目前由人工规则 + 财务指标推导。人工设定可解释、可审计；模型学习需要大量标注数据，且金融样本少。未来可以半监督：代码给初值，LLM/Critic 微调，但必须有上下限和解释。

**Q69: 如何避免估值模型主观化？**

1) 权重由财务指标计算，不是 LLM 写；2) 每个折价/溢价因子都有明确公式和阈值；3) 最终估值透明展示为 PE 折价链；4) Critic 和 Audit 检查估值合理性。

**Q70: 为什么需要 PE 折价链？**

让估值从黑盒变成白盒。用户能看到“行业 PE 40 倍 → 周期折价 -30% → ROE 不足 -20% → CAPEX 压力 -15% → 成长溢价 +20% = 最终 22 倍”。每个因子都有依据，便于质疑和迭代。

**Q71: 为什么行业 PE 40 倍最后折算成 22 倍？**

因为行业中枢是给“普通公司”的，具体公司可能处于周期高点、ROE 低于行业、资本开支过大。折价链把公司特异性量化，得到合理 PE，避免简单套用行业倍数。

**Q72: 折价因子应该由 LLM 判断还是代码计算？**

框架由 LLM 选择（判断周期/成长/价值），具体折价幅度由代码根据规则计算。这样既利用 LLM 语义理解，又保证可复现性。

**Q73: ROE 为什么影响估值？**

ROE 反映资本回报能力。同样 PE 下，高 ROE 公司更值钱。系统用 ROE 质量系数（0.3x–1.6x）调整 PE：ROE 高且稳定 → 溢价；ROE 低或波动大 → 折价。

**Q74: 资本开支为什么应该影响估值？**

高 CAPEX 意味着利润含金量低，自由现金流被挤压。系统用 CAPEX/CFO 比例作为质量因子，CAPEX 过高会降低合理 PE，并触发 FCF 预警。

**Q75: 为什么高增长公司也可能不值得投资？**

如果高增长来自周期反弹、一次性订单或并表，不可持续；如果估值已经透支未来增长（高 PE + 高 PEG），也没有安全边际。系统会区分“真成长”和“伪成长”。

**Q76: PEG 为什么不能简单使用？**

PEG 需要“可持续增速”。单季暴增、周期高点、并表基数都会扭曲增速。系统要求用 2–3 年复合增速或机构一致预期，否则标注“PEG 暂不可用”。

**Q77: 周期股为什么应该使用正常化利润？**

周期股利润波动大，单年利润不能代表真实盈利能力。用周期高点利润会高估，用周期低点会低估。正常化利润（多年平均或中性周期利润）更能反映真实价值。

**Q78: 为什么 EV/EBITDA 适合半导体封测行业？**

半导体封测是重资产行业，折旧大、资本开支高。净利润容易被折旧和财务费用扭曲，EBITDA 更能反映经营现金流能力。EV 剔除了资本结构影响，适合跨公司比较。

**Q79: 什么时候使用 PE，什么时候使用 PB，什么时候使用 EV/EBITDA？**

| 方法 | 适用场景 |
|------|----------|
| PE | 盈利稳定、利润有代表性的公司 |
| PB | 银行、保险等净资产驱动的行业，或亏损公司 |
| EV/EBITDA | 重资产、高折旧、资本开支大、利润被非经营因素扭曲的公司 |

**Q80: 三种估值方法结果冲突怎么办？**

系统会呈现框架分歧：价值框架（PE 锚点）vs 趋势框架（成长溢价）。不把冲突压平，而是告诉用户两套逻辑的假设和适用范围，让用户自己判断。

**Q81: 如何融合多个估值结果？**

目前用“主估值 + 情景估值”双轨。主估值来自代码规则（PE/质量乘数/EPS）；情景估值来自 LLM 对悲观/基准/乐观假设的判断，但结果由代码校验。最终报告同时展示两者，并解释差异。

**Q82: 你的情景估值如何生成？**

Analyst 输出悲观/基准/乐观的 EPS、PE、假设、概率；代码层校验算术（EPS×PE=价格）、单调性（悲观≤基准≤乐观）、概率合计；然后代码重算概率加权价值。

**Q83: 悲观、中性、乐观概率怎么确定？**

由 LLM 根据公司确定性、行业周期、催化剂可兑现程度判断。但系统要求概率合计为 100%，并在代码层归一化。未来可以引入历史回测校准概率。

**Q84: 为什么情景估值不能完全依赖 LLM？**

因为 LLM 会算错 EPS×PE、概率合计不为 100%、价格排序颠倒。代码层强制校验并修正，保证数值自洽。

**Q85: 如何验证估值模型合理性？**

1) 回测：历史报告的交易建议用历史 K 线验证胜率；2) 横向对比：同行业公司估值是否在同一区间；3) 异常检测：合理价值与当前价差距过大时触发框架分歧；4) 专家评估：请有行业经验的人判断估值逻辑是否成立。

**Q86: 为什么不能只看净利润？**

净利润容易被非经常性损益、会计估计、并表范围操纵。扣非净利润、经营现金流、自由现金流更能反映主业真实盈利能力。

**Q87: 为什么需要 FCF？**

FCF 是“利润扣除维持性资本开支后的真现金”。一家公司净利润很高但 FCF 为负，说明它赚的钱都变成了设备、厂房，股东拿不到真金白银。

**Q88: 为什么长电科技 CFO 很高但是 FCF 为负？**

因为半导体封测行业资本开支巨大，经营现金流（CFO）虽然高，但扣除 CAPEX 后自由现金流（FCF）为负。这说明公司处于扩张期，利润含金量不高。

**Q89: 经营现金流优秀是否代表现金流健康？**

不一定。如果 CAPEX 远高于 CFO，公司仍在烧钱扩张。还要看 FCF、应收账款变化、资本开支方向。系统用 FCF 为负 + CAPEX/CFO>0.3 自动预警。

**Q90: CAPEX 对半导体公司的意义是什么？**

CAPEX 决定技术路线和产能规模。先进封装（如 Chiplet）需要大量资本投入，CAPEX 不足会落后，CAPEX 过高会拖累 ROE 和 FCF。系统把它作为估值折价因子和风险信号。

**Q91: 为什么资本开支过大会降低 ROE？**

因为 ROE = 净利润 / 净资产。大规模 CAPEX 增加净资产（分母），但产能投产和利润释放有滞后，短期内净利润（分子）增长慢，ROE 被拉低。

**Q92: 如何自动发现财报中的风险？**

代码层规则：应收账款/净利润 > 3、经营现金流为负、归母扣非差距 > 30pp、FCF 为负且 CAPEX/CFO 高、毛利率连续下滑。LLM 层：Critic 检查风险描述是否遗漏或轻描淡写。

**Q93: 你的系统如何判断利润质量？**

看扣非净利润/归母净利润比例、经营现金流/净利润比例、应收账款增速 vs 营收增速、毛利率稳定性。比例异常或趋势背离都标记为利润质量风险。

**Q94: 如何判断收入增长是否可持续？**

1) 看收入是否来自核心客户/产品；2) 看订单/合同负债；3) 看行业周期位置；4) 看毛利率是否同步提升。如果收入增长但毛利率下降，可能是降价抢市场，不可持续。

**Q95: 如何分析毛利率变化原因？**

Prompt 要求 Analyst 从“产品结构变化、原材料价格、产能利用率、竞争格局、汇率/政策”五个维度分析。并结合行业模板判断是周期因素还是公司竞争力变化。

**Q96: 如果公司收入增长但现金流下降，你怎么看？**

可能是赊销增加、应收账款积压、库存占用资金。这是危险信号，说明收入增长没有转化为现金。系统会注入回款风险警告。

**Q97: 如果利润下降但现金流增加，如何解释？**

可能是一次性资产处置、压缩库存、收紧信用政策、前期资本开支进入收获期。需要区分是一次性因素还是经营改善。Critic 会检查叙事是否与数据一致。

**Q98: 如何分析周期行业？**

看行业产能利用率、库存周期、价格指标、资本开支周期、龙头公司毛利率变化。系统用行业模板提示周期分析框架，并要求给出当前周期位置。

**Q99: 如何判断公司处于周期哪个阶段？**

用财务指标组合：营收增速、毛利率、ROE、库存周转、CAPEX 强度。如果营收和毛利率同时回升但 CAPEX 还没大规模启动，可能是复苏早期；如果 CAPEX 高峰已过、利润率开始下滑，可能是周期顶点。

**Q100: 如何避免把周期反弹误认为成长？**

用混合权重分类：如果 ROE 波动大、CAPEX 强度高、行业产能周期明显，就赋予周期属性权重。估值时用正常化利润和周期折价，而不是直接用高增速给高 PE。

### Critic 深挖

**Q101: Critic Agent 的输入输出格式是什么？**

输入：Analyst/Valuation 输出的 JSON、原始财务数据、行业模板、评分卡、当前股价。输出：结构化修复清单 `[CRITIC_FIXES]`，每项包含 `字段`、`问题描述`、`严重等级`、`修正建议`。不直接改报告，只提清单。

**Q102: Critic 如何发现“投资逻辑矛盾”？**

Critic 检查 JSON 中“投资逻辑→风险→评级→操作建议”是否一致。例如：逻辑说“公司处于周期高点、估值过高”，但评级是 BUY，操作是“重仓”，就是矛盾。

**Q103: Critic 为什么能够发现：“估值过高但建议持有”的矛盾？**

因为 Critic 同时读取估值结果（合理价值、当前价、安全边际）和操作建议（评级、仓位、止损）。如果合理价值远低于当前价但评级不是 SELL，代码层和 LLM 层都会标记。

**Q104: Critic 是重新分析公司还是检查已有分析？**

Critic 不重新分析公司，只检查已有分析。这是“验证者”而非“替代者”角色。重新分析会引入新的偏见，只做检查才能保持独立性。

**Q105: Critic 是否需要访问工具？**

Financial Critic 需要访问工具做代码预检（FCF、PE/ROE 匹配）。Logic/Industry Critic 主要依赖已有数据和 LLM 语义判断，不需要新增工具调用。

**Q106: Critic 发现错误后是否自动修改报告？**

不直接修改。Critic 输出修复清单，交给 Repair Agent 修改。这样分离“找问题”和“改问题”的角色，避免 Critic 既当裁判又当运动员。

**Q107: 为什么你的 Critic 不直接生成最终报告？**

如果 Critic 直接生成报告，就失去了反向验证的意义。Critic 的产出是“问题清单”，Repair 根据清单修改，Audit 再验证，形成闭环。

**Q108: Critic Agent 是否可能产生新的幻觉？**

可能。Critic 也是 LLM，可能误报或漏报。缓解方法：1) Financial Critic 先做代码预检，减少主观判断；2) 代码层 Audit 二次过滤；3) 动态重试预算，避免过度修改；4) 用户可见审计摘要，可人工复核。

**Q109: 如何评价 Critic 的准确率？**

用离线 benchmark：构造已知错误的报告，看 Critic 能否检出。统计精确率、召回率、误报率。还可以用真实报告的人工质检结果做对照。

**Q110: Critic 和 Human Analyst 的角色区别是什么？**

Critic 是自动化“找茬”角色，覆盖规则明确、重复性高的检查。Human Analyst 负责判断复杂商业逻辑、行业洞察、非结构化信息。Critic 把人从“查错”中解放出来，让人专注于“判断”。

### 工程实现

**Q111: 为什么选择这个技术栈？**

LangGraph：固定流程 + 状态持久化；LangChain：Tool/Model 统一接口；FastAPI：数据服务；Streamlit：快速前端；ChromaDB：本地 RAG；Redis：共享缓存。选型原则是“够用、可本地运行、零运维成本”。

**Q112: Agent 调用链如何实现？**

LangGraph StateGraph 定义节点和边。每个节点是 Python 函数，接收 `state`、返回 `dict`。`state` 通过 Checkpointer（SqliteSaver）持久化，支持跨轮次恢复。

**Q113: 如何管理多个 Agent 状态？**

用 LangGraph 的 `State` 对象 + SqliteSaver。每个节点返回的 dict 会与 state 合并，普通 list 字段需要节点自己读取旧值追加。`thread_id` 区分不同用户会话。

**Q114: Agent 输出如何结构化？**

核心靠 `backend/schemas.py` 里的 Pydantic 模型：`AnalystOutput`、`ValuationOutput`、`CriticOutput`、`AuditOutput`。LLM 节点通过 LangChain 的 `with_structured_output(schema)` 直接拿到 Pydantic 对象。如果框架级结构化失败，`_StructuredFallbackChain` 会回退到原始文本并手动解析。最终所有字段都经过 Pydantic 校验，缺失或类型错误会触发重试或降级默认值。

**Q115: 如何保证 JSON Schema 正确？**

1) 用 Pydantic 定义 schema，字段类型、可选/必填、默认值都显式声明；2) `with_structured_output()` 在模型解码层约束输出格式；3) `temperature=0` 降低随机性；4) 代码层对返回的 Pydantic 对象做二次校验；5) 结构化失败时由 `_StructuredFallbackChain` 手动解析并回退；6) 最终仍失败则抛出明确错误，避免带着错误格式继续下游。

**Q116: LLM 输出格式错误怎么办？**

先用 `raw_decode` 从文本中提取第一个有效 JSON；如果失败，用正则匹配代码块；再失败，返回降级 JSON 或错误提示，并记录到 `processing_log`。不会直接崩溃。

**Q117: 如何设计 Retry 机制？**

三个层面：
1. **工具层**：同一工具连续失败 3 次熔断，切换到备用数据源；
2. **LLM 层**：`LLMFallbackChain` 按 slot 1→2→3 顺序尝试，主模型失败自动切备用模型；结构化输出失败时，`_StructuredFallbackChain` 先换模型再手动解析；
3. **审计层**：发现问题最多重试 3 次。所有重试都有预算，避免无限循环。

**Q118: 如何降低 API 调用成本？**

1) TTL 缓存（30s 股价/30min 财报/1h 行业）；2) 工具调用去重；3) Analyst 节点单次 invoke，不用 ReAct；4) 小模型处理简单任务，大模型只用于复杂分析；5) Prompt 压缩和上下文摘要。

**Q119: 哪些步骤可以缓存？**

数据工具结果：股价、财报、估值、行业、公告。分析结果：同一只股票短期内复用。RAG 检索结果：知识库不变时可缓存。Prompt 模板：热加载但可缓存。

**Q120: 如果同时分析100家公司，系统如何扩展？**

远程模式：FastAPI 数据服务 + Redis 共享缓存 + scheduler 预取。分析请求可以并行（ThreadPoolExecutor）。LLM 调用是瓶颈，可引入异步队列或批量请求。

**Q121: 如何设计异步执行？**

分析请求可拆分为数据获取（I/O 密集型，async）和 LLM 推理（可并发）。使用 FastAPI async + Redis + 任务队列。LangGraph 节点目前同步，可逐步改造成 async。

**Q122: 如何保存历史分析结果？**

当前通过 Checkpointer 保存 graph 状态。未来可写入数据库（PostgreSQL/SQLite），按用户 + 股票 + 时间索引，支持查询和对比。

**Q123: 如何支持用户查看过去报告？**

在 Streamlit 前端增加历史报告页面，按 `thread_id` 或用户 ID 查询数据库。展示报告列表、生成时间、关键评分、可点击查看完整报告和 Trace。

**Q124: 如何监控 Agent 性能？**

记录每个节点耗时、LLM token 消耗、工具调用次数、缓存命中率。日志写入文件或发送到监控。关键指标：端到端延迟、成功率、Critic 检出率、Audit 重试率。

**Q125: 如何定位一次分析失败原因？**

查看 `processing_log` 和 Execution Trace：哪个节点失败、哪个工具返回 error、哪次 LLM 输出格式错误、哪条审计规则触发。Trace 是 Debug 的核心。

### 面试官挑战问题

**Q126: “你的系统是不是只是 Prompt Chain？”**

不是。Prompt Chain 是线性调用 LLM，没有反馈和修正。FinBrain 有工具调用、代码层计算、Critic 反向检查、Repair 自动修正、Audit 最终校验，形成闭环。每个节点不是简单 LLM，而是有明确输入输出和状态管理。

**Q127: “为什么不用 GPT-5 直接分析？”**

GPT-5 再强也会幻觉、也会算错数、也会用错行业框架。金融分析需要可复现、可审计、可追溯。FinBrain 用代码负责确定性计算，用 LLM 负责语义推理，两者互补。模型能力提升会让 LLM 部分更好，但护栏和代码层仍然必要。

**Q128: “你的 Agent 哪里体现智能？”**

智能体现在：1) 自主决定调用哪些工具获取数据；2) 从数据中提炼投资逻辑；3) 识别自身输出中的矛盾和错误；4) 根据修复清单定向修改。不是无约束地“思考”，而是在结构化流程中做有边界的有用决策。

**Q129: “如果 LLM 能力继续提升，你这个系统还有价值吗？”**

有价值。LLM 能力提升会让 Analyst/Valuation/Critic 的推理更强，但系统的价值在于：数据工程、代码化计算、审计机制、多 Agent 协作、可解释性。这些不是模型能力本身，而是工程和组织能力。

**Q130: “你的估值模型是不是人工规则？”**

部分是。核心公式（行业 PE × 质量乘数 × EPS）是人工设计的规则，但输入（ROE、增速、CAPEX）是真实数据，权重由 Classifier 代码计算。规则提供可解释性和稳定性，LLM 负责框架选择和叙事补充。未来可以用数据部分优化参数，但规则骨架保留。

**Q131: “既然用了规则，为什么还需要 Agent？”**

规则只能回答“值多少钱”，无法回答“为什么”。Agent 负责把数据、行业知识、投资逻辑串成故事，识别规则覆盖不到的风险，处理规则边界案例。规则是骨架，Agent 是血肉。

**Q132: “你的系统最大的创新点是什么？”**

最大创新是“确定性计算 + 概率性推理的强制边界”。不是简单用 LLM 写研报，而是用代码固定财务指标和评分，用 LLM 做叙事和审查，用 Critic 做反向验证，用 Audit 做最终兜底。这种设计在金融这种对数字敏感、对一致性要求高的领域特别有效。

**Q133: “你的系统和同花顺、雪球的区别是什么？”**

同花顺/雪球是信息终端，提供数据但不产生投资逻辑。FinBrain 是研究 Agent，主动抓取数据、生成分析逻辑、检查一致性、输出可审计报告。它不会替代行情软件，而是帮助用户从数据到决策。

**Q134: “如果让你继续迭代三个月，你会做什么？”**

按已规划的 Phase 2/3 推进：1) 回测引擎增强（最大回撤、夏普、胜率、盈亏比、每日盈亏曲线）；2) 评估体系（N 只股票 × M 次运行 → 收益分布、验证“高评分是否真跑赢”）；3) Human-in-the-Loop：模拟交易前用 LangGraph `interrupt()` 让人工审批；4) Streaming：`graph.stream()`/`astream_events()` 把节点事件实时推送到前端；5) 接入 Wind/Choice 付费数据源，补充一致预期、产业链、机构持仓。

**Q135: “你认为当前系统最大的不足是什么？”**

1) 免费数据源不稳定且数据质量有限；2) 回测引擎只检测信号触发，没有策略收益评估；3) 估值模型参数主要靠人工设定，未做历史回测校准；4) 没有 Human-in-the-loop；5) 单元测试覆盖还不够细。

**Q136: “如果加入更多数据，你优先加入什么？”**

优先加入机构一致预期（盈利预测、目标价）和产业链/订单数据。一致预期能修复当前报告“预期陈旧”问题；产业链/订单数据能验证公司成长逻辑，而不只是看历史财报。

**Q137: “如何让这个系统达到机构研究员水平？”**

需要三层：1) 数据层接入 Wind/Choice 等付费数据，覆盖一致预期、产业链、管理层；2) 模型层引入更专业的金融模型（DCF、分部估值、产业链模型）；3) 人机协作层加入研究员反馈闭环，让 Critic 从专家标注中学习。

**Q138: “如何验证你的分析比普通 GPT 更好？”**

构建 benchmark：选一组股票，用 FinBrain 和普通 GPT 同时出报告，人工盲评以下维度：数据准确性、逻辑一致性、风险覆盖、估值合理性、操作建议具体性。统计胜率。也可以用回测验证交易建议的表现。

**Q139: “你的系统有没有实际投资价值？”**

当前定位是研究辅助工具，不是直接给出买卖信号。它帮助研究员快速生成分析框架、发现数据矛盾和风险点，但投资决策仍需人工判断。随着数据源和回测完善，可以逐步向量化策略辅助演进。

**Q140: “如果 AI 分析错了一家公司，责任如何处理？”**

系统层面：明确免责声明，报告标注“不构成投资建议”，保留数据溯源和审计摘要。产品层面：保留 Human-in-the-loop，关键决策人工确认。技术层面：持续优化 Critic 和 Audit，降低错误率。责任最终由使用者承担，系统只是辅助工具。

## 补充：Phase 0 / Phase 1 新增高频追问

> 下面这些题目来自实习/校招面试官对 FinBrain 这类财报分析 Agent 的深挖。回答核心思路：先讲“为什么拆”，再讲“边界在哪”，最后落到“代码怎么实现”。

### 1. 你为什么选择做 FinBrain（财报分析 Agent），而不是一个通用 Coding Agent？

财报分析有几个天然适合 Agent 的特性：
- 输入结构化（财报、行情、行业数据），输出有明确范式（投资逻辑、估值、评级）；
- 容错率低，数字错误会直接误导决策，必须引入 Critic 和 Audit；
- 领域知识明确，RAG 能发挥行业模板和会计准则的价值。

通用 Coding Agent 需要处理任意代码库、任意语言、任意业务逻辑，不确定性更高。财报分析 Agent 在“有边界的复杂任务”里更容易做出可验证、可度量的成果。

### 2. 你的 Agent 和 Claude Code / Cursor / Devin 这类 Coding Agent 的区别是什么？

| 维度 | FinBrain | Coding Agent |
|------|----------|--------------|
| 任务边界 | 财报分析，流程固定 | 任意代码任务，流程开放 |
| 状态管理 | LangGraph StateGraph，8 节点流水线 | 通常对话式或文件系统级操作 |
| 工具 | 金融数据工具，确定性输入 | 文件读写、终端、浏览器，副作用大 |
| 验证 | 代码层 Audit + LLM Critic 双重校验 | 主要靠测试/编译反馈 |
| 输出 | 结构化研报 | 代码文件或 diff |

本质区别：Coding Agent 强调“在开放环境里完成未知任务”，FinBrain 强调“在已知流程里保证输出一致和可信”。

### 3. 一个系统满足什么条件才算 Agent，而不是普通 LLM 应用？

三个缺一不可：
1. **工具使用**：能自主调用外部函数获取真实数据（不是只依赖训练数据）。
2. **状态/记忆**：跨步骤保留上下文，能根据历史状态决定下一步。
3. **目标导向的自主决策**：遇到异常不是简单报错，而是尝试恢复（如数据缺失重取、模型失败切换）。

普通 LLM 应用 = 单次 prompt → 单次回答。Agent = 多轮工具调用 + 状态更新 + 错误恢复。

### 4. 没有 Planner Agent、没有动态 RAG，你的系统还能称为 Agent 吗？

能。Planner 和动态 RAG 是 Agent 的增强组件，不是必要条件。判断标准是系统是否具备自主工具调用、状态管理和目标导向的恢复能力。FinBrain 的 Data Agent 自己决定调用哪些工具、LangGraph 节点根据状态做条件路由、LLM 失败时自动 fallback，这些都是 Agent 行为。

Planner 更适合任务类型不固定、需要动态分解的场景。财报分析任务边界明确，固定图结构更可控。

### 5. Agent 和 Workflow 的本质区别是什么？

- **Workflow**：步骤是预定义且确定的，if/else 由代码写死，没有基于上下文的自主调整。
- **Agent**：节点可以根据状态自主决定下一步（如 LLM 选择工具、LangGraph 条件边根据数据完整性决定重取）。

FinBrain 是“确定性 Workflow + 有限自主性”的混合：大部分节点顺序执行，但在 Data（缺失重取）和 Critic（是否 Repair）两个关键点引入了 Agent 式决策。

### 6. 你的系统里面哪些部分体现了 Agent 的自主性？

1. Data Agent 自主选择调用哪些工具（财报、估值、行情、行业、公告等）。
2. LangGraph 条件边：数据缺失时自动重取，Critic 无问题时跳过 Repair。
3. LLM Fallback Chain：主模型失败时自动切换到备用槽位。
4. Critic Agent 自主发现逻辑/财务/行业问题并输出修复清单。
5. Audit Engine 自主判断评分一致性、估值算术、时效性矛盾。

### 7. 为什么你的系统需要 Multi-Agent，而不是一个大模型一次性完成所有任务？

三个原因：
1. **专业分工**：Data Agent 负责数据、Analyst 负责叙事、Valuation 负责估值框架、Critic 负责审查。单一 LLM 同时扮演所有角色容易混淆。
2. **可验证性**：每个 Agent 输出结构化对象，中间产物可检查。一次性完成所有任务的输出是黑盒。
3. **可控性**：可以单独替换/优化某个 Agent（如升级 Critic 的 prompt 或接入新数据源），不影响整体。

### 8. Multi-Agent 架构相比单 Agent 的优势和代价是什么？

**优势**：
- 专业化：每个 Agent 只关注一个领域，输出质量更高；
- 可调试：失败定位到具体 Agent；
- 可扩展：新增 Agent 不影响已有流程；
- 可审计：每个节点有明确输入输出和 Trace。

**代价**：
- 调用成本上升：多 Agent 意味着多次 LLM 调用；
- 延迟增加：尤其 Critic 三路并行 + Repair 会多轮调用；
- 通信复杂度：需要定义清晰的 state 和数据结构；
- 调试难度：多节点失败时，需要 Execution Trace 定位根因。

### 9. 你的 Multi-Agent Workflow 是固定流程还是动态规划？为什么这么设计？

固定流程为主，局部动态决策。LangGraph 的图结构是编译时确定的：Data → Classify → Analyst → Valuation → Critic → Repair → Reporter。但 Data 后是否重取、Critic 后是否 Repair 是运行时根据状态决定的。

财报分析任务步骤明确，不需要每次生成不同 DAG。固定流程保证可预测性和可调试性；局部动态决策（条件边）处理异常和跳过。

### 10. 为什么没有采用 Planner Agent 自动生成 DAG？

Planner Agent 适合任务类型不确定、需要动态分解的场景。财报分析任务的流程是固定的：数据 → 分类 → 分析 → 估值 → 审查 → 修复 → 报告。引入 Planner 的收益有限，但会带来：
- 不可预测性：每次生成不同 DAG，难调试；
- 额外延迟：生成计划本身需要一次 LLM 调用；
- 幻觉风险：Planner 可能生成不合理流程。

固定图 + 条件边在 FinBrain 场景下更可靠。

### 11. Planner Agent 在什么场景下有价值？什么时候反而会降低系统效果？

**有价值**：任务类型多变、需要动态组合工具（如“帮我调研这个行业的竞争格局、估值、政策风险” vs “帮我生成一份纯财务诊断报告”）。

**降低效果**：流程高度确定、对一致性和可审计性要求高的场景。Planner 引入了不确定性，可能生成跳过硬编码检查的路径，导致输出质量下降。

### 12. 如果加入 Planner Agent，你会如何设计？

1. 把 Planner 放在最前面，只负责选择“分析模式”（如完整研报 / 快速诊断 / 事件驱动 / 妖股追踪）。
2. 每种模式对应一个固定的子图，Planner 不生成任意 DAG，而是路由到预定义模板。
3. Planner 输出必须是枚举类型（选择模式），而不是自由文本，降低幻觉风险。
4. 用代码层做后校验：Planner 选的模式必须有对应的子图实现，否则 fallback 到默认模式。

### 13. 你的任务 DAG 如何生成？节点之间如何表示依赖关系？

DAG 是代码里显式写的，不是自动生成的。LangGraph 用 `StateGraph` 定义：

```python
graph = StateGraph(FinBrainState)
graph.add_node("data_collector", data_collector_node)
graph.add_node("analyst", analyst_node)
graph.add_edge("data_collector", "analyst")   # 顺序依赖
graph.add_conditional_edges("data_collector", route_after_data)  # 条件依赖
```

依赖关系通过 State 对象传递：每个节点读取需要的字段，返回新的字段。普通字段直接覆盖，list 字段需要节点自己读取旧值追加。

### 14. DAG 调度过程中如何处理任务失败？

1. **工具失败**：重试 3 次，失败后切换到 fallback 数据源（`datasource_tier.py`）。
2. **LLM 失败**：`LLMFallbackChain` 切换到下一个 LLM 槽位。
3. **节点失败**：LangGraph 抛出异常，Execution Trace 记录失败节点和原因。
4. **数据缺失**：`route_after_data` 自动回退重取一次。
5. **Critic/ Audit 失败**：触发 Repair 或 Analyst 重推理，最多 3 次。

### 15. 如果一个 Agent 输出错误结果，下游 Agent 如何感知？

通过 State 中的 `processing_log` 和结构化输出传递。Critic 会检查 Analyst 的输出，Audit 会检查 Reporter 的输出。每个下游 Agent 不仅看到上游结果，还看到代码层计算的分数和校验结果。

例如，Analyst 输出的评级如果和代码层 `compute_investment_rating` 不一致，Audit 会报 `[Framework Divergence]`，强制 Reporter 处理。

### 16. 你的 Agent 之间如何通信？采用什么数据结构？

通过 LangGraph 的 `State`（`TypedDict`）通信。每个节点返回一个 dict，与当前 state 合并。核心字段包括：
- `user_question`: 用户输入
- `collected_data`: Data Agent 收集的 JSON 数据
- `analysis`: Analyst 输出（结构化 JSON）
- `valuation`: Valuation 输出
- `processing_log`: 节点执行日志
- `report`: 最终报告

### 17. 为什么不用自然语言作为 Agent 间通信协议？

自然语言作为通信协议有两个问题：
1. **不可解析**：下游 Agent 无法稳定提取字段，容易丢信息或误解；
2. **不可审计**：中间产物是文本，很难用代码校验。

用结构化 JSON / Pydantic 对象通信，下游 Agent 可以精确读取字段，代码层可以做校验，也便于 Trace 和 Debug。

### 18. Agent 间共享 Context 会导致什么问题？

- **上下文污染**：Critic 看到 Repair 的修改，可能失去独立性；
- **上下文过长**：所有节点共享 messages，可能超出 LLM 上下文窗口；
- **调试困难**：一个问题可能是多个节点共同写入导致的，定位难。

FinBrain 的缓解措施：每个 Agent 只读取自己需要的字段；Analyst 无 tools；Critic 输入不直接包含 Repair 输出；上下文压缩模块控制历史长度。

### 19. 如何设计 Agent Memory？

Agent Memory 分两层：
1. **短期 Memory**：当前分析流程的 State，通过 LangGraph SqliteSaver 持久化，跨轮次可恢复。
2. **长期 Memory**：历史报告、用户偏好、股票画像，未来可写入数据库（PostgreSQL/SQLite），按用户/股票/时间索引。

短期 Memory 解决当前对话连贯性；长期 Memory 支持个性化推荐和跨报告对比。

### 20. 你的系统有没有 Memory？如果没有，未来如何扩展？

当前系统有短期 Memory（LangGraph Checkpointer），没有长期 Memory。未来扩展：
- 用户维度：保存用户偏好（如更偏价值投资还是趋势投资）；
- 股票维度：保存历史评分、投资逻辑、验证结果，做时间序列对比；
- 全局维度：保存 Critic 和 Audit 的反馈，用于优化 Agent。

### 21. 为什么不用 RAG 直接生成最终报告？

RAG 只负责提供“知识上下文”，不能替代分析、估值、审查流程。直接让 RAG 生成报告会导致：
- 无法获取最新财报数据（RAG 知识库不是实时数据源）；
- 无法做定量计算和一致性校验；
- 无法保证不同报告的结构和口径一致。

RAG 在 FinBrain 中是 Data Agent 的输入之一，不是最终输出生成器。

### 22. RAG 和 Agent 的区别是什么？

RAG 是“检索 + 生成”的管道，只解决“知识从哪里来”的问题。Agent 是在 RAG 基础上增加了：
- 工具调用获取实时数据；
- 多步骤推理和规划；
- 错误恢复和状态管理；
- 多 Agent 协作和验证。

RAG 是 Agent 的一个子组件，不是 Agent 本身。

### 23. 一个 RAG Pipeline 为什么不能被称为 Agent？

因为它缺少 Agent 的三个核心能力：
1. 不自主调用工具获取实时数据；
2. 没有跨步骤的状态管理；
3. 不做基于目标的条件决策和错误恢复。

RAG Pipeline 是单次检索 + 单次生成的函数调用，Agent 是循环的、有状态的、目标导向的系统。

### 24. RAG 在你的系统里面解决什么问题？

RAG 解决“领域知识”问题：
- 行业模板（半导体、电力、医药等）提供估值框架和历史 PE 区间；
- 会计准则知识（如商誉减值、收入确认）帮助 Analyst 理解财务指标；
- 交易策略知识（如妖股识别、资金流向）供 Phantom Hunter 使用。

RAG 不解决数据获取问题，数据由工具调用负责。

### 25. 为什么你的系统需要动态 RAG，而不是固定知识库？

动态 RAG 根据输入决定检索什么。财报分析里，不同公司需要不同知识：
- 半导体公司需要“周期复苏 + AI 主题”知识；
- 电力公司需要“煤价、电价、装机量”知识；
- 医药公司需要“集采、研发管线”知识。

固定知识库会把所有内容都塞进上下文，既浪费 token 又降低相关性。动态 RAG 按公司行业/类型选择对应 KB 和查询。

### 26. 动态 RAG 如何决定什么时候检索？

Data Agent 在行业分类确定后，根据 `行业` 字段选择对应 KB。Analyst 和 Valuation Agent 的 prompt 中预留 `[INDUSTRY]` 占位符，系统用 RAG 检索结果填充。目前是在固定节点触发检索，未来可以扩展到 Agent 根据问题自主决定是否需要额外检索。

### 27. 如果 RAG 检索到错误信息怎么办？

三层缓解：
1. **多 KB 隔离**：财报、行业、交易、会计准则分开，降低错误知识污染；
2. **检索置信度过滤**：按距离阈值排除低相关 chunk；
3. **Critic 审查**：Industry Critic 会检查 RAG 知识是否与分析结论一致。

最终，关键投资逻辑仍需要真实财务数据支撑，RAG 只作为背景知识。

### 28. 如何评价 RAG 的效果？

离线评价：
- 召回率：给定问题，相关 chunk 是否被检索到；
- 精确率：检索到的 chunk 中多少是相关的；
- 答案质量：是否帮助生成更好的投资逻辑。

未来可以做人工标注：给一组标准问题，让人工判断 RAG 检索结果是否相关。

### 29. 如何降低 RAG 的幻觉？

1. 知识库只放权威来源（会计准则、行业研报、教材）；
2. 检索结果注入 prompt 时明确标注来源；
3. Critic 检查 RAG 知识与真实数据是否一致；
4. 关键结论不直接依赖 RAG，而是依赖真实财务数据。

### 30. 金融领域知识库应该如何设计？

按“分层 + 行业 + 用途”组织：
- **通用层**：会计准则、估值原理、财务分析框架；
- **行业层**：半导体、医药、电力、消费等模板，包含行业周期、估值模式、关键指标；
- **策略层**：妖股识别、资金流向、市场广度；
- **公司层**：特定公司历史报告、管理层信息、重大事件。

FinBrain 目前实现了通用层 + 行业层 + 策略层。

### 31. Embedding 模型如何选择？

当前用 ONNX MiniLM-L6-V2（80MB，本地运行）。优点是轻量、零运维；缺点是中文语义理解不如中文专用模型。未来可以切换到：
- `bge-large-zh`：中文检索效果更好；
- `m3e`：开源中文 embedding，适合金融文本；
- 付费 API：OpenAI / Cohere embedding，但增加成本。

选择标准是：在领域文本上测召回率，优先选与领域相关的模型。

### 32. 为什么不用 Fine-tuning 替代 RAG？

Fine-tuning 把知识塞进模型参数，适合：稳定、高频、数据量大的任务。RAG 适合：知识会更新、需要溯源、数据量不大的场景。

财报分析里，行业模板、会计准则、最新公告都会更新，且需要来源可追溯，所以 RAG 更合适。Fine-tuning 可以用于让 LLM 更熟悉输出格式，但不替代 RAG。

### 33. 你的系统为什么需要多个 Agent 调用 LLM，而不是写死规则？

写死规则只能处理已知的、确定的逻辑。但财报分析里有大量需要语义理解的环节：
- 公司商业模式描述；
- 投资逻辑链因果表述；
- 行业竞争格局判断；
- 风险因素表述。

这些交给 LLM，规则负责固定财务计算和校验。两者分工明确。

### 34. 哪些逻辑应该交给代码，哪些逻辑应该交给 LLM？

**交给代码**：
- 财务指标计算（EPS、ROE、PE、FCF）
- 评分和估值
- 数据一致性校验
- 审计规则
- 数据获取和缓存

**交给 LLM**：
- 投资逻辑链生成
- 行业趋势解读
- 竞争优势分析
- 风险叙事
- 修复清单的语义理解

边界：凡是能写成确定公式的，都代码化；凡是需要语义理解和综合判断的，才用 LLM。

### 35. 为什么评分系统不用 LLM，而使用代码计算？

因为评分需要：
- 可复现：同一只股票跑 100 次得分一致；
- 可审计：每个得分都有明确公式和阈值；
- 不幻觉：LLM 可能根据叙事风格给出不同分数，代码不会。

LLM 可以输出对评分的解释，但分数必须由代码计算。

### 36. 为什么财务指标计算不能交给 LLM？

LLM 会把 13.30% 毛利率、13.44 亿营收混在一起，甚至算错增速、算错 EPS。财务指标计算是确定性的，必须代码化。代码计算还能：
- 处理单位换算（亿元 / 万元）；
- 处理 TTM 和年化；
- 检测异常值（如 PE 为负、收入为负）。

### 37. 如何保证 Agent 输出的可信度？

四层：
1. **数据源可信**：工具调用真实 API，不是 LLM 编造；
2. **计算可信**：财务指标和评分代码化；
3. **审查可信**：Critic 检查逻辑、财务、行业；
4. **审计可信**：Audit Engine 检查数学一致性、时效性、评分-评级一致性。

每层都有独立输入和输出，形成互相验证。

### 38. 你的系统如何进行结果验证？

1. 代码层 Audit：9 项确定性检查；
2. LLM 层 Critic：3 路并行审查；
3. 结构化输出校验：Pydantic 模型保证字段类型；
4. 交叉验证：用不同数据源（如公告快报 vs datacenter 快报）核对关键数字；
5. 人工 Trace：前端展示每个节点的输入、输出和耗时。

### 39. 为什么需要 Critic Agent？

因为人（或 LLM）写的分析报告容易有：
- 逻辑漏洞（因果不成立）；
- 财务误读（把季节性波动当趋势）；
- 行业误述（竞争格局判断错误）；
- 过度自信（绝对化表述）。

Critic 专门负责“找问题”，是反向验证机制。没有 Critic，报告质量完全依赖 Analyst 一次生成的质量。

### 40. Critic Agent 和普通审核规则有什么区别？

| 维度 | 普通规则 | Critic Agent |
|------|---------|--------------|
| 能力 | 只能检查预定义模式 | 能做语义理解、因果判断 |
| 例子 | “PE 不能为负” | “把 Q1 利润年化得出 PE 986 倍，与季节性提示矛盾” |
| 稳定性 | 100% 稳定 | 可能误报/漏报 |
| 成本 | 低 | 需要 LLM 调用 |

两者互补：规则做确定性检查，Critic 做语义检查。Financial Critic 先做代码预检，再让 LLM 做语义审查。

### 41. Critic Agent 是否应该调用 LLM？为什么？

应该。因为有些问题（如逻辑矛盾、过度自信、行业误述）无法用规则穷举。但 Critic 应该和代码层结合：
- 规则先过滤一遍，减少 LLM 输入；
- LLM 只做语义级审查；
- Audit 再做最终校验。

纯 LLM 的 Critic 成本高、不稳定；纯规则的 Critic 覆盖不了语义问题。混合是最佳实践。

### 42. 如果 Critic Agent 使用 LLM，它如何避免自己产生错误？

1. 输入先做代码预检，提供客观事实（如 FCF、PE、ROE），减少 LLM 主观判断；
2. 输出结构化（`CriticOutput`），只输出“问题列表”和“建议”，不重新生成报告；
3. Audit Engine 对 Critic 发现的问题做二次过滤；
4. 限制 Critic 修改范围，只针对 Analyst 原始输出，不能修改数据。

### 43. Critic Agent 的输入和输出分别是什么？

**输入**：
- 上游 Agent 的输出（如 Analyst 的 JSON 或 Reporter 的报告）；
- 代码层计算的客观指标（FCF、评分、估值水位）；
- 原始财务数据摘要。

**输出**：`CriticOutput` 结构化对象：
- `通过`: bool
- `逻辑漏洞`: List[str]
- `财务误读`: List[str]
- `行业误述`: List[str]
- `建议`: str

### 44. Critic Agent 如何判断一个投资逻辑是否合理？

从三个维度：
1. **因果链**：前提是否成立？推论是否有跳跃？
2. **数据一致性**：结论是否与财务数据一致？如“高增长”但营收增速为负。
3. **行业常识**：判断是否符合行业规律？如周期股用成长股估值框架。

Critic 用专门的 prompt 训练，里面包含大量投资逻辑错误案例。

### 45. 为什么不让 Critic Agent 直接重新生成报告？

如果 Critic 直接生成报告，它既是“裁判”又是“运动员”，失去独立性。而且 Critic 的输出是“问题清单”，Repair 根据清单修改，Audit 再验证，形成分工闭环。重新生成会让错误难以追溯。

### 46. Critic 发现问题后，系统如何处理？

1. Critic 输出修复清单；
2. Repair Agent 根据清单修改 Analyst 的 JSON；
3. Reporter 用修复后的 JSON 生成报告；
4. Audit Engine 检查最终报告是否还有问题；
5. 如果仍有严重问题，触发 Analyst 重推理（最多 3 次）。

### 47. Critic 是一次性检查还是循环优化？

设计上是一次性检查 + 可选循环。默认 Critic 只跑一次，输出问题清单。如果问题严重，Repair 后 Reporter 输出，Audit 再次检查。如果 Audit 仍有重大问题，循环回 Analyst 重推理。循环有次数上限，避免无限迭代。

### 48. 如果 Analysis Agent 和 Critic Agent 意见冲突怎么办？

分两种情况：
1. **事实冲突**（如数据解读不同）：以代码层计算为准，Critic 必须基于代码预检结果；
2. **判断冲突**（如行业周期判断不同）：保留分歧，在报告中以 `[Framework Divergence]` 呈现两种视角，让用户判断。

### 49. 多个 Agent 如何解决意见不一致？

1. 事实层面：代码层计算说了算；
2. 判断层面：保留分歧，报告中同时呈现；
3. 严重冲突：触发 Audit 和重推理，最终由 Audit 决定是否通过。

### 50. 如何设计 Agent 的置信度评分？

可以从以下维度综合：
- 数据完整度（缺失字段越少置信度越高）；
- 评分极端程度（分数分布均匀还是两极化）；
- Critic 问题数（问题越少置信度越高）；
- 数据一致性（财务指标是否自相矛盾）；
- 历史表现（同类型公司历史预测胜率）。

目前 FinBrain 的“置信度”由评分系统根据数据完整度和一致性计算，未来可以引入历史回测校准。

### 51. 如何判断 Agent 输出质量？

1. 代码层：输出是否符合 schema、字段是否完整、数值是否合理；
2. Critic 层：问题数量和严重程度；
3. Audit 层：最终报告是否通过所有检查；
4. 人工评估：可读性、逻辑性、可投资性；
5. 回测：交易建议的长期表现。

### 52. 你的系统有没有自动 Evaluation？

有基础版。`backend/evaluation.py` 和 `tests/test_e2e.py` 提供：
- 评分一致性（同一只股票多次运行得分一致）；
- 字段完整性（报告是否包含关键字段）；
- 工具成功率（数据获取是否成功）；
- 输出合规性（评级与操作一致性、EPS 单调性）。

但还缺少“投资建议收益评估”——这是 Phase 2 的重点。

### 53. 如何构造 Agent Benchmark？

1. 选一组股票，覆盖不同行业、周期阶段、数据完整度；
2. 让 FinBrain 和普通 GPT（或基线模型）分别生成报告；
3. 人工盲评：数据准确性、逻辑一致性、风险覆盖、估值合理性、操作建议具体性；
4. 统计胜率、召回率、F1；
5. 用回测验证交易建议的收益表现。

### 54. 如何测试你的 Agent 比普通 GPT 提升多少？

- 对比实验：同一只股票，FinBrain 和普通 GPT 各生成 10 份报告；
- 人工盲评 + 自动指标；
- 关键指标：数据错误率、逻辑矛盾率、风险遗漏率、估值算术错误率。

FinBrain 的核心优势应该是：数据错误率更低、逻辑一致性更高、估值更合理。

### 55. 如何设计 Offline Evaluation？

1. 构建固定测试集（股票列表 + 已知财报数据）；
2. 运行 FinBrain，收集报告和 Trace；
3. 自动检查：字段完整性、评分一致性、算术正确性；
4. 人工抽查：投资逻辑合理性、风险覆盖；
5. 统计指标，回归时对比。

### 56. 如何设计 Online Evaluation？

1. 跟踪生成的投资建议，记录建议时间、价格、评级；
2. 一段时间后（如 3 个月）计算收益；
3. 分析“高评分股票是否跑赢低评分股票”；
4. A/B 测试：不同 Agent 版本的表现对比。

### 57. Agent 系统最大的工程难点是什么？

**可控性**。Agent 比传统软件更不可控：LLM 可能输出错误、工具可能失败、多 Agent 可能互相放大错误。工程难点是：
- 如何把不确定性限制在可控边界内；
- 如何快速定位哪个 Agent 出错；
- 如何在不牺牲智能的前提下保证输出质量。

### 58. Agent 系统为什么容易出现不可控行为？

因为 LLM 是概率性模型，输出不是确定函数。加上多 Agent 协作、工具调用、状态共享，一个小错误可能在下游被放大。而且 LLM 可能产生“隐式规划”——做 prompt 没明确要求的事情。

### 59. 如何 Debug 一个 Agent 系统？

1. 看 Execution Trace：哪个节点失败、哪个工具返回 error；
2. 看 State 快照：每个节点前后的状态变化；
3. 看 LLM 输入输出：prompt 是否给了足够上下文、输出是否被正确解析；
4. 看 Critic 和 Audit 日志：问题被哪个环节发现；
5. 单元测试：隔离每个 Agent 单独测试。

### 60. 如何记录 Agent Execution Trace？

每个节点执行时写入 `processing_log`：节点名、状态、耗时、关键动作、错误信息。LangGraph SqliteSaver 也保存完整 state。前端用 collapsible panel 展示时间线，用户可以展开看每个节点的输入/输出。

### 61. 为什么 Execution Trace 对 Agent 很重要？

因为 Agent 的输出不是单一 LLM 的结果，而是多节点协作的结果。Trace 让用户和开发者知道：
- 结论从哪些数据来；
- 哪个节点做了什么；
- 错误发生在哪一步；
- 如何改进。

没有 Trace，Agent 就是黑盒，无法信任。

### 62. 你的系统如何做到可观测性？

1. **节点级别**：每个节点记录状态、耗时、token 消耗；
2. **工具级别**：记录工具调用、参数、返回值、缓存命中率；
3. **LLM 级别**：记录模型名、prompt 长度、输出长度、是否 fallback；
4. **前端级别**：Streamlit 展示 Trace、评分、审计结果；
5. **日志级别**：文件日志记录错误和异常。

### 63. 如何定位一次错误来自哪个 Agent？

通过 `processing_log` 里的 `phase` 字段。每个节点写入自己的 phase 和状态。如果最终报告出错，往前回溯：
- 是数据错了？→ Data Agent 或工具层；
- 是分析错了？→ Analyst Agent；
- 是估值错了？→ Valuation Agent 或 Classifier；
- 是审查没发现？→ Critic 或 Audit。

### 64. 如果一个 Agent latency 很高，如何优化？

1. 缓存：数据工具和 LLM 结果缓存；
2. 并行：Critic 三路并行；Data Agent 多工具并行；
3. 异步：把 LLM 调用改成 async；
4. 模型降级：简单任务用小模型；
5. 预取：scheduler 提前拉取热门股票数据。

### 65. 如何降低 LLM 调用成本？

1. 缓存 RAG 检索结果和常见股票分析；
2. 数据工具用缓存，避免重复请求；
3. Critic 先做代码预检，减少 LLM 输入；
4. 用更便宜的模型做简单任务，贵模型只做复杂分析；
5. Prompt 压缩，减少上下文长度；
6. 未来接入 Anthropic prompt caching / DeepSeek 上下文缓存。

### 66. 如何减少不必要的 Agent 调用？

1. Audit 预检：如果确定性检查都通过，LLM 审计只跑 warning-only 模式；
2. Critic 过滤：代码层先检查明显错误，减少 LLM 审查输入；
3. 缓存命中：同一只股票短期内直接复用分析；
4. 条件路由：Critic 无问题时跳过 Repair。

### 67. 为什么不能所有步骤都调用 GPT？

1. 成本高：每次调用都要花钱；
2. 不稳定：LLM 输出有随机性；
3. 不必要：财务计算、评分、审计规则完全可以用代码做得更好；
4. 慢：多步骤都调 GPT 会显著增加延迟。

GPT 应该只用于它真正擅长的地方：语义理解、综合判断、叙事生成。

### 68. 你的系统如何设计缓存？

两层缓存：
1. **本地内存缓存**：TTL 过期，用于股价、财报、估值等高频数据；
2. **Redis 缓存**：远程模式共享缓存，用于多用户场景。

缓存 key 包含股票代码、数据源、时间窗口。写缓存时记录命中率和过期时间。

### 69. 哪些结果可以 Cache？

- 数据工具结果：股价、财报、估值、行业、公告；
- RAG 检索结果：知识库不变时可复用；
- 同一只股票的分析结果：短期内复用（需注意数据时效性）；
- Prompt 模板和策略配置。

不能缓存：依赖用户实时输入的结果、高度个性化的分析。

### 70. Prompt Engineering 在你的系统中起什么作用？

Prompt 是 Agent 行为的“说明书”。它定义了：
- 每个 Agent 的角色和任务边界；
- 输出格式（JSON schema、字段说明）；
- 分析框架（10 步投资框架、估值约束）；
- 禁止事项（不预测精确买卖点、不给出仓位）。

但 Prompt 不是万能：再强的 Prompt 也替代不了代码层校验和审计。

### 71. 如何设计高质量 Agent Prompt？

1. 角色明确：Agent 是“做什么的”，不是“万能的”；
2. 输出格式具体：给出 JSON 示例和字段说明；
3. 边界清晰：明确 Agent 不能做什么（如 Analyst 不能调用工具）；
4. 步骤化：复杂任务拆成可执行的步骤；
5. 示例驱动：给 1-2 个输入输出示例；
6. 注入上下文：用 `[INDUSTRY]`、`[TOOLS]` 占位符填充真实数据。

### 72. System Prompt、User Prompt、Context 的优先级是什么？

- **System Prompt**：最高优先级，定义角色和全局规则；
- **Context**：中间优先级，提供当前任务需要的数据；
- **User Prompt**：最低优先级，通常是具体指令。

但 LLM 对位置敏感，最重要的约束应放在 prompt 最前面或最后面（重复强调）。

### 73. 如何防止 Prompt Injection？

1. 不对用户输入直接拼接进 LLM prompt；
2. 用户输入只作为路由参数或股票代码，经过校验后再使用；
3. 系统 prompt 明确禁止 LLM 泄露配置、执行非分析任务；
4. 工具层对参数做校验，避免恶意输入触发外部请求。

### 74. 如果用户输入恶意指令影响 Agent 怎么办？

1. 输入层：限制股票代码格式，过滤非股票输入；
2. Prompt 层：明确 Agent 只处理财报分析，不执行其他指令；
3. 工具层：参数校验，防止 SQL 注入或非法 URL；
4. 输出层：审计检查输出是否偏离分析任务。

### 75. Tool Calling 和普通函数调用有什么区别？

普通函数调用：代码直接调用函数。Tool Calling：LLM 输出“我要调用某某函数”，框架拦截、执行、回传结果。区别在于谁决定调用：普通函数调用由代码决定；Tool Calling 由 LLM 自主决定。

### 76. 你的 Agent 如何选择调用哪个 Tool？

LLM 根据 tool 的 docstring 和当前任务描述，输出函数名和参数。Data Agent 内部使用 `create_agent` + tools，LLM 自主决定调用顺序。工具描述写得好，LLM 选择就越准确。

### 77. Tool 返回错误数据怎么办？

1. 工具层重试 3 次；
2. 切换到 fallback 数据源（`datasource_tier.py`）；
3. 返回带 `error` 字段的结果，不抛异常；
4. 下游 Agent 看到 `error` 后，在报告中标注数据缺失；
5. Audit 检查数据缺失是否导致结论错误。

### 78. Tool Calling 是否需要 Planner？

不一定。简单任务可以直接把所有工具给 Agent，LLM 自己决定调哪个。复杂任务可以先由 Planner 选择工具子集，再让 Agent 调用。FinBrain 的 Data Agent 不需要 Planner，因为它需要获取所有相关数据。

### 79. Function Calling 和 MCP 有什么区别？

Function Calling 是 LLM 调用函数的通用机制。MCP（Model Context Protocol）是 Anthropic 提出的标准化协议，让外部工具以统一格式暴露给多个 AI 客户端。Function Calling 是“能力”，MCP 是“协议”。FinBrain 的 `MCP_server.py` 用 MCP 暴露部分工具给 Claude Code。

### 80. 为什么现在很多 Agent 使用 MCP？

MCP 让工具一次编写，多处复用（Claude Code、Cursor、其他 MCP 客户端）。对 FinBrain 来说，MCP Server 可以让外部 AI 工具直接调用我们的数据工具，而不需要单独集成。

### 81. 如果让你把系统改造成 MCP Server，你怎么设计？

1. 把 `tools.py` 里的函数包装成 MCP tool；
2. 暴露数据查询接口（股价、财报、估值、行业）；
3. 不暴露分析流程，只暴露数据层，避免外部客户端绕过 Critic/Audit；
4. 配置 SSE / stdio 传输协议；
5. 保持认证和日志。

### 82. 你的系统如何扩展新的数据源？

1. 在 `tools.py` 新增一个函数；
2. 在 `datasource_tier.py` 注册数据源 ID；
3. 在 `frontend/app.py` 的 Data Sources 设置里添加选项；
4. 不需要改 Agent 核心逻辑，因为 Data Agent 只调用工具接口。

### 83. 数据源增加后需要修改多少代码？

理想情况下只改工具层和数据源配置层。Agent 层通过工具接口调用，不需要感知具体数据源。但如果新数据源返回不同字段格式，可能需要新增适配器。

### 84. 如何设计插件化 Agent 架构？

1. 工具接口标准化：所有工具返回统一格式的 dict；
2. Agent 节点通过接口调用，不依赖具体实现；
3. 策略配置热加载（`strategies.json`）；
4. 数据源 tier 可插拔；
5. 未来 Agent 本身也可以插件化：通过配置文件定义节点和边。

### 85. Agent 如何动态加载能力？

目前通过 `strategies.json` 热加载策略（prompt 和触发词）。未来可以扩展：
- 动态加载 Agent 节点 Python 模块；
- 通过配置文件注册新工具；
- 运行时切换 LLM 模型和 fallback 链。

### 86. 为什么你的 Data Agent 和 Analysis Agent 要分开？

1. 权限隔离：Analyst 不能调用工具，避免幻觉；
2. 确定性：数据获取是代码行为，分析是 LLM 行为；
3. 可复用：Data Agent 获取的数据可以复用给多个下游 Agent；
4. 可调试：数据错了找 Data Agent，分析错了找 Analyst。

### 87. 如果 Data Agent 返回错误财务数据怎么办？

1. 工具层有数据源 fallback；
2. 数据返回有 `error` 字段标注；
3. Analyst 看到缺失数据会降低置信度；
4. Audit 检查数据一致性；
5. 严重错误会触发重试或报告数据不可用。

### 88. 如何保证金融数据准确性？

1. 多源交叉验证（如财报 API + 公告快报）；
2. 代码层计算指标，不依赖 LLM；
3. 数据单位统一处理（亿元/万元）；
4. 异常值检测（负利润、PE 异常）；
5. 时效性检查（快报已出但结论说等待半年报）。

### 89. 为什么财报分析不能完全依赖 LLM？

LLM 会幻觉、会算错数、会用错行业框架、会混淆数据单位。财报分析对数字准确性和逻辑一致性要求极高，必须把计算、校验、审计交给代码，LLM 只负责语义理解和叙事。

### 90. 你的系统如何避免金融幻觉？

1. 工具隔离：Analyst 无 tools，不能编造数据；
2. 代码层覆盖：Reporter 用代码计算覆盖 LLM 可能乱改的分数和评级；
3. 数据溯源：报告中标注数据来源；
4. Critic 和 Audit 检查数据与结论一致性；
5. 结构化输出：减少自由文本中的胡编。

### 91. 财务指标计算为什么应该代码化？

因为财务计算有明确公式和单位，代码化可以：
- 保证可复现；
- 统一单位（亿元/万元）；
- 检测异常；
- 不依赖 LLM 的算术能力；
- 便于审计和测试。

### 92. 为什么估值模型应该由规则引擎控制？

估值模型需要透明、可解释、可审计。规则引擎（如 `scoring_config.py` + `scoring.py`）可以明确展示：
- 行业 PE 锚点；
- 折价/溢价因子；
- 最终 PE 和合理价值。

如果完全由 LLM 控制，估值会变成黑盒，无法解释为什么给 22 倍 PE。

### 93. 为什么 PE 估值不能简单套行业 PE？

因为同行业内公司质地差异很大：
- ROE 不同；
- 成长性不同；
- 资本开支强度不同；
- 周期位置不同。

简单套行业 PE 会高估周期股顶部、低估困境反转股。FinBrain 用 PE 折价链（行业 PE → 周期折价 → ROE 折价 → CAPEX 折价 → 成长溢价）来反映公司个体差异。

### 94. 周期股、成长股、价值股估值有什么区别？

| 类型 | 估值重点 | 常用方法 |
|------|---------|---------|
| 周期股 | 用正常化利润，避免周期顶部/底部失真 | 正常化 PE、EV/EBITDA |
| 成长股 | 看未来盈利增速，容忍高 PE | PEG、DCF |
| 价值股 | 看稳定分红和资产价值 | 高股息模型、PB |

FinBrain 的 Classifier 输出混合权重，让估值模型同时考虑多种类型。

### 95. 为什么长电科技不能直接使用成长股估值？

长电科技是半导体封测，兼具周期性和成长性。用纯成长股估值会忽略：
- 封测行业周期波动大；
- 资本开支高，折旧压低利润；
- 传统业务成熟，AI 业务占比仍在提升。

所以系统用“50% 周期制造 + 30% 成长制造 + 20% AI 主题”的混合权重，结合正常化 PE、EV/EBITDA、PEG 多种方法。

### 96. 混合权重估值模型如何设计？

1. Classifier 从财务数据提取特征：ROE 波动性、收入增速、CAPEX/CFO；
2. 输出混合权重（如周期 50%、成长 30%、主题 20%）；
3. Valuation Agent 根据权重选择估值方法和参数；
4. 代码层综合各方法结果，生成合理价值和情景估值。

权重由人工规则设定，但输入特征来自真实数据。未来可以用历史回测优化权重。

### 97. 为什么采用“50%周期制造 + 30%成长制造 + 20% AI主题”？

这是针对长电科技这类公司的经验设定，反映：
- 主业务仍是半导体封测，周期属性强；
- 先进封装带来成长属性；
- AI 相关订单提供主题溢价。

权重不是固定公式，而是根据公司特征动态调整。Classifier 输出权重，Valuation Agent 使用。

### 98. 权重如何自动确定？

目前由 Classifier 代码根据财务特征映射到权重。未来可以：
- 用历史数据训练回归模型，预测最优权重；
- 用回测结果反馈优化；
- 引入专家规则库。

### 99. 如何避免估值模型主观性？

1. 权重和调整因子必须透明，报告中展示；
2. 输入数据来自真实财报，不是 LLM 判断；
3. 用多种估值方法交叉验证；
4. 用回测验证历史表现，迭代优化参数。

### 100. PE 折价链为什么需要透明化？

因为投资者需要知道“22 倍 PE 是怎么来的”。透明化 PE 折价链（行业 PE 40 → 周期 -30% → ROE -20% → CAPEX -15% → 成长 +20%）可以：
- 解释估值逻辑；
- 发现不合理折价；
- 让用户/审计者质疑和修正。

不透明的话，估值就是黑盒，无法信任。

### 101. 如果模型认为公司很好但估值过高，最终如何决策？

代码层 `compute_investment_rating` 会给出 SELL 或 HOLD。评级由估值和安全边际决定，不只看公司质量。报告中会解释：公司基本面优秀，但当前估值透支未来，建议等待合理买入区间。

### 102. 投资建议如何从多个 Agent 输出中融合？

1. 代码层综合评分、估值、安全边际，生成基础评级；
2. Analyst 的叙事提供定性判断；
3. Valuation 提供估值框架和区间；
4. Critic 的问题会修正 Analyst 输出；
5. Audit 最终校验评级与叙事一致性。

最终投资建议由代码层规则主导，LLM 提供解释和叙事。

### 103. 你的评分系统如何设计？

评分系统分维度：盈利能力、成长性、财务健康、估值合理，外加动态权重。每个维度有具体指标和阈值。代码层按公司类型动态赋权，计算加权总分，再映射到评级和安全边际。

### 104. 为什么评分不能全部交给 LLM？

评分需要可复现、可审计、不幻觉。LLM 可能根据文本风格给出不同分数，也可能被叙事误导。代码评分基于客观数据，同一只股票 100 次结果一致。

### 105. 评分规则变化如何影响结果？

评分规则变化会直接影响最终评级和买入区间。因此规则调整必须：
1. 用离线测试集验证；
2. 对比调整前后的评分分布；
3. 用回测验证评级与收益的关系；
4. 记录版本，便于回滚。

### 106. 如何避免评分模型过拟合？

1. 规则基于财务逻辑，不是数据挖掘；
2. 用多只股票、多周期回测验证；
3. 避免用未来信息；
4. 定期用新数据测试规则稳定性。

### 107. 你的系统有没有使用 Chain of Thought？

有。Analyst 的 prompt 要求按 10 步框架逐步推理：公司分类 → 商业模式 → 护城河 → 行业周期 → 财务验证 → 成长逻辑 → 市场预期 → 情景估值 → 催化剂 → 决策。Critic 也要求指出具体问题，而不是简单打分。

### 108. 如何保护模型内部推理过程？

目前系统不隐藏推理过程，反而把 Execution Trace 展示给用户，增强可解释性。如果未来需要保护，可以：
- 只展示最终结论和关键推理摘要；
- 内部 reasoning 不写入持久化日志；
- 用脱敏后的摘要替代原始 prompt。

### 109. 如何设计结构化输出？

用 Pydantic 模型定义每个 Agent 的输出 schema。字段尽量 Optional，允许数据缺失时仍返回合法对象。用 LangChain `with_structured_output(schema)` 让模型直接输出对象。失败时手动解析并回退。

### 110. 为什么 Agent 输出应该 JSON 化？

1. 下游 Agent 可以精确读取字段；
2. 代码层可以做校验和计算；
3. 便于存储和索引；
4. 避免自然语言的信息丢失和歧义。

### 111. 如何保证 LLM 输出符合 Schema？

1. 用 Pydantic 定义 schema；
2. `with_structured_output()` 在模型解码层约束；
3. 代码层二次校验；
4. 失败时回退或重试；
5. 所有字段 Optional，降低解析失败率。

### 112. 如果 LLM 输出格式错误怎么办？

1. 用 `raw_decode` 或正则提取 JSON；
2. 用 Pydantic 校验，修复可兼容错误；
3. 切换到下一个 LLM 槽位重试；
4. 仍失败则返回降级输出或明确错误。

### 113. 如何做自动重试？

LangChain 本身支持 retry decorator。FinBrain 在关键节点（LLM 调用、JSON 解析、审计）做有限重试，并记录重试次数到 `processing_log`。达到上限后停止，避免无限循环。

### 114. 为什么需要代码 Precheck？

代码 Precheck 可以在 LLM 调用前过滤掉明显错误，减少不必要的 LLM 调用。例如，如果 FCF 已经算出为负，可以直接注入风险，不需要 LLM 再“发现”。这样既省钱又稳定。

### 115. 你的 Audit System 检查哪些问题？

9 项检查：
1. 评分合计等于各维度得分之和；
2. 评级与操作建议一致性；
3. 情景估值 PE×EPS 是否等于价格；
4. 情景价格单调性（悲观 ≤ 基准 ≤ 乐观）；
5. 估值差距在合理范围；
6. 现金流标签与数据一致；
7. 稀释系数正确应用；
8. 时效性（快报已出但结论等待半年报）；
9. 风险清单非空。

### 116. 为什么 Audit 不直接替代 Critic？

Audit 只能检查规则能覆盖的问题，Critic 能做语义理解。Audit 是“硬约束”，Critic 是“软检查”。两者互补：Audit 兜底，Critic 发现规则写不到的问题。

### 117. Rule-based Audit 和 LLM Critic 如何结合？

1. 规则先跑，快速过滤明显错误；
2. 规则通过的问题，Critic 做语义级检查；
3. Critic 发现的问题，代码层 Audit 验证是否真实存在；
4. 两者都通过，报告才发布。

### 118. 如果重新设计整个系统，你会怎么拆模块？

1. **数据层**：数据工具 + 缓存 + 数据源 tier；
2. **RAG 层**：知识库、检索、向量化；
3. **Agent 编排层**：LangGraph 节点和边；
4. **Agent 实现层**：Analyst、Valuation、Critic、Repair、Reporter；
5. **计算层**：评分、估值、财务指标；
6. **审计层**：规则审计、LLM 审计；
7. **前端层**：Streamlit UI + Trace 展示；
8. **API/部署层**：FastAPI、Redis、任务调度。

### 119. 你的架构最大的技术债是什么？

1. `agent.py` 过大（约 2900 行），节点和 prompt 混在一起，未来应拆分；
2. 数据工具强依赖国内免费 API，稳定性差；
3. LLM 调用仍是同步，扩展性受限；
4. 缺少长期 Memory 和用户级持久化。

### 120. 如果用户量增加 100 倍，你怎么扩展？

1. 部署：远程模式 + FastAPI + Redis + 负载均衡；
2. 数据：预取热门股票，多层缓存；
3. LLM：异步队列、批量请求、小模型降级；
4. 数据库：持久化 State 和历史报告；
5. 监控：记录每个节点延迟和成功率，快速扩容。

### 121. 如何设计异步 Agent 执行？

1. 把 LangGraph 节点改成 async；
2. 数据工具用 async HTTP client；
3. LLM 调用用 async LangChain 接口；
4. 用 FastAPI 接收请求，放入 Celery/RQ 队列；
5. 前端用 WebSocket/SSE 实时推送进度。

### 122. 如何设计任务队列？

1. 用 Redis 做队列；
2. 每个分析任务一个 job；
3. Worker 从队列取任务，执行 LangGraph；
4. 任务状态（pending/running/done/failed）写入 Redis；
5. 前端轮询或 SSE 获取状态。

### 123. 如何保证 Agent 执行幂等？

1. 任务 ID 唯一；
2. 数据依赖股票代码和时间窗口，不依赖用户请求顺序；
3. 缓存工具结果，重复执行结果一致；
4. LLM 用 `temperature=0`，输出稳定。

### 124. 如何部署你的 Agent 系统？

两种模式：
1. **本地模式**：用户 `pip install` 后 `streamlit run frontend/app.py`，API Key 存在本地 `.env`；
2. **远程模式**：服务器跑 FastAPI + Redis + scheduler，客户端只做 LLM 推理。数据 API 纯读，可横向扩展。

### 125. 为什么选择这个技术栈？

LangGraph：固定流程 + 状态持久化；LangChain：Tool/Model 统一接口；FastAPI：数据服务；Streamlit：快速前端；ChromaDB：本地 RAG；Redis：共享缓存。选型原则是“够用、可本地运行、零运维成本”。

### 126. FastAPI 在系统中的作用是什么？

FastAPI 是远程模式的数据服务层。它暴露数据接口（股价、财报、估值等），让前端或客户端调用。好处是：数据获取可以集中缓存、预取、限流，而 LLM 推理仍在本地或私有服务器。

### 127. 如何设计数据库存储 Agent Trace？

1. 表：trace（trace_id, user_id, stock_code, created_at, status）；
2. 表：node_log（trace_id, node_name, phase, input_summary, output_summary, latency_ms, error）；
3. 表：report（trace_id, report_content, scores, rating）；
4. 用 JSONB 存结构化字段，方便查询。

### 128. 如何保存历史分析结果？

当前通过 Checkpointer 保存 graph 状态。未来可写入数据库：
- 用户维度：保存用户所有历史报告；
- 股票维度：保存某只股票的历史评分和评级变化；
- 全局维度：保存系统整体统计指标。

### 129. 如何支持多用户？

1. 用户认证（登录/注册）；
2. 每个用户独立 `thread_id` 和 API Key；
3. 数据库按 user_id 隔离历史报告；
4. Redis 缓存按 user_id 前缀隔离；
5. 服务端不存储用户 API Key，只在本地 .env 使用。

### 130. 如何保护用户金融数据？

1. 用户 API Key 只存在本地 `.env`；
2. 远程模式只接收股票代码，不接收 API Key；
3. 数据接口只读，不做写操作；
4. 日志脱敏，不记录 API Key；
5. 传输用 HTTPS。

### 131. 你的项目中最大的技术挑战是什么？

让 LLM 在“有创造力”和“受约束”之间取得平衡。太约束会限制分析质量，太自由会产生幻觉和不一致性。解决方法是：结构化输出 + 代码层覆盖 + Critic 审查 + Audit 兜底。

### 132. 你遇到过什么错误？如何定位？

典型错误：LLM 输出 JSON 中把“情景估值”的价格写错，导致悲观价 > 合理价值。定位方法：
1. 看 Audit 日志，发现情景单调性检查失败；
2. 看 Analyst 输出，确认 LLM 算错；
3. 加入 `_validate_scenarios` 代码校验；
4. 修复后重新跑测试。

### 133. 如果重新做一次，你会改什么？

1. 更早引入 Pydantic 结构化输出；
2. 更早把 `agent.py` 拆成多个文件；
3. 更早接入多数据源交叉验证；
4. 更早建立离线 benchmark。

### 134. 这个项目最大的创新点是什么？

“确定性计算 + 概率性推理的强制边界”。不是简单让 LLM 写研报，而是用代码固定财务指标、评分、估值，用 LLM 做叙事和审查，用 Critic 和 Audit 做反向验证。这种设计在金融领域特别有效。

### 135. 和普通 ChatGPT 分析股票相比，你的优势在哪里？

1. 数据来源：真实 API，不是训练数据；
2. 计算层：代码化财务指标和估值，不幻觉；
3. 验证层：Critic + Audit 双重检查；
4. 可解释性：Trace 展示数据来源和推理过程；
5. 领域知识：RAG 注入行业模板。

### 136. 你的项目中哪些地方体现工程能力？

1. 数据工程：多数据源、缓存、TTL、数据 tier；
2. 并发：ThreadPoolExecutor、Redis、FastAPI；
3. 测试：68 项 e2e 测试；
4. 可观测性：Execution Trace、processing_log；
5. 容错：fallback 数据源、LLM fallback chain、条件路由。

### 137. 哪些地方体现 AI Agent 能力？

1. 多 Agent 协作：Data、Analyst、Valuation、Critic、Repair；
2. 自主工具调用：Data Agent 决定调用哪些工具；
3. 反向验证：Critic 找问题、Audit 兜底；
4. 错误恢复：数据缺失重取、LLM 失败切换；
5. 结构化输出：Pydantic schema 约束 Agent 行为。

### 138. 哪些地方体现算法能力？

1. 财务指标计算：TTM、年化、FCF、稀释系数；
2. 评分系统：多维度动态加权；
3. 估值模型：PE 折价链、混合权重、安全边际；
4. 审计规则：时效性检测、情景校验、评分一致性；
5. 数据融合：业绩快报双通道合并。

### 139. 如果面试官说“这不就是几个 Prompt 串起来吗”，你怎么回答？

不是。Prompt Chain 是线性调用 LLM，没有反馈、没有工具、没有状态、没有验证。FinBrain 有：
- 工具调用获取真实数据；
- LangGraph 状态管理和条件路由；
- 代码层计算和审计；
- Critic 反向验证和 Repair 自动修正；
- 结构化输出和 Execution Trace。

Prompt 只是其中一个组件，不是全部。

### 140. 如果面试官说“为什么不用一个 GPT Prompt 完成”，你怎么回答？

一个 Prompt 完成所有任务，会出现：
1. 输出太长，容易遗漏字段；
2. 没有中间校验，幻觉和计算错误无法被检查；
3. 无法复用数据获取和计算逻辑；
4. 难以调试，不知道哪一步出错；
5. 无法做工具调用和状态管理。

多 Agent + 结构化输出 + 代码校验，把大问题拆成可控的小问题，每个小问题都有明确输出和检查。

### 141. 你的项目未来最值得优化的方向是什么？

1. 回测与评估闭环（Phase 2）：验证投资建议是否真的能赚钱；
2. Human-in-the-Loop（Phase 3）：关键交易节点人工审批；
3. 数据源升级：接入付费数据，提升一致预期和产业链覆盖；
4. 长期 Memory：用户偏好、历史报告、跨股票对比；
5. 异步和 Streaming：提升并发和实时体验。

### 142. 如果加入更多 Agent，你会增加哪些角色？

1. **Macro Agent**：宏观环境判断（利率、政策、行业景气）；
2. **News Agent**：实时新闻和事件解读；
3. **Peer Agent**：同行业公司对比；
4. **Risk Agent**：专门识别尾部风险（应收账款、商誉、质押）；
5. **Portfolio Agent**：组合层面风险和相关性分析。

### 143. 如果加入 Human-in-the-loop，你怎么设计？

1. 在模拟交易 `place_order` 前加入 LangGraph `interrupt()`；
2. 前端弹窗展示交易建议、理由、风险，等待用户确认/拒绝；
3. 用户确认后继续执行，拒绝则记录原因并终止；
4. 审批日志写入数据库，便于审计。

### 144. 如何让 Agent 自我改进？

1. 收集 Critic 和 Audit 的反馈；
2. 用人工标注纠正错误案例；
3. 定期用新数据微调 prompt；
4. 建立离线 benchmark，验证改进是否有效；
5. 把高频错误注入 prompt 或规则。

### 145. Agent 的未来发展方向是什么？你怎么看？

Agent 发展方向：从“单轮回答”到“多轮自主执行”，从“通用”到“垂直”，从“黑盒”到“可审计”。在金融领域，Agent 最有价值的不是替代人类做决策，而是承担“数据收集、初步分析、错误检查”的重复工作，让人类专注于判断和决策。关键是“可信”——没有可审计性和可控性的 Agent 无法落地。
