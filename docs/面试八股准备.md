# FinBrain 面试准备 — Agent / RAG / 系统工程

---

## 一、Agent 架构与工作模式

### 1. 你怎么理解 Agent？它和普通 LLM 应用最大的区别是什么？

**核心区别：Agent 能自主决策行动序列，普通 LLM 应用只能做单次推理。**

```
普通 LLM 应用：输入 → LLM → 输出（一次性，无交互）
Agent：       输入 → LLM → 决策 → 调工具 → 拿到结果 → 再决策 → ... → 输出
```

三个关键能力让 Agent 区别于普通 LLM：

| 能力 | 说明 | FinBrain 实例 |
|------|------|-------------|
| **工具使用** | 调用外部函数/API 获取真实数据 | 10 个 `@tool`，LLM 自主决定调用哪个 |
| **记忆** | 跨轮次保留上下文 | SqliteSaver 持久化对话状态 |
| **规划** | 分解复杂任务为多步执行 | DataCollector→Analyst→Reporter 三节点流水线 |

**比喻**：普通 LLM 是百科全书的检索员，Agent 是能自己上网查资料、做笔记、出报告的助理。

---

### 2. Agent 常见的工作模式有哪些？

| 模式 | 核心思想 | 适用场景 | 项目落地 |
|------|---------|---------|---------|
| **ReAct** | Thought→Action→Observation 循环，思考一步做一步 | 开放式任务 | DataCollector 子 Agent 用 `create_agent`（内置 ReAct） |
| **Plan-and-Execute** | 先生成完整计划，再逐步执行 | 多步骤复杂任务 | 三节点流水线：先调度→搜集→分析→报告 |
| **Router** | 根据输入分类路由到不同处理器 | 多意图系统 | `_classify_request()` 分发到 Chat/Analysis/Phantom |
| **Multi-Agent** | 多个专业化 Agent 协作 | 复杂系统 | DataCollector + Analyst + Reporter + Phantom Hunter |
| **Reflection** | 执行后自我评估，必要时重试 | 质量要求高的输出 | 计划中的 Auditor Agent |

**我们对 ReAct 的落地**：

DataCollector 子 Agent 使用 `create_agent`（LangChain 内置 ReAct 实现），LLM 在 Thought→Action→Observation 循环中自主决定调哪些工具。但外层编排用 StateGraph 固定流水线——ReAct 用于"要调哪些工具"这个开放子问题，StateGraph 用于"分析流程"这个确定性问题。各取所长。

---

### 3. 项目里有没有用到现成框架？

**LangGraph + LangChain**。

架构分层：

```
编排层: LangGraph StateGraph (手写节点和边)
  └── 子Agent: LangChain create_agent (ReAct循环)
        └── Tool层: LangChain @tool 装饰器
              └── 数据层: tools.py (纯Python，零框架依赖)
```

选择 LangGraph 而非 AutoGen/CrewAI 的原因：
- **确定性编排**：我的 Data→Analyze→Report 流程是固定的，不需要 Agent 间"协商"
- **显式状态管理**：FinBrainState TypedDict 定义了每个字段谁写谁读
- **工具权限控制**：Analyst 和 Reporter 无 tools，从框架层面保证隔离
- AutoGen 更适合开放式多 Agent 对话，我的场景不需要

### 4. 意图识别模块是怎么做的？

```python
def _classify_request(user_input: str) -> str:
    triggers = _STRATEGY["triggers"]  # 从策略配置读取触发词
    if any(t in user_input for t in triggers.get("phantom", [])):
        return "phantom"   # → Phantom Hunter Agent
    if any(t in user_input for t in triggers.get("analysis", [])):
        return "analysis"  # → 三节点深度流水线
    return "chat"          # → 轻量聊天 Agent
```

设计考量：
- **触发词可配置**：不同策略可定义不同触发词（默认 vs 纯价值 vs 趋势动量）
- **简单但有效**：关键词匹配而非 LLM 分类，零延迟、零 token 消耗
- **可升级路径**：如果规则增多（>20 条），可以切换到 LLM 分类（`LLM.invoke` + 分类 Prompt），或训练一个 BERT 分类器

---

### 5. 怎么避免路径震荡（重复规划）？

**问题**：Agent 在 ReAct 循环中可能反复调同一个工具、陷入无限循环。

**我们的防御措施**：

1. **StateGraph 固定流水线** — 节点间不形成循环，执行路径在编译时确定
2. **DataCollector 只跑一轮** — Prompt 明确要求"调完所有工具后汇总"，不反复
3. **max_tokens 限制** — LLM 配置 `max_tokens=4096`，天然限制循环深度
4. **如果后续加循环** — 可用 LangGraph 的 `interrupt_before` + `max_iterations` 参数

## 二、工具调用与 MCP

### 6. 项目里有哪些工具？怎么注册和管理？

**10 个工具**：

| # | 工具 | 数据源 | Agent 可用范围 |
|---|------|--------|-------------|
| 1 | stock_price | 新浪财经 | Chat / Analysis / Phantom |
| 2 | stock_history | 新浪财经 | Chat / Analysis / Phantom |
| 3 | financial_statements | 东方财富 datacenter | Analysis / Phantom |
| 4 | valuation | 东方财富 datacenter | Analysis / Phantom |
| 5 | industry_info | 东方财富+同花顺 | Analysis / Phantom |
| 6 | screen_stocks | 新浪全市场 | Analysis |
| 7 | fund_flow | 同花顺 10jqka | Analysis / Phantom |
| 8 | limit_up_pool | 新浪 | Phantom |
| 9 | concept_ranking | 同花顺 | Phantom |
| 10 | dragon_tiger_list | 新浪(AkShare) | Phantom |

**工具注册流程**：

```python
# Step 1: tools.py 写纯函数
def fetch_stock_price(symbol: str) -> dict: ...

# Step 2: agent.py 用 @tool 装饰器包装
@tool
def stock_price(symbol: str) -> str:
    """查询A股实时价格。输入股票代码如 '601991'"""  # ← docstring = 工具描述
    from backend.tools import fetch_stock_price
    return json.dumps(fetch_stock_price(symbol), ensure_ascii=False)

# Step 3: 注册到 Agent
_CHAT_TOOLS = [stock_price, stock_history, ...]           # 聊天 Agent
_PHANTOM_TOOLS = [limit_up_pool, ...]                      # 妖股 Agent
_data_collector_tools = [stock_price, ..., screen_stocks]  # 分析 Agent
```

**管理方式**：不同 Agent 注册不同工具集，实现**工具权限隔离**。Analyst 和 Reporter 节点根本没有 tools 参数。

---

### 7. 工具调用 vs Function Calling vs 普通 Prompt

| 维度 | 普通 Prompt | Function Calling | 工具调用 (Tool Use) |
|------|-----------|-----------------|-------------------|
| LLM 输出 | 自由文本 | JSON `{name, arguments}` | JSON `{name, arguments}` |
| 谁执行 | 无 | 调用方代码 | 框架（LangGraph 自动拦截） |
| 结果回传 | 无 | 调用方手动拼接 Prompt | 框架自动插入上下文 |
| 多轮 | 不支持 | 需手动实现循环 | 框架内置 ReAct 循环 |
| 框架依赖 | 无 | 无 | LangChain/LangGraph |

**FinBrain 里的实际流程**：

```
LLM 输出: {"name": "stock_price", "arguments": {"symbol": "300502"}}
    ↓
LangGraph 框架拦截（不是我们手写的 if/elif）
    ↓
执行 stock_price("300502") → 返回 {"name": "新易盛", "price": 510}
    ↓
框架自动把结果插入消息列表
    ↓
LLM 看到结果后继续思考 → 可能调更多工具 → 最终输出文本
```

### 8. 你怎么理解 MCP / Prompt / CLI / Skill / Harness？

| 概念 | 定义 | FinBrain 实例 |
|------|------|-------------|
| **MCP** | Model Context Protocol。AI 客户端与工具服务器之间的标准通信协议 | `MCP_server.py`：暴露 5 个 Tool 给 Claude Code |
| **Prompt** | LLM 的系统指令，定义角色、规则、输出格式 | `DATA_COLLECTOR_PROMPT`、`ANALYST_PROMPT`、`_FORMAT_MANDATORY` |
| **CLI** | 命令行交互入口 | `run.py`：终端 Agent 交互 |
| **Skill** | MCP 生态里的可复用能力单元，类似插件 | 每个 `@tool` 可视为一个 Skill |
| **Harness** | Agent 工程基座：权限、熔断、编排、记忆、降级 | 工具分级(L1/L2/L3)、SqliteSaver、压缩、路由分发 |

**关系**：MCP 是协议层，Skill/Tool 是能力层，Prompt 是指令层，Harness 是工程层，CLI 是入口层。

**为什么用 MCP？**
- 标准协议：Claude Desktop、Cursor、Windsurf 都支持 MCP
- 一次开发多端使用：`MCP_server.py` 写好后，任何 MCP 客户端都能调
- OAuth 2.1 不是我们当前的需求（单用户），但在 MCP 规范里预留了扩展点

## 三、RAG 与知识库

### 9. 知识库是怎么搭的？从文档接入到检索的流程？

（FinBrain RAG 为计划中模块，以下为设计方案）

```
文档接入 → 切片 → Embedding → 向量库 → 检索 → 重排 → 注入 Prompt
```

**文档接入**：
- 会计准则：PDF → `unstructured` 库提取文本
- 游资资料：结构化 JSON 存储（YOUZI_DB）
- 题材舆情：定时爬取财联社电报 + 政策文件

**切片策略**：
- 会计准则是结构化文档 → 按条款号切（Article-based chunking）
- 游资风格是短文本 → 按游资名称切，每条独立
- 题材舆情是长文本 → 按段落切 + 滑动窗口重叠（overlap=20%）

**Embedding 选型**：
- 首选 `text-embedding-3-small`（OpenAI）：1536 维，性价比高，中文尚可
- 备选 `bge-large-zh-v1.5`（BAAI）：1024 维，中文 SOTA，本地部署零成本
- 选型指标：MTEB 中文榜单排名、推理速度（tokens/s）、向量维度（影响存储）、价格

**检索流程**：
1. 用户 query → embedding → 向量相似度 Top-K（通常 K=5~10）
2. BM25 关键词召回（互补向量检索的语义漂移）
3. 融合排序（Reciprocal Rank Fusion）
4. 注入 LLM Prompt 的 context 字段

---

### 10. Lost in the Middle 怎么处理？

**现象**：LLM 对长文档开头和结尾信息记得牢，中间部分容易忽略。

**解决方案**：

1. **切片时加元数据标注** — 每条 chunk 带上位置信息（文档标题/章节/页码）
2. **重排时位置加权** — 靠前和靠后的 chunk 适当提权
3. **分层检索** — 先搜章节标题（粗粒度），再搜段落（细粒度）
4. **Prompt 显式提示** — "以下信息按重要性排序，请优先使用排在前面的内容"

---

### 11. 怎么控制模型幻觉？

**分层防御**：

| 层 | 手段 | 实例 |
|------|------|------|
| **RAG 层** | 检索到的内容必须有来源标注 | 每个 chunk 带 `source: "2025年报第45页"` |
| **Prompt 层** | 明确指令"严禁编造数字""缺失数据标 N/A" | `_FORMAT_MANDATORY` 要求依据栏引用具体数值 |
| **输出约束** | 固定 JSON Schema，缺少字段直接报错 | format_report 无法渲染非法 JSON |
| **工具层** | Analyst 无工具，只能分析已有数据 | 纯 `LLM.invoke()`，无 tools 参数 |
| **验证层**（计划）| Auditor Agent 交叉验证关键数字 | 对比 3 个数据源 |

## 四、系统工程与性能优化

### 12. 针对可靠性与并发，做了哪些优化？

| 优化 | 方案 |
|------|------|
| **并发安全** | Portfolio `threading.Lock`，SqliteSaver 单例 |
| **资源管理** | `atexit` 注册清理函数，SQLite 连接不泄露 |
| **上下文管理** | 超阈值自动摘要压缩，token 不爆炸 |
| **容错** | 工具返回 `{"error": "..."}` 不崩溃，Agent 标 N/A 继续 |
| **SSL** | 模块级 `_SSL_CTX` 集中管理，附原因注释 |
| **重试**（计划）| `tenacity` 指数退避 + 最多 3 次 |

---

### 13. SSE 流式输出时用户关闭浏览器怎么处理？

```
1. 服务端检测到连接断开（SSE onclose 事件）
2. 已生成的内容保留在服务器端（LangGraph 的 checkpointer 已自动存档每个节点输出）
3. 用户下次打开 → 根据 thread_id 恢复，从断点继续或显示已生成部分
4. 流式输出的中间状态不是"脏数据"——checkpointer 存的是确定性的节点输出
```

### 14. 怎么提升模型响应速度？

| 优化方向 | 手段 |
|---------|------|
| **模型层** | 换小模型（DeepSeek-v2→v1）；speculative decoding |
| **Prompt 层** | 精简 System Prompt；cache 重复前缀 |
| **工具层** | 并行调多个独立工具；工具结果预缓存 |
| **架构层** | 非关键路径异步化；Analyst 节点不用 ReAct（单次 `invoke`） |
| **网络层** | 数据接口加 Redis 缓存（TTL=5min）；CDN 加速静态资源 |

### 15. 一个比较完整的 Prompt 包含哪些部分？

```
1. Role（角色）       → "你是 FinBrain 高级分析师"
2. Context（上下文）    → 注入的检索结果 / 对话历史 / 用户画像
3. Rules（规则）       → "严禁编造数字""缺失数据标 N/A"
4. Steps（步骤）       → "1.查数据 2.评分 3.出结论"
5. Examples（示例）     → Few-shot 示例
6. Output Format（输出）→ JSON Schema / 表格格式
7. Constraints（约束）  → "不使用emoji""结论置于末尾"
8. Guardrails（护栏）  → "以上不构成投资建议"
```

**一个好的 Prompt 不是越长越好**——Rule 和 Format 要精确，背景信息要精简。

### 16. Token 和字符有什么区别？

```
"你好"     → 2 个字符，中文约 2-3 个 token
"hello"    → 5 个字符，英文约 1-2 个 token
"一个完整的句子" → 7 个字符，可能是 10-15 个 token
```

**Token 是 LLM 的计费单位，不是字符数。**

粗略换算（经验值）：
- 英文：1 token ≈ 0.75 个单词 ≈ 4 个字符
- 中文：1 token ≈ 0.5 个汉字 ≈ 1.5 个字符
- 代码：1 token ≈ 0.5-1 个字符（与语言相关）

**工程实践**：
- 估算 token 用 `tiktoken` 库，不要手动除
- DeepSeek 价格按 token 计费：输入 1元/百万token，输出 2元/百万token
- 压缩历史消息的首要目标就是省 token

## 五、场景设计与开放题

### 17. 用 Agent 实现 AI 面试程序怎么设计？

```
Candidate   → [Question Agent] → 出题（根据简历/岗位动态生成）
            → [Evaluator Agent] → 评估回答（正确性/深度/表达）
            → [Follow-up Agent] → 追问（递进式深挖）
            → [Scoring Agent] → 综合评分 → 生成面试报告
```

关键技术点：
- **状态管理**：SqliteSaver 记录每轮问答
- **防作弊**：摄像头监控 + 切屏检测（非 Agent 范畴）
- **自适应出题**：根据前面表现调整难度
- **多模态**：如果含代码题，候选人写代码 → 自动运行测试用例

---

### 18. 这个产品的缺点？

1. **单用户设计** — 全局单例 Agent、`st.session_state` 无法多租户
2. **免费数据源不稳定** — push2 间歇性断开，THS 鉴权依赖第三方库内部文件
3. **无自动化测试** — 工具和 Agent 都靠手动跑
4. **输出格式依赖 LLM** — DeepSeek 偶尔不按 JSON Schema 输出
5. **性能** — 深度分析 20-40 秒，对交互式体验偏慢

---

### 19. Agent 后续怎么发展？哪些场景容易落地？

**容易落地的场景**：
1. **客服** — 规则明确，答案库固定，幻觉容忍度高
2. **代码审查** — 输入输出结构化，易于评估
3. **数据查询** — Text-to-SQL + RAG，已有成熟方案
4. **合规审查** — 对照检查清单逐条验证（类似我们的 Checklist 评分模式）

**不容易落地的场景**：
- 完全自主交易（合规风险）
- 医疗诊断（幻觉代价不可接受）
- 法律判决（需要人类承担责任）

**趋势判断**：
- Agent 从"单步工具调用"走向"多步规划+执行+反思"
- 模型能力提升会让 Agent 越来越可靠，但永远需要护栏（Guardrails）
- MCP 协议会标准化，类似 HTTP 对于 Web 的意义
