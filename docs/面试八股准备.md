## 零、设计哲学（面试必答）

**Q: 为什么不做实时PE/PB/目标价的精确计算？**

> 这是刻意为之的设计决策。FinBrain 定位为基本面研究 Agent，不是实时行情终端。免费 API 拿不到 Level-2 数据，硬追精确 PE 会在最不该犯错的地方犯错——之前出现过把去年净利润当今年一致预期、PE 差 30% 的情况。所以所有 PE/PB/市值标注"基于财报，非实时行情"，砍掉了精确买入价、期望收益、仓位百分比。Agent 的算力应该花在它能做好的事情上：三年毛利率趋势、现金流为何与利润背离、护城河能否持续。

**Q: 怎么防止 LLM 乱改数据？**

> 分层防御。第一层：工具权限隔离，Analyst 无 tools。第二层：Reporter 后处理强制覆盖——`_fix_and_decide` 对每只股票调 `calculate_scores`，覆盖盈利能力/成长性/财务健康/估值合理四个维度 + 估值水位 PE/PB/市值。第三层：关键信号代码检查，不依赖 LLM"注意"——Q1 经营现金流为负自动注入风险警告。第四层：证伪条件——任何投资逻辑都有可验证的失败条件。

**Q: 你这个系统怎么部署？能支持多少用户？**

> 两种模式。本地模式：用户 `pip install` 后直接 `streamlit run`，全部在自己电脑上跑，天然单用户。远程模式：服务器跑 FastAPI + Redis + 定时预取调度器，客户端只做 LLM 推理。用户的 API Key 只在本地 `.env`，永不传到服务器。数据 API 是纯读缓存（毫秒级），配合 FastAPI 的异步并发 + Nginx 反向代理，轻松支撑数千并发。三层缓存（scheduler 预取 → Redis → Nginx proxy_cache）保证冷数据请求也不排队。

**Q: 你的 Agent 能做什么、不能做什么？**

> 能做：读懂财报（营收/利润/现金流趋势）、判断商业模式（护城河来源+复制难度）、行业周期定位、识别财务预警信号。不能做：精确预测股价、给出精确买卖点、替代交易终端。FinBrain 是行业研究员，不是交易员。

## 一、Agent 架构与工作模式

### 1. 你怎么理解 Agent？它和普通 LLM 应用最大的区别是什么？

**核心区别：Agent 能自主决策行动序列，普通 LLM 应用只能做单次推理。**

三个关键能力：

| 能力 | 说明 | FinBrain 实例 |
|------|------|-------------|
| **工具使用** | 调用外部函数/API 获取真实数据 | 13 个 `@tool`，LLM 自主决定调用哪个 |
| **记忆** | 跨轮次保留上下文 | SqliteSaver 持久化对话状态 |
| **规划** | 分解复杂任务为多步执行 | DataCollector→Analyst→Reporter 三节点流水线 |

### 2. Agent 常见的工作模式有哪些？

| 模式 | 核心思想 | 项目落地 |
|------|---------|---------|
| **ReAct** | Thought→Action→Observation 循环 | DataCollector 子 Agent 用 `create_agent` |
| **Plan-and-Execute** | 先生成计划再逐步执行 | 三节点流水线 |
| **Router** | 根据输入分类路由 | `_classify_request()` + 显式模式切换 |
| **Multi-Agent** | 多个专业化 Agent 协作 | DataCollector + Analyst + Reporter + Phantom |
| **Post-Processing** | 执行后代码修正 | Reporter 强制覆盖评分 + 计算投资评级 |

### 3. 项目里有没有用到现成框架？

**LangGraph + LangChain**。

架构分层：
```
编排层: LangGraph StateGraph (手写节点和边)
  └── 子Agent: LangChain create_agent (ReAct循环)
        └── Tool层: LangChain @tool 装饰器
              └── 数据层: tools.py (纯Python，零框架依赖)
              └── 决策层: scoring.py (纯Python，零LLM依赖)
```

选择 LangGraph 而非 AutoGen/CrewAI：我的 Data→Analyze→Report 流程固定，不需要 Agent 间"协商"。StateGraph 编译时确定图结构，执行路径可预测。Analyst 和 Reporter 无 tools，从框架层面保证权限隔离。

### 4. 投资评级是 LLM 算的还是代码算的？

**代码算的**。LLM 只写叙事（亮点/风险/逻辑链）。`scoring.py` 的 `compute_investment_rating()` 按公司类型动态赋权 + 算安全边际 + 出 BUY/HOLD/SELL。Reporter 的 `_fix_and_decide` 强制覆盖 LLM 可能乱改的分数和评级。同一只股票跑 100 次得分完全一致。

### 5. 怎么避免路径震荡（重复规划）？

1. **StateGraph 固定流水线** — 节点间不形成循环
2. **DataCollector 只跑一轮** — Prompt 明确"调完所有工具后汇总"
3. **max_tokens 限制** — 天然限制循环深度
4. **评分后处理覆盖** — 代码最终说了算

## 二、工具调用与 MCP

### 6. 13 个工具怎么管理？

`tools.py` 纯 Python 函数（零 Agent 框架依赖） → `agent.py` 的 `@tool` 装饰器薄包装。不同 Agent 注册不同工具集实现权限隔离。工具可独立测试、可同时给 LangGraph Agent、MCP Server 和 Streamlit 前端使用。

### 7. 工具调用 vs Function Calling vs 普通 Prompt

| 维度 | 普通 Prompt | Function Calling | Tool Use (LangChain) |
|------|-----------|-----------------|---------------------|
| LLM 输出 | 自由文本 | JSON `{name, arguments}` | JSON `{name, arguments}` |
| 谁执行 | 无 | 调用方代码 | 框架自动拦截+执行+回传 |
| 多轮 | 不支持 | 需手动循环 | 框架内置 ReAct |

### 8. 你怎么理解 MCP / Prompt / Harness？

| 概念 | FinBrain 实例 |
|------|-------------|
| **MCP** | `MCP_server.py`暴露 5 个 Tool 给 Claude Code |
| **Prompt** | `ANALYST_PROMPT`、`_FORMAT_MANDATORY`、策略热加载 |
| **Harness** | 工具分级、SqliteSaver、压缩、路由、降级、评分确定性 |

## 三、RAG 与知识库

### 9. 知识库是怎么搭的？

```
文档上传(PDF/DOCX/TXT) → 格式识别 → 解析器分发
  → 三级切片(段落→句子→字符, 500字/块, 50字重叠)
  → ONNX MiniLM 向量化 → ChromaDB 持久化
```

4 个知识库：会计准则(15条预置+用户上传)、行业研报、交易策略、游资风格。

### 10. Lost in the Middle 怎么处理？

1. 切片时加元数据标注（文档标题/章节/页码）
2. 分层检索：先搜粗粒度再搜细粒度
3. Prompt 显式提示优先使用靠前内容
4. 过滤过短的块（<20字符）

### 11. 怎么控制模型幻觉？

| 层 | 手段 |
|------|------|
| **RAG 层** | 每个 chunk 带 source 标注 |
| **Prompt 层** | "缺失数据标 N/A""依据栏引用具体数值" |
| **输出约束** | 固定 JSON Schema，缺失字段标 null |
| **工具层** | Analyst 无 tools，只能分析已有数据 |
| **代码层** | Reporter 强制覆盖客观维度分数 |
| **逻辑层** | 证伪条件——任何投资逻辑必须有可验证的失败条件 |

## 四、系统工程与性能

### 12. 针对可靠性与并发，做了哪些优化？

| 优化 | 方案 |
|------|------|
| **并发安全** | Portfolio `threading.Lock`，SqliteSaver 单例 |
| **资源管理** | `atexit` 注册清理，SQLite 连接不泄露 |
| **上下文管理** | 超阈值自动摘要压缩 |
| **容错** | 工具返回 `{"error": "..."}` 不崩溃，Agent 标 N/A 继续 |
| **评分确定性** | 代码后处理强制覆盖 LLM 输出 |
| **懒初始化** | 重资源 double-check locking，不在 import 时加载 |

### 13. 怎么提升模型响应速度？

| 方向 | 手段 |
|------|------|
| **架构层** | Analyst 节点不用 ReAct（单次 `invoke`） |
| **工具层** | TTL 分层缓存（30s股价/30min财报/1h行业） |
| **Prompt 层** | 精简 System Prompt，策略热加载按需注入 |
| **网络层** | 数据接口加缓存，避免重复 API 调用 |

### 14. 一个比较完整的 Prompt 包含哪些部分？

```
1. Role（角色）         → "你是 FinBrain 高级投资分析师"
2. Framework（框架）     → 十步分析链
3. Rules（规则）         → "纯JSON""多股票数组"
4. Context（上下文）      → 行业分析模板
5. Format（输出格式）     → JSON Schema
6. Quality（质量标准）    → 归母≠扣非/现金流≠含金量/证伪条件
7. Examples（示例）       → 正确vs错误的数组格式
8. Guardrails（护栏）    → "不构成投资建议"
```

## 五、投资决策引擎

### 15. 动态权重是怎么设计的？

5 种公司类型 × 6 维权重矩阵。价值型重估值(30%)+财务(25%)，成长型重成长(30%)+行业(20%)+壁垒(20%)，周期型重行业周期(30%)+估值(25%)。

安全边际 = 类型基准(20-50%) + ROE质量调整 + 负债率调整。

合理价值 = 行业PE基准 × ROE质量系数(6档, 0.3x~1.6x) × EPS。

买入区间 = 合理价值 × (1 - 安全边际)。全部代码计算，不依赖 LLM。

### 16. 十步分析链 vs 传统财务分析

传统工具只回答"公司财务健康吗"。FinBrain 回答"是否值得投资"：

公司画像 → 商业模式 → 护城河 → 行业周期 → 财务验证 → 成长逻辑 → 市场预期 → 情景估值 → 催化剂 → 投资决策

关键洞察：增加"市场预期拆解"模块——分析当前估值隐含的增长率、市场主要担忧、以及可能的预期差。股票投资的核心是找到市场预期与实际价值之间的差距。

## 六、场景设计与开放题

### 17. 用 Agent 实现 AI 面试程序怎么设计？

```
Candidate → [Question Agent] 出题 → [Evaluator Agent] 评估
  → [Follow-up Agent] 追问 → [Scoring Agent] 综合评分
```

### 18. 这个产品的缺点？

1. **单用户设计** — 全局单例 Agent，`st.session_state` 无法多租户
2. **免费数据源不稳定** — push2 间歇性断开，THS 鉴权依赖第三方库
3. **无自动化测试** — 工具和 Agent 都靠手动跑
4. **行业分类依赖 LLM** — 未做公司分类的代码化
5. **DCF 缺失** — 估值只有 PE 法，无现金流折现

### 19. Agent 后续怎么发展？

**容易落地**：客服（规则明确）、代码审查（结构化输入输出）、数据查询（Text-to-SQL+RAG）、合规审查（对照检查清单）。

**趋势**：Agent 从"单步工具调用"走向"多步规划+执行+反思"。模型能力提升会让 Agent 越来越可靠，但永远需要护栏。MCP 协议会标准化。
