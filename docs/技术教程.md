## 1. LangGraph StateGraph

### 是什么

LangGraph 是 LangChain 生态里的**有状态工作流编排框架**。StateGraph 让你显式定义节点（Node）和边（Edge），数据通过一个 State 对象在节点间流转。

和普通函数调用链的区别：StateGraph 的每个节点执行后，框架自动把 State 写入 Checkpointer（如 SQLite），支持暂停、恢复、回溯。

### 为什么用它

```
方案A: 一个巨大的 prompt，LLM 一次性输出 → 不可控，容易漏数据
方案B: 自己写 if/else 调用链 → 能跑，但没有持久化、没有可视化
方案C: AutoGen/CrewAI 多 Agent 对话 → 太重，我们的流程是固定的
方案D: LangGraph StateGraph → 固定流程 + 自动持久化 + 节点级工具隔离
```

FinBrain 的分析流程是确定性的 8 节点流水线：Data → Classify → Analyst → Valuation → Critics(三路并行) → Repair → Reporter。不需要 Agent 间"协商"，只需要显式编排。StateGraph 编译时确定图结构，执行路径可预测。

### 怎么做

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

# 1. 定义 State —— 节点间传递的数据
class FinBrainState(TypedDict):
    messages: Annotated[list, add_messages]  # 对话历史，自动追加
    user_question: str
    collected_data: str    # DataCollector 产出
    analysis: str          # Analyst 产出
    report: str            # Reporter 产出

# 2. 定义三个节点
def data_collector_node(state: FinBrainState) -> dict:
    # 调工具搜集数据
    return {"collected_data": "..."}

def analyst_node(state: FinBrainState) -> dict:
    # LLM 分析
    return {"analysis": "..."}

def reporter_node(state: FinBrainState) -> dict:
    # 格式化输出
    return {"report": "..."}

# 3. 组装图（8节点流水线）
graph = StateGraph(FinBrainState)
graph.add_node("data_collector", data_collector_node)
graph.add_node("classifier", company_classifier_node)
graph.add_node("analyst", analyst_node)
graph.add_node("valuation", valuation_agent_node)
graph.add_node("critics", critics_node)      # 三路并行
graph.add_node("repair", repair_node)
graph.add_node("reporter", reporter_node)
graph.add_edge(START, "data_collector")
graph.add_edge("data_collector", "classifier")
graph.add_edge("classifier", "analyst")
graph.add_edge("analyst", "valuation")
graph.add_edge("valuation", "critics")
graph.add_edge("critics", "repair")
graph.add_edge("repair", "reporter")
graph.add_edge("reporter", END)

# 4. 编译并执行
app = graph.compile()
result = app.invoke({"user_question": "分析贵州茅台", "messages": []})
print(result["report"])
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

StateGraph 的核心设计哲学是 **"显式优于隐式"**。每个节点写了什么、读了什么，在 State 定义里一目了然。节点间的边是编译时确定的，不会有运行时"Agent A 突然决定不找 Agent B 了"的情况。这让系统可审计、可调试。

工具权限隔离是 FinBrain 最看重的设计——Analyst 和 Reporter 节点**根本没有 tools 参数**。这不是忘了加，是刻意为之。Analyst 只能分析 DataCollector 拿到的真实数据，从框架层面防止幻觉。

## 2. LangChain @tool

### 是什么

`@tool` 是 LangChain 提供的装饰器。把一个普通 Python 函数装饰后，LangChain 自动完成：
- 函数签名 → JSON Schema（LLM 需要知道参数格式才能调用）
- docstring → 工具描述（LLM 据此判断"这个工具能做什么"）
- 返回值 → 自动序列化

### 为什么用它

没有 `@tool` 时，你需要手工写 JSON Schema 告诉 LLM 每个工具的参数格式。`@tool` 帮你自动完成这一步。

更重要的是，`@tool` 是**薄包装**——底层函数（`tools.py`）完全不依赖 LangChain。这意味着工具可以独立测试、独立部署，也可以给 MCP Server 和 Streamlit 前端复用。

### 怎么做

```python
from langchain_core.tools import tool
import json

# tools.py 里的纯函数（零框架依赖）
def fetch_stock_price(symbol: str) -> dict:
    url = f"https://hq.sinajs.cn/list=sh{symbol}"
    # ... HTTP 请求 ...
    return {"name": "贵州茅台", "price": "1800.00"}

# agent.py 里的 @tool 包装（薄薄一层）
@tool
def stock_price(symbol: str) -> str:
    """查询A股实时价格。输入股票代码如 '601991'"""
    from backend.tools import fetch_stock_price
    return json.dumps(fetch_stock_price(symbol), ensure_ascii=False)
```

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

`docstring` 是 LLM 了解工具的唯一途径。写 docstring 时要换位思考——如果你是 LLM，看到这个描述，你会知道什么时候该调这个工具吗？好描述的标准：**给一个具体例子**（如 `'601991'`），而不是抽象描述（如 `'股票代码'`）。

## 3. LangChain Chat Models

### 是什么

LangChain 对各大 LLM 提供商（OpenAI、Anthropic、DeepSeek）做了统一封装。你只需要换一个类名和参数，就能切换模型。

### 为什么用它

FinBrain 的用户可能用不同的 LLM。有人用 DeepSeek（便宜），有人用 Claude（质量高），有人用 GPT-4o。统一封装意味着**改一行环境变量就能切换**，不需要改任何业务代码。

### 怎么做

```python
import os
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

def _get_llm():
    provider = os.getenv("LLM_PROVIDER", "deepseek")
    if provider == "deepseek":
        return ChatOpenAI(
            model="deepseek-chat", temperature=0, max_tokens=4096,
            api_key=os.getenv("DEEPSEEK_API_KEY"),
            base_url="https://api.deepseek.com",
        )
    elif provider == "openai":
        return ChatOpenAI(model="gpt-4o", temperature=0, max_tokens=4096)
    else:
        return ChatAnthropic(model="claude-sonnet-5", temperature=0, max_tokens=4096)
```

`temperature=0` 是刻意设置的——财务分析需要一致性和可复现性，不需要创造性。同样的问题问两次应该得到同样的结论。

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

DeepSeek 通过 OpenAI 兼容接口访问（`ChatOpenAI` + `base_url`），这是一个常见的模式。很多国产模型（DeepSeek、Moonshot、Qwen）都兼容 OpenAI API 格式，不需要单独的 SDK。

## 4. ChromaDB + ONNX

### 是什么

ChromaDB 是一个**嵌入式向量数据库**——不需要单独部署服务器，数据存在本地文件里，`pip install chromadb` 就能用。ONNX MiniLM 是一个 80MB 的轻量嵌入模型，能把文本转成 384 维向量。

### 为什么用它

对比方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| Pinecone/Weaviate 云服务 | 功能全 | 要钱、要网络、数据在外网 |
| FAISS（Meta） | 极快 | 无持久化、无元数据管理 |
| Elasticsearch | 全文+向量双检 | 太重，要 Java 运行环境 |
| **ChromaDB** | 本地运行、持久化、自带嵌入模型 | 功能较新，社区在成长 |

FinBrain 选 ChromaDB 因为它**零运维**——`pip install` 后直接能用，数据存本地，不需要 Docker。ONNX MiniLM 80MB 不大不小，中文效果尚可。

### 怎么做

```python
import chromadb
import os

# 初始化（一次）
DB_DIR = os.path.join(os.path.dirname(__file__), "..", "data", "raw", "chroma")
os.makedirs(DB_DIR, exist_ok=True)
client = chromadb.PersistentClient(path=DB_DIR)

# 嵌入模型（ONNX，首次自动下载 80MB）
embed_fn = chromadb.utils.embedding_functions.ONNXMiniLM_L6_V2()

# 创建/获取集合
collection = client.get_or_create_collection(
    name="accounting_kb",
    embedding_function=embed_fn,
)

# 写入
collection.add(
    documents=["资产减值测试是指..."],
    ids=["doc_1"],
    metadatas=[{"source": "会计准则第8号", "page": 45}],
)

# 检索
results = collection.query(query_texts=["商誉减值测试条件"], n_results=3)
for i, doc in enumerate(results["documents"][0]):
    print(f"[{results['distances'][0][i]:.3f}] {doc[:100]}")
```

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

ChromaDB + ONNX 的瓶颈在**嵌入模型的精度**。ONNX MiniLM 是英文预训练的，中文语义理解不如 `bge-large-zh` 等中文专用模型。如果检索质量不满意，可以换嵌入模型——ChromaDB 支持自定义 `embedding_function`。

## 5. FastAPI

### 是什么

FastAPI 是 Python 最快的 Web 框架之一，基于 Starlette + Pydantic。写一个 API 端点只需要一个装饰器。

### 为什么用它

对比 Flask/Django：FastAPI 原生异步（`async def`），自带 Swagger 文档（`/docs`），自动请求校验（Pydantic）。FinBrain 需要的是**纯数据服务**——没有 HTML 模板、没有 session、没有 ORM。FastAPI 最轻量。

### 怎么做

```python
from fastapi import FastAPI, Query
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI(title="FinBrain Data API", version="2.0")
app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"], allow_headers=["*"])

@app.get("/api/data/stock_price")
def api_stock_price(symbol: str = Query(...)):
    from backend.tools import fetch_stock_price
    return fetch_stock_price(symbol)

@app.get("/health")
def health():
    return {"status": "ok"}

# 启动: uvicorn backend.api:app --host 0.0.0.0 --port 8000 --workers 4
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

`--workers 4` 是 Gunicorn/Uvicorn 的多进程模式，每个 worker 独立处理请求。对于 FinBrain 这种 I/O 密集型（大部分时间在等 Redis 返回），4 个 worker 能轻松支撑数百并发。CPU 密集型才需要更多 worker。

浏览器打开 `http://localhost:8000/docs` 能看到所有端点的 Swagger 交互文档——这是 FastAPI 最实用的特性，调试时不用 curl。

## 6. Streamlit + Plotly

### 是什么

Streamlit 是**纯 Python 的数据应用框架**——不需要写 HTML/CSS/JS，用 Python 表达式直接渲染页面。Plotly 是交互式图表库。

### 为什么用它

FinBrain 的定位是工具型应用，不是内容型网站。Streamlit 最适合这个场景：快速原型、数据展示、交互式控件。对比 Dash/Shiny：Streamlit 不需要回调函数，代码是线性执行的。

### 怎么做

Streamlit 的最大坑是**每次交互都全量重跑**——点一个按钮、输一个字符，整个脚本从头执行一遍。`@st.cache_data` 是救命稻草：标记一个函数的返回值可以复用，TTL 内不重算。FinBrain 的 Market 页面如果不加缓存，每次切页面都重新爬 API，三秒卡一次。

## 7. Redis 缓存

### 是什么

Redis 是内存键值数据库，读写延迟 <1ms。FinBrain 用它做**多进程共享缓存**。

### 为什么用它

内存 dict 的问题是：4 个 Uvicorn worker 各有各的内存，worker A 缓存了股票数据，worker B 还要再爬一遍。Redis 是所有 worker 共享的——谁存进去，大家都读得到。

`SET key value EX ttl` 是 Redis 最常用的命令——存一个值，到时间自动删除。不需要手动清理过期数据。

### 怎么做

```python
import redis, json, os

r = redis.from_url(os.getenv("REDIS_URL", "redis://localhost:6379"))

# 写缓存（带 TTL）
r.setex("finbrain:stock_price:601991", 30, json.dumps({"price": 5.78}))

# 读缓存
raw = r.get("finbrain:stock_price:601991")
if raw:
    data = json.loads(raw)

# 生产环境用连接池（redis-py 默认自带）
# r = redis.Redis(connection_pool=pool)
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

Redis 是 FinBrain 远程模式的关键——没有它，多 worker API 要么重复爬虫（慢），要么共享内存（不安全）。但 Redis 不是银弹：单机 Redis 有内存上限（要配置 `maxmemory`），高并发需要主从/集群。FinBrain 的起步阶段一台 Redis 足够。

## 8. Threading

### 是什么

Python 的 `threading` 模块提供线程级并发。注意：Python 有 GIL（全局解释器锁），CPU 密集型任务多线程不会加速。但 I/O 密集型（网络请求、文件读写）多线程有效——等网络时 GIL 会释放。

### 为什么用它

FinBrain 两处用了线程：
1. **并行数据预取**：`ThreadPoolExecutor` 同时拉 4 只股票的数据，节省 10-15 秒
2. **线程安全**：`threading.Lock` 保护缓存和持仓文件的并发写入

### 怎么做

```python
from concurrent.futures import ThreadPoolExecutor
import threading

# 并行拉取——等网络时切换线程，总时间≈最慢的一个请求
def fetch_all(symbols):
    def fetch_one(code):
        return code, get_financial_statements(code)
    with ThreadPoolExecutor(max_workers=4) as ex:
        return dict(ex.map(fetch_one, symbols))

# 线程安全写入
_lock = threading.Lock()
_cache = {}

def set_safe(key, value):
    with _lock:
        _cache[key] = value
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

`max_workers=4` 不是随便设的。免费 API（新浪/东财）对并发有隐性限制——同时 10 个请求可能被封 IP。4 个是一个安全的平衡点。如果上付费 API，可以开到 10-20。

## 9. urllib + SSL

### 是什么

`urllib` 是 Python 标准库的 HTTP 客户端——不需要 `pip install`，Python 自带。配合 `ssl` 模块处理 HTTPS。

### 为什么用它

FinBrain 的工具层（`tools.py`）**零第三方依赖**——不用 `requests`，不用 `httpx`，只用标准库。原因：`tools.py` 既要被 LangGraph 调，也要被 MCP Server 调，也要被命令行脚本调。减少依赖就是减少部署麻烦。

代价：`urllib` 的 API 比 `requests` 繁琐。`requests.get(url)` 是一行，`urllib` 需要 5 行。

### 怎么做

```python
import urllib.request, ssl, json

# SSL 证书绕过——免费 API 的证书经常过期/自签
ctx = ssl.create_default_context()
ctx.check_hostname = False
ctx.verify_mode = ssl.CERT_NONE

url = "https://hq.sinajs.cn/list=sh601991"
req = urllib.request.Request(url)
req.add_header("User-Agent", "Mozilla/5.0")
req.add_header("Referer", "https://finance.sina.com.cn")

with urllib.request.urlopen(req, timeout=10, context=ctx) as resp:
    text = resp.read().decode("gbk")  # 新浪用的是 GBK！
    data = json.loads(text)
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

`ssl.CERT_NONE` 是**安全妥协**——生产环境不应该跳过证书验证。但免费金融 API 的证书质量参差不齐，不跳过就没法用。FinBrain 的策略：集中管理（整个项目只有一个 `_SSL_CTX`），加注释说明为什么这么做。如果接入付费 API，可以去掉。

## 10. JSON 解析

### 是什么

Python 的 `json` 模块处理 JSON。`json.loads()` 把字符串转成 Python 对象。`json.JSONDecoder().raw_decode()` 从一段文本中提取第一个有效的 JSON 值。

### 为什么用 raw_decode

LLM 输出的 JSON 经常不干净——前面有"以下是分析结果："，后面有"以上不构成投资建议"。`json.loads()` 会直接报错。`raw_decode()` 从第一个 `{` 或 `[` 开始解析，自动跳过前后的废话。

更重要的：**raw_decode 能正确处理嵌套 JSON**。正则 `\[[\s\S]*\]` 在遇到嵌套数组（如 `"股票数据":[{大唐},{国电}]`）时会匹配到内部的 `[...]` 而不是外层的 `[...]`。raw_decode 天生知道括号配对，不会犯这个错。

### 怎么做

```python
import json

# 方案A: 直接解析（要求纯JSON）
data = json.loads(text)

# 方案B: 正则提取 ```json ... ``` 代码块
import re
m = re.search(r'```(?:json)?\s*([\s\S]*?)```', text)
if m:
    data = json.loads(m.group(1))

# 方案C: raw_decode——从文本中定位第一个 JSON 值
decoder = json.JSONDecoder()
for i, ch in enumerate(text):
    if ch in '[{':
        data, end_index = decoder.raw_decode(text[i:])
        break
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

raw_decode 是处理 LLM 输出的瑞士军刀。它不依赖正则、不要求纯 JSON、自动处理嵌套。唯一要注意：raw_decode 只返回第一个 JSON 值——如果 LLM 输出了一段文字后又写了一个 JSON，第二个会被忽略。

## 11. SqliteSaver

### 是什么

LangGraph 的 SQLite 持久化后端。每个节点执行后，State 自动写入 SQLite 文件。下次相同的 `thread_id` 调用时，自动恢复到中断点。

### 为什么用它

```
手写 JSON: 每次都要自己序列化/反序列化，容易丢字段
LangChain Memory: 只存摘要，恢复粒度不够（工具调用链丢了）
SqliteSaver: 框架自动保存完整 State，恢复时原样还原
```

### 怎么做

```python
from langgraph.checkpoint.sqlite import SqliteSaver
import atexit

# 初始化（用 Context Manager 管理生命周期）
cm = SqliteSaver.from_conn_string("data/raw/checkpoints.db")
checkpointer = cm.__enter__()
atexit.register(lambda: cm.__exit__(None, None, None))  # 进程退出时自动关闭

# 每次调用带 thread_id
config = {"configurable": {"thread_id": "user_session_123"}}
result = agent.invoke({"messages": [...]}, config=config)

# 下次用同样的 thread_id，自动从上次中断点继续
```

### 思考

`atexit.register` 是 Python 的资源管理兜底——如果忘记 `cm.__exit__()`，进程退出时也会自动关闭连接。这是防御性编程：不该依赖它，但不能没有它。

## 12. RAG

### 是什么

Retrieval Augmented Generation（检索增强生成）。在 LLM 推理前，先从知识库检索相关内容，注入 Prompt。LLM 的"短期记忆"来自检索结果，不需要把知识全写在 Prompt 里。

### 为什么用它

LLM 的训练数据有截止日期，不知道最新的会计准则变化。而且 Prompt 长度有限（DeepSeek 64K token），不可能把整个会计准则塞进去。RAG 解决"按需加载"——用户问"商誉减值怎么测"，就只检索商誉减值相关的内容。

### 怎么做

```python
# 1. 文档→切片
def chunk_text(text, chunk_size=500, overlap=50):
    paragraphs = text.split("\n\n")
    chunks = []
    current = ""
    for para in paragraphs:
        if len(current) + len(para) <= chunk_size:
            current += para
        else:
            chunks.append(current)
            current = para
    return chunks

# 2. 切片→向量→存储
chunks = chunk_text(accounting_standards_pdf)
for i, chunk in enumerate(chunks):
    collection.add(documents=[chunk], ids=[f"chunk_{i}"])

# 3. 检索→注入 Prompt
query = "商誉减值测试方法"
results = collection.query(query_texts=[query], n_results=3)
context = "\n".join(results["documents"][0])
prompt = f"参考以下会计准则:\n{context}\n\n问题: {query}"
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

切片大小是 RAG 最需要调参的地方。500 字适合会计准则这种条文式文档。太小（100 字）丢掉上下文，太大（2000 字）检索精度下降。overlap 保证块边界处不断裂。20 字的块直接过滤掉——没有检索价值。

## 13. TTL 缓存模式

### 是什么

Time-To-Live 缓存：每个缓存条目有一个过期时间，过期自动失效。FinBrain 的 TTL 按数据类型分层：

```python
TTL = {
    "stock_price": 30,          # 股价变化快，30秒
    "financial_statements": 1800,  # 财报日级更新，30分钟
    "industry_info": 3600,      # 行业分类基本不变，1小时
}
```

### 为什么分层

不是所有数据都该用同样的缓存时间。股价每秒都在变，财报一个季度才变一次。统一 TTL 的话——设短了财报频繁重爬浪费资源，设长了股价变成"昨日行情"。分层是缓存设计的基线。

### 怎么做

```python
import time

_cache = {}

def get(key):
    entry = _cache.get(key)
    if entry and time.time() - entry["ts"] < entry["ttl"]:
        return entry["data"]
    return None

def set(key, data, ttl=60):
    _cache[key] = {"ts": time.time(), "ttl": ttl, "data": data}
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

TTL 的取值是一场博弈：长了数据不准，短了 API 压力大。FinBrain 的策略是**按数据变化频率定价**——股价 30s 是底线（新浪免费 API 本身有 3-5 秒延迟），财报 30min 是因为东方财富 API 一天只更新一次，再多也无意义。

## 14. Prompt 工程

### 是什么

Prompt 是你和 LLM 之间的"接口协议"。代码和 LLM 的边界就在 Prompt 上——代码把数据整理好，Prompt 告诉 LLM 怎么处理这些数据。

### 为什么需要结构化 Prompt

FinBrain 的 ANALYST_PROMPT 包含：
1. **角色定义**："你是 FinBrain 高级投资分析师"
2. **框架指引**：十步分析链
3. **规则约束**："纯JSON输出""多股票必须输出数组"
4. **质量标准**："归母≠扣非""趋势看三年不只看一季""证伪条件要具体"
5. **输出格式**：完整的 JSON Schema

### 怎么做

```python
ANALYST_PROMPT = """[铁律] 纯JSON输出，多股票必须输出数组。

[十步分析框架]
第1步 公司分类 → 第2步 商业模式 → ... → 第10步 投资决策

[质量标准]
1. 归母≠扣非：用扣非净利润判断主业增长
2. 现金流≠含金量：重资产行业折旧推高OFC/NI
...

[输出格式]
{"代码":"xxx", "评分":{...}, "亮点":[...], ...}
"""
```

### 审计退回与Analyst重推理

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

好 Prompt 的精髓是"约束而非建议"。"你应该..."和"你必须..."对 LLM 的效果完全不同。"禁止"比"不建议"有效十倍。FinBrain 的经验：铁律放最前面、用具体错误示例、把格式示例和反例并列。LLM 对"不要这样做"比对"要这样做"更敏感。

## 15. 服务器-客户端分离架构

### 是什么

把系统拆成两部分：
- **服务器**：定时爬数据 → 缓存 → 暴露 HTTP API。不调 LLM。
- **客户端**：调 API 拿数据 → 填 Prompt → 用户本地调 LLM。用户的 API Key 不上传。

### 为什么这样设计

```
全在服务端: 用户的 Key 要上传 + LLM Token 费用由服务器承担 → 不可持续
全在客户端: 每个用户自己爬数据 → 重复爬、IP 被封 → 体验差
分离架构: 服务器维护一套数据（所有人共享） + 客户端各自用自己的 Key（账单自付）
```

这是典型的"云数据底座 + 本地智能终端"模式。服务器只做最擅长的事（维护数据），客户端掌握最敏感的东西（API Key 和推理）。

### 怎么做

```
┌─────────────────────────┐
│  服务器                  │
│  scheduler.py → 定时预取 │
│  api.py → FastAPI 数据服务│
│  Redis → 共享缓存        │
└─────────────────────────┘
           │ HTTP
           ▼
┌─────────────────────────┐
│  客户端                  │
│  client.py → 调API+本地LLM│
│  Streamlit → 前端        │
│  .env → DEEPSEEK_API_KEY │
└─────────────────────────┘
```

环境变量控制：
```bash
# 本地模式
FINBRAIN_DATA_MODE=local

# 远程模式

```

### 思考

这个架构的核心原则是**谁的数据谁负责，谁的 Key 谁保管**。用户花自己的 Token 钱，服务器花自己的运维钱——边界清晰，没有隐形成本。这也是为什么没有做消息队列——每个请求是独立的，不需要异步解耦。

## 16. Harness 工程基座

### 是什么

Harness 是独立于业务逻辑的**防护层**。它不是"做什么"（那是 Agent 的事），而是"什么不能做"——不该调的别调、不该算的别算、错了要知道。

### 为什么需要

LLM 是不可靠的——它会编数字、会漏字段、会重复调同一个工具。不能指望 LLM "注意"到这些问题。Harness 的核心原则是**代码兜底**：关键检查由代码执行，不依赖 LLM。

### FinBrain 的 Harness 三层架构

**调用层**（防止浪费）：
```python
# 数据不足拦截——亏损公司 PE 没意义
if not val_data and not financial_data.get("profit"):
    return {"盈利能力": {"得分": None, "依据": "数据不足，无法计算"}}

# TTL 感知工具去重——30秒内不重复调股价
dup = _dedup_check(symbol, "stock_price")
if dup: return dup  # "已调用过，跳过"

# 注定失败拦截——同一工具连续3次失败→熔断
if _should_skip("fund_flow"):
    return {"error": "fund_flow 已连续失败3次，本轮跳过"}

# 交易时段判断
if now.weekday() >= 5:
    return {"info": "非交易日，无分时数据"}
```

**输出层**（保证正确）：
```python
# 格式校验——JSON 缺必填字段→重试一次
missing = ["代码", "名称", "评分"]  # 缺了这些
if missing:
    retry_response = llm.invoke([..., "请重新输出完整JSON"])

# 评分强制覆盖——LLM 写的分数不可信
for dim in ["盈利能力", "成长性", "财务健康", "估值合理"]:
    item["评分"][dim] = calculate_scores(data)[dim]

# 成长-估值匹配检查——营收+57%+PE 13倍还判"谨慎"？改掉
if revenue_growth > 30 and pe < 15:
    decision["评级"] = "BUY"
    item["偏见修正"] = "高增长低估值，强制上调"
```

**运维层**（保证可靠）：
```python
# LLM API 熔断——连续3次失败→停止，别无限重试
if _llm_failures >= 3:
    raise RuntimeError("LLM API 连续失败3次，已熔断")

# 结构化日志
logger.info("DataCollector: %d stocks in %.0fms, %d errors", n, elapsed, errors)

# 配置校验——scoring.json 缺 section→告警
missing = [s for s in required if s not in config]
if missing:
    logger.warning("scoring.json missing: %s", missing)
```

审计失败时，失败摘要不是简单的一句话——而是完整的问题列表(描述+修正建议)加上审计Agent的原始JSON输出，一起喂回给Analyst。这样Analyst能看到每条问题的完整上下文(类型、严重程度、修正建议)，覆盖任何类型的审计发现，不限于定增或成长性。



### 思考

Harness 的设计哲学和业务代码完全不同。业务代码追求"功能多"，Harness 追求"不出错"。Harness 越多，系统越稳——但 Harness 太多也会让系统变慢（每个检查都有开销）。FinBrain 的策略是：**昂贵的检查（LLM 重试）只做一次，便宜的检查（缓存命中）每次必做**。

## 16. 多 Agent 协作 — Critics 三路并行 + Repair 闭环

### 是什么
将单一的 Critic Agent 拆分为三个专业化审查员（Logic/Financial/Industry），并行运行后由 Aggregator 合并去重，输出结构化修复清单，Repair Agent 据此自动修正分析报告。形成"生成 → 质疑 → 修正 → 验证"的闭环。

### 为什么用
- **单 Critic 覆盖不全**：一个 prompt 同时检查逻辑、财务、行业，容易遗漏
- **财务需要代码预检**：CFO/CAPEX/FCF 是确定性计算，不应交给 LLM 口算
- **修复闭环**：发现问题只是第一步，自动修正才是完整 Agent

### 怎么实现
1. 三个 Critic Prompt：Logic（因果链/过度自信）、Financial（代码预检+LLM语义）、Industry（竞争描述/技术路线）
2. `critics_node` 用 `ThreadPoolExecutor(max_workers=3)` 并行调用三个 LLM
3. Financial Critic 先跑 `_financial_code_check()`（FCF=CFO-CAPEX、PE/ROE匹配、CFO折旧解释），代码发现注入 LLM prompt
4. Aggregator 合并三个结果，按类别去重，生成 `[CRITIC_FIXES]` JSON
5. `repair_node` 接收修复清单，调用 Repair Agent LLM 定向修改 JSON 中被标记字段
6. Pipeline：valuation → critics → repair → reporter

### 设计思考
- Critics 不重新分析股票，只做反向验证——角色隔离降低确认偏差
- Financial Critic 的代码层不像 Audit Engine 那样做 8 项检查，只做三项最容易被 LLM 误读的数值验证
- Repair Agent 只改被标记字段，不改无关内容——避免"修一个问题引入三个新问题"

## 17. 回测引擎 + 多账户模拟盘

### 是什么
从分析报告中提取交易信号（建仓价/止损/目标价/持仓周期），用历史 K 线逐日回放，模拟入场→持有→出场的全过程，计算胜率和收益率。模拟盘支持多个命名账户，可并行测试不同策略。

### 为什么用
- **验证 Agent 质量**：不是看报告写得好不好，而是看交易建议能不能赚钱
- **多策略对比**：不同账户跑不同策略（如只执行 BUY 信号 vs 全部执行），横向比较

### 怎么实现
1. `extract_signal(report)` 用正则提取建仓价、止损价、目标价、持仓周期、股票代码
2. `run_backtest(signal, lookback_days)` 获取历史日 K 线，从最早到最新逐日遍历：
   - 当日最低价 ≤ 建仓价 → 入场
   - 收盘价 ≤ 止损 → 止损出场（亏损）
   - 最高价 ≥ 目标价 → 止盈出场（盈利）
   - 持仓天数超限 → 到期出场（平盘）
3. `batch_backtest(reports)` 批量回测，输出胜率、平均收益、盈亏比、明细表
4. 多账户：`portfolio_{name}.json` 独立文件，`get_portfolio(name)` 按名缓存

### 设计思考
- 回测引擎只用日线，不用分钟线——FinBrain 的定位是中线基本面研究，不是日内交易
- 信号提取用正则而非 LLM——交易信号是结构化数据，不需要语义理解
- 多账户文件隔离：删一个账户不影响其他，适合并行测试

## 18. 上下文压缩 — 长对话记忆管理

### 是什么
当对话历史超过阈值（默认 12 条）时，将旧消息压缩为一条 LLM 摘要（80 字），只保留最近 N 条原文。避免 token 随对话轮数线性增长导致上下文窗口溢出。

### 为什么用
- LLM 的上下文窗口有限，长对话需要压缩
- Deep Analysis 模式：每次分析走完整流水线，不依赖对话历史——但 Chat 模式的闲聊和指令需要记忆上下文
- LangGraph 的 SqliteSaver 已持久化 graph 状态，但 Chat Agent 的 ReAct 循环不走 graph

### 怎么实现
1. `compress_history(history)`：当 `len(history) > COMPRESS_TRIGGER` 时触发
2. 旧消息（前 N-KEEP 条）拼接为文本 → LLM 生成 80 字摘要
3. 返回 `[{"role": "user", "content": "[历史摘要] ..."}] + recent` 替代完整历史
4. CLI 和 Streamlit 前端都调用此函数，阈值可通过 Settings 页面调整
5. 配置：`COMPRESS_TRIGGER=12`（触发阈值）、`COMPRESS_KEEP=6`（保留条数）

### 设计思考
- 摘要放在 user 角色而非 system——因为 LangChain 的 `_dicts_to_messages` 按角色构建消息列表
- 只压缩 Chat 模式的历史，Deep Analysis 每次独立运行不走压缩——因为 Deep Analysis 的 processing_log 已提供执行 trace

## 19. 业绩快报双通道接入 + 情景估值校验

### 场景

旧报告的问题：公告列表里列了半年度业绩快报，核心结论却仍在“等待半年报确认拐点”。根因有两个：

1. **财报慢源滞后**：`RPT_DMSK` 等三大报表接口在正式财报披露前仍停留在 Q1。
2. **快报只进展示层**：公告标题被列出，但营收/利润/同比数据没有回灌到利润表行，Analyst 看不到。

### 双通道设计

FinBrain 把“最新业绩”拆成两条独立通道，再用代码合并：

| 通道 | 来源 | 优点 | 缺点 |
|------|------|------|------|
| **财报快源** | 东财 `RPT_FCI_PERFORMANCEE` | 快报发布次日入库、结构化 | 无扣非净利润字段 |
| **公告快报** | `np-cnotice` 正文接口 | 有扣非、有同比、有管理层说明 | 表格排版，需要正则解析 |

```python
# tools.py 中的数据优先级（最终态）
正式财报(RPT_DMSK) > 业绩快报datacenter(RPT_FCI) > 公告快报正文(正则解析)
                         └ 结构化、无扣非 ─┘        └ 有扣非、文本解析 ─┘
                    两者经 merge_flash_into_profit 合并为完整最新期间行
```

### 怎么做：快源接入

```python
from backend.tools import get_financial_statements

# 600131 在 2026-07-17 发布半年度业绩快报
# 正式中报约 8 月底才披露，但快源已可在 2026-07-18 入库
fin = get_financial_statements("600131")
profit = fin["profit"]

print(profit[0]["date"], profit[0]["报告期"], profit[0]["归母净利润"]/1e8)
# 输出: 2026-06-30 半年报 2.31
```

`get_financial_statements` 内部会同时查询 `RPT_DMSK` 和 `RPT_FCI_PERFORMANCEE`，如果快源更新鲜，就合成一条新行并插入到利润表最前。

### 怎么做：公告快报回灌扣非

快源没有扣非字段，但公告快报正文里有。`merge_flash_into_profit()` 负责合并：

```python
from backend.tools import merge_flash_into_profit

profit = [
    {"date": "2026-06-30", "报告期": "半年报",
     "营业总收入": 34.33e8, "归母净利润": 2.31e8, "扣非净利润": None, "_快报源": True},
]
flash = {"报告期": "2026年半年度", "扣非净利润(亿元)": 2.29}
merge_flash_into_profit(profit, flash)
print(profit[0]["扣非净利润"])   # 2.29e8
```

### 怎么做：情景估值校验

旧报告的情景估值“三错二”：悲观价 0.25×20 写成 7.00，乐观价 0.59×25 写成 12.50。FinBrain 现在要求 Analyst 输出结构化 `EPS`/`PE` 字段，并由代码层做算术校验：

```python
from backend.tools import _validate_scenarios

item = {
    "情景估值": {
        "悲观": {"价格": 7.00, "EPS": 0.25, "PE": 20, "概率": "20%"},
        "基准": {"价格": 9.38, "EPS": 0.375, "PE": 25, "概率": "55%"},
        "乐观": {"价格": 12.50, "EPS": 0.59, "PE": 25, "概率": "25%"},
    }
}
_validate_scenarios(item)

print(item["_scenario_check"])
# 结果：悲观价 7.00 → 5.00，乐观价 12.50 → 14.75，概率加权价值代码重算
```

校验内容包括：
- 算术校验：`价格 ≈ EPS × PE`，不符按计算值修正（容差 15%）
- 单调性：`悲观价 ≤ 基准价 ≤ 乐观价`
- 概率合计：归一化到 100%
- 概率加权价值：不信任 LLM 的口算，由代码重算

### 怎么做：时效性代码检测

不能指望 LLM 自觉不写“等待中报”。`_detect_timeliness_conflict()` 在代码层检查：

```python
def _detect_timeliness_conflict(item, flash):
    if flash and "等待中报" in item.get("综合结论", ""):
        return {
            "type": "时效性矛盾",
            "desc": "半年度业绩快报已发布，结论仍要求等待半年报",
        }
    return None
```

检测到的冲突会进入 `_code_issues`，阻止审计跳过，触发 Analyst 重推理。

### 设计思考

- **免费数据源有节奏差异**：财报接口和公告接口不是同一个系统，必须做双通道。
- **代码校验比 LLM 自检可靠**：让 LLM 自己检查算术，它可能睁一只眼闭一只眼；代码会严格执行。
- **时效性必须代码兜底**：LLM 不知道哪条公告“重要”，除非把它写进 prompt 最前面并强制规则。
- **新数据源接入成本要低**：`RPT_FCI_PERFORMANCEE` 接入只改 `get_financial_statements` 内部，所有调用方零改动。
